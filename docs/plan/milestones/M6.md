> **Milestone design doc — self-contained.** Read this together with `docs/plan/roadmap.md` (the map, esp. §5 M6 + the red lines **§1.2 single-tenant / multi-user**, **§2.6 one unified reminder engine notifying the responsible party**, **§2.10 single-context / repository layer**, **§2.11 logic-in-app-not-SQL**) and, for "why we exist", `docs/inspiration/investigation.md`. >
> Conventions baked in: atomic steps (§9), blind-review checkpoints (§10), and 🟢 deploy self-test points (§11) so both manual and orchestrator execution can hang off this doc.
>
> **Scope note (read first).** M6 turns the single-admin app into a real **multi-user household** and pays down the security debt that M4/M5 explicitly deferred here. It delivers, in **five phases** (A roles & permissions, B user accounts & invitations, C responsible-party routing & per-user notification prefs, D audit + security hardening, E frontend) of **13 atomic steps**: a **three-role RBAC** model (admin / member / viewer) enforced by a code-defined permission matrix; **user administration** (list / role / enable-disable / delete) with a last-admin safety guard; **invitations** (one-time token link, optionally emailed) plus **self password-change** and **admin-initiated password reset**; **responsible-party assignment** on item definitions (default) and stock instances (override) that **routes reminders to the right person** (falling back to everyone when unassigned — zero-disruption migration of M4 behaviour); **minimal per-user notification preferences** (opt out of the in-app inbox and/or the email digest); a **security/admin audit log**; and the deferred **hardening pass** — auth-endpoint **rate limiting with exponential backoff**, an **SSRF guard** on the outbound webhook (and the future MQTT broker host), **per-request session authorization on `/media`**, and **sliding-window session expiry**.

## 1. Goal & non-goals

**Goal (the roadmap's M6 promise — real multi-user within the single household + hardening):**

- **Roles & permissions** — three fixed roles, **admin / member / viewer**, with a permission matrix defined **in code** (not a DB-driven permission table — roadmap §1.2/§2.11) and enforced by a single `require_permission` route dependency. `admin` = everything (user management, global settings/channels, audit, all data); `member` = full data CRUD but no user management, no global settings, no audit; `viewer` = read-only everything.
- **User administration** — an admin can list users, change a user's role, deactivate/reactivate, and delete, all behind a **last-admin guard** (the household can never lose its last active admin). Listing users is available to any authenticated user (for responsible-party pickers); only the *mutations* are admin-gated.
- **Invitations + passwords** — an admin invites a new user by email + role; the system mints a **one-time token** and returns a copyable **accept link** (and, when SMTP is configured, **sends it by email** — the two paths share one token). The invitee opens the link and **sets their own password** (the admin never learns it). Plus **self password-change** and an **admin-initiated password reset** (token link, same mechanism).
- **Responsible-party routing** — a `responsible_user_id` on **item definitions** (default) and **stock instances** (override). The reminder engine routes each reminder to the lot's/definition's **effective responsible party**, **falling back to all active users when none is set** (preserving the exact M4 behaviour for all existing data — a zero-disruption migration). This delivers the roadmap's "assignment-based reminders reach the right person."
- **Per-user notification preferences (minimal)** — each user can opt out of the **in-app inbox** and/or the **email digest**. (HTTP/MQTT are single global endpoints, so they have no per-user meaning — §2.)
- **Audit log** — an append-only `audit_log` of **security/admin events** (login success/failure, logout, user create/role-change/disable/delete, password change/reset, invitation issued/accepted, settings/channel changes), with an admin-only viewer.
- **Security hardening** (the M4/M5 deferrals) — **rate limiting + exponential backoff** on authentication endpoints; an **SSRF guard** on the outbound webhook (lightweight, always-on, **does not block private LAN** so Home Assistant on the LAN keeps working — §2); **per-request session authorization** on `/media` (the M5 capability-URL is upgraded to "must be logged in"); and **sliding-window session expiry** (activating the M0-prewired `sessions.last_seen_at` hook).

**Done when (🟢, expanded in §11):** invite a second user via a link, accept it by setting a password, and log in as them; a `viewer` is blocked from every write while a `member` can edit data but not reach user management / settings / audit; assign a durable to a user and watch its warranty reminder reach **only** that user while an unassigned item still reaches everyone; a user opts out of the email digest and stops receiving it (but still sees the in-app inbox); the audit log shows the logins, the invite, and the role change; brute-forcing the login backs off exponentially; `/media` returns 401 without a session; the outbound webhook refuses `http://169.254.169.254/…`. CI stays green, including the no-drift contract gate; migrations `0001`–`0032` apply cleanly to a fresh DB.

**Non-goals (explicitly out of M6 — deferred or later milestones):**
- **No custom / DB-driven roles or per-resource ACLs.** The three roles + code matrix are the ratified granularity (roadmap §7). A `custom_role` / fine-grained permission table is a parking-lot refinement.
- **No per-row data ownership / access scoping.** This is a single household where **all users share all data** (roadmap §1.2). Responsible-party is purely a **reminder-routing** hint, **not** an access-control boundary — a `member` can still edit any item regardless of who's responsible.
- **No per-user notification *cadence* / per-channel routing beyond the two opt-outs.** The full "per user × per channel × per schedule" matrix from M4 §12 stays parked; M6 ships only the in-app + email-digest opt-outs (§2, the author's "minimal version").
- **No OIDC / SSO / 2FA / email-verification-on-signup.** Auth stays opaque session cookies (roadmap parking lot). The invite link *is* the trust path; there is no separate email-confirmation step.
- **No SSRF private-network blocking, no DNS-rebinding IP-pinning, no allow-listing.** M6's guard blocks loopback/link-local/cloud-metadata/reserved targets but **allows private LAN** (the self-hosted + Home-Assistant-on-LAN core use case). Strict modes are §12.
- **No global API rate limiting.** Only the unauthenticated/auth endpoints are limited; the rest of the API is already gated by a session token (the author's call — §2).
- **No distributed / persistent rate-limit store.** The limiter is **in-memory, single-process** (matches the single-container SQLite deployment); it resets on restart (§12).
- **No audit of ordinary data mutations.** Stock changes are already audited by the M2 movement ledger; a generic data-change audit is out (§2/§12).

## 2. Locked decisions (settled in M6 planning; rationale in the planning discussion + roadmap §1.2/§2)

| Area | Decision |
|---|---|
| **Three fixed roles + code permission matrix** | Roles are the existing `users.role` **String(64)** column, app-validated against `{"admin","member","viewer"}` (no DB enum/CHECK — §2.11). A module-level `PERMISSIONS: dict[Role, set[Permission]]` maps each role to a small **Permission** set (`VIEW`, `EDIT`, `MANAGE_USERS`, `MANAGE_SETTINGS`, `VIEW_AUDIT`). A `require_permission(perm)` FastAPI dependency factory resolves the current user and raises **403 `auth.forbidden`** when the role lacks `perm`. **No migration** for the role column (it already exists; the bootstrap admin is already `"admin"`). |
| **Permission map (the matrix)** | `viewer → {VIEW}`; `member → {VIEW, EDIT}`; `admin → {VIEW, EDIT, MANAGE_USERS, MANAGE_SETTINGS, VIEW_AUDIT}`. **Self-service is never gated by a permission**: any authenticated user may read/update *their own* profile (`/auth/me`, prefs, change own password) and *their own* notification inbox, regardless of role. **Listing** users is `VIEW` (needed by responsible-party pickers); only user *mutations* are `MANAGE_USERS`. |
| **Enforcement = additive sweep over existing routers** | Today every authenticated user is effectively admin (no checks). M6 adds `Depends(require_permission(EDIT))` to every **data-mutating** route (POST/PATCH/PUT/DELETE on locations, categories, item kinds, definitions, instances, movements, attachments, tags, notes, barcodes, custom-fields, responsible-party). Read routes stay as-is (any authed user = `VIEW`). Settings/channels + `POST /reminders/run` → `MANAGE_SETTINGS`; user mutations + invitations + reset → `MANAGE_USERS`; `GET /audit` → `VIEW_AUDIT`. |
| **Invitations: pending row in `user_tokens`, user created on accept** | An invite does **not** create a half-baked `users` row. A `user_tokens` row (`purpose="invite"`, `email`, `role`, `token_hash`, `expires_at`) holds the pending invite; the `users` row is created **on accept** (with the chosen password, `is_active=True`). The raw token is returned to the admin **once** (for the copyable link) and **hashed** (sha256) at rest, so a DB leak never exposes a live link. Optional SMTP send reuses the M4 email transport. |
| **Admin reset + self change-password share the token table** | `user_tokens.purpose="password_reset"` (with `user_id`) backs the admin-initiated reset (one-time link). **Self change-password** (`POST /auth/change-password`, verify current → set new) needs no token. One token table, two purposes — DRY. |
| **Responsible party on both definition (default) + instance (override)** | A nullable `responsible_user_id` FK (`ondelete=SET NULL`) on **`item_definitions`** and **`stock_instances`**. A lot's **effective** responsible party = `instance.responsible_user_id` → else `definition.responsible_user_id` → else **none**. (Same override→default→none shape as the M4 lead-time chain.) `SET NULL` means deleting a user cleanly drops their assignments back to the fallback. |
| **Reminder routing + fallback-to-all (zero-disruption)** | `best_before` / `warranty` (per-lot) route to the **lot's** effective responsible party; `low_stock` (per-definition) routes to the **definition's** responsible party. **When the effective party is `None`, the source falls back to *all active users*** — i.e. exactly M4's behaviour. Because every existing row starts unassigned, M6 ships behaving identically to M4 and *narrows* only as the household assigns owners. |
| **Per-user notification prefs: two opt-outs** | Two booleans on `users` — `notify_in_app` (default true) and `notify_email_digest` (default true). For a routed recipient, a `notifications` row is created **iff** they want *at least one* channel (keeps the dedup ledger + email source intact); the in-app inbox query is gated by `notify_in_app`; the email digest skips users with `notify_email_digest=False`. **HTTP/MQTT are global single endpoints → no per-user pref** (they fire once per notification regardless). Exposed via the existing `PATCH /auth/me`. |
| **Audit log: append-only, security/admin events only** | An `audit_log` table written by an `AuditService` from the auth/user/settings code paths. Append-only (no update/delete in the app). `actor_user_id` (`SET NULL`) **plus a denormalized `actor_email` snapshot** so failed logins (no user) and post-deletion records still read sensibly. Admin-only `GET /audit` with filters. Ordinary data mutations are **not** audited (the movement ledger already covers stock). |
| **Rate limiting: auth endpoints only, in-memory, exponential backoff** | A small in-process `RateLimiter` keyed by `(scope, client_ip[, email])` guards `login`, `setup`, `change-password`, and the two token-`accept` endpoints. After a per-window attempt threshold it imposes an **exponentially growing lockout** (doubling per violation, capped), returning **429 `auth.rate_limited`** with a `Retry-After` header; a success clears the counter. Single-process store (matches the single container); resets on restart. The rest of the API is unthrottled (already session-gated — author's call). |
| **SSRF guard: lightweight, always-on, LAN-allowed** | A `validate_outbound_url(url)` helper runs before the webhook POST: scheme ∈ `{http,https}`, host present, then **resolve the host and reject loopback (127/8, ::1), link-local (169.254/16 incl. `169.254.169.254`, fe80::/10), unspecified, multicast, reserved** — but **allow** private LAN (10/8, 172.16/12, 192.168/16, fc00::/7). The webhook call also sets `follow_redirects=False` and a hard timeout. A lighter `validate_broker_host` applies the same block-list to the (currently disabled) MQTT host. Rationale + the public-IP/Home-Assistant analysis: §4.7. |
| **`/media` per-request session auth** | The M5 `/media/{shard}/{digest}` route gains `Depends(get_current_user)` → **401 without a valid session**. The browser sends the session cookie automatically on same-origin `<img>`/download requests, so galleries keep working. In a single household where all authed users share all data, "must be logged in" *is* sufficient authorization (viewer included). The route stays `include_in_schema=False`. |
| **Sliding-window session expiry** | `sessions.verify()` activates the prewired `last_seen_at`: on a valid request, when less than half the TTL remains, it refreshes `expires_at = now + TTL` and `last_seen_at = now` (throttled so most requests don't write). Active users stay logged in; idle sessions still expire. TTL stays the M0 24h (operator-tunable later). |
| **Error codes (additions)** | New: `auth.forbidden` (403), `auth.rate_limited` (429), `auth.invalid_token` (400 — invite/reset token bad/expired/used), `auth.password_incorrect` (400 — wrong current password), `user.not_found` (404), `user.email_exists` (409 — email already a user), `user.last_admin` (409 — would remove the last active admin), `invitation.not_found` (404 — revoke a missing invite). **Eight** new `ErrorCode` entries, all through the M1.5 uniform error envelope. |

> These extend M0–M5 foundations. The new tables/columns, every new schema, and the new endpoints flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope. No new runtime dependencies (rate limiter, SSRF guard, and token hashing all use the stdlib + the already-present `argon2`/`httpx`). The `/media` auth change and the role checks are the only "foundations" touch and need no `AGENTS.md` dependency edits.

## 3. Data model

Three new tables + four additive column deltas. All use M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `Base`, rules in the service layer, DB access **only** through repositories. The column deltas are nullable or server-defaulted plain adds (SQLite batch-mode `add_column`, no table rebuild). **Current head = `0027`; M6 adds `0028`–`0032`.**

### 3.1 `user_tokens` — new table (migration `0028`)

One-time tokens for invitations and admin password resets.

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `purpose` | String(32) | no | `invite` / `password_reset` (app-validated; no DB CHECK — §2.11). |
| `email` | String(254) | yes | The invitee email (invites only; lower-cased). NULL for resets. |

## 4. Backend design

### 4.1 Layering (extends M0–M5)

- **RBAC core** (`app/auth/permissions.py`, new): the `Role` constants, the `Permission` constants, the `PERMISSIONS` map, and `has_permission(role, perm) -> bool`.
- **Dependencies** (`app/api/deps.py`): a new `require_permission(perm)` factory returning a dependency that calls `get_current_user`, checks `has_permission(user.role, perm)`, and raises `auth.forbidden`/403 otherwise (returns the `User` so routes can reuse it). Thin shortcuts `require_edit`, `require_manage_users`, `require_manage_settings`, `require_view_audit` for readability.
- **Repositories** (`app/repositories/`): extend `UserRepository` (list-all incl. inactive, set-role, set-active, delete, count-active-admins, set-notify-prefs, set-password-hash, public selectable list); new `UserTokenRepository` (create, get-by-token-hash, get-pending-invite-by-email, list-pending, mark-consumed, delete, purge-expired); new `AuditLogRepository` (append, list with filters + pagination).
- **Services** (`app/services/`):
  - `UserAdminService` — list/get/change-role/activate-deactivate/delete with the **last-admin guard** (`count_active_admins()` must stay ≥ 1 across role-demote / deactivate / delete; else `user.last_admin`). Emits audit events.
  - `InvitationService` — create invite (dup-email guard → `user.email_exists`; revoke any prior pending invite for the same email; mint token; optional SMTP), list/revoke, **accept** (validate token → create user → consume → audit). Reset variants for `password_reset`.
  - `PasswordService` helpers — self change (verify current → `auth.password_incorrect` on mismatch → set hash → audit; optionally revoke the user's *other* sessions), admin reset (issue token), accept-reset (set hash, consume, audit).
  - `AuditService` — `record(event_type, *, actor, target_type, target_id, params, request)` building the row (pulls `actor_email` + `ip_address`); `list(filters)`.
  - `RateLimiter` (`app/core/rate_limit.py`, new) — in-memory `check(scope, key)` / `register_failure` / `clear`; exponential-backoff policy (§4.6).
  - `outbound` URL/host validation (`app/core/net_guard.py`, new) — `validate_outbound_url`, `validate_broker_host` (§4.7).
- **Reminder engine** (`app/services/reminder_engine.py`) — recipient resolution changes (§4.4) and per-user-pref gating (§4.5). The notification *model* is unchanged.

### 4.2 RBAC enforcement (the additive sweep)

`require_permission` plugs in beside the existing `get_authenticated_context`/`get_current_user`. Concretely, a mutating route gains one parameter:

```python
@router.post("/locations")
def create_location(
    body: LocationCreate,
    ctx: RequestContext = Depends(get_authenticated_context),
    _: User = Depends(require_permission(Permission.EDIT)),
) -> LocationResponse:
    ...
```

The sweep (Step 1) is **mechanical**: every existing POST/PATCH/PUT/DELETE on a data router gets `require_permission(EDIT)`; read routes are unchanged (any authed user already satisfies `VIEW`). Settings/channels + `POST /reminders/run` get `MANAGE_SETTINGS`. The new user/invitation/audit routes declare their own. **Self routes (`/auth/me`, change-password, the user's own notification inbox) get no permission gate.** A blind reviewer checks that *no* mutating data route was missed (viewer must be unable to write anything) and *no* read route was over-gated (viewer must be able to read everything).

### 4.3 Invitations, passwords & the token lifecycle

- **Create invite** (`POST /invitations`, `MANAGE_USERS`): validate email isn't already a user (`user.email_exists`); revoke any prior pending invite for that email; mint `token = secrets.token_urlsafe(32)`, store `sha256(token)`; build `accept_url = <app_origin>/invite/accept?token=<token>`. If SMTP is configured, send the link by email (best-effort; failure doesn't fail the request — the link is still returned). **Response includes the raw `accept_url`** (admin is trusted) + `emailed: bool`. Audit `invitation.issued`.
- **Accept invite** (`POST /invitations/accept`, **public**, rate-limited): `{token, password}` → hash the token, look up a pending row → else `auth.invalid_token`; create the user (`email`, `role`, `password_hash`, `is_active=True`); set `consumed_at`; audit `invitation.accepted`. Does **not** auto-login (consistent with setup). A companion `GET /invitations/accept?token=` validates and returns `{email, role}` to render the form.
- **Admin reset** (`POST /users/{id}/reset-password`, `MANAGE_USERS`): mint a `password_reset` token for the user; return `reset_url`; optional SMTP. Audit `password.reset` (issued).
- **Accept reset** (`POST /password-reset/accept`, **public**, rate-limited): `{token, password}` → validate → set the user's `password_hash` → consume → audit `password.reset` (completed). Optionally revoke that user's existing sessions.
- **Self change-password** (`POST /auth/change-password`, any authed): `{current_password, new_password}` → verify current (`auth.password_incorrect` on mismatch) → set hash → audit `password.changed` → revoke the user's *other* sessions (keep the current one).
- **Token hygiene**: `purge_expired` for `user_tokens` runs in the startup lifespan beside the existing session purge.

### 4.4 Reminder recipient routing (the easy-to-get-wrong logic)

Replaces the unconditional `recipients = self._user_repo.list_active()` (engine line 306) with **per-source, per-subject** resolution:

```
effective_responsible(lot):                       # for best_before / warranty
    return lot.responsible_user_id
        or lot.definition.responsible_user_id
        or None

effective_responsible(definition):                # for low_stock
    return definition.responsible_user_id or None

recipients_for(subject):
    rid = effective_responsible(subject)
    if rid is not None and the user is active:
        return [that user]
    return all_active_users                        # fallback == M4 behaviour
```

- The **date sources** (`_evaluate_date_source`) compute `recipients_for(lot)` **per lot** (not one global recipient list), then run the existing per-(user, lot, date) dedup/window logic for exactly those recipients.
- The **low-stock** path (`_evaluate_low_stock_for_user`) iterates `recipients_for(definition)` per low definition. The event-trigger path (`evaluate_low_stock`) uses the same resolution for the one scoped definition.
- A responsible user who is **inactive** or whose row was deleted (`SET NULL`) collapses to the fallback — never a dropped reminder.
- **Lead-time resolution is unchanged** (per-item → per-user → global, M4 §4.3): once the recipient is chosen, their per-user lead override still applies.

### 4.5 Per-user notification-pref gating

- **Row creation**: in the engine, before writing a `notifications` row for recipient `U`, skip `U` entirely if `not U.notify_in_app and not U.notify_email_digest` (they want nothing). Otherwise create the row (it remains the dedup ledger + the email source).
- **In-app inbox**: the inbox list endpoint (M4) returns rows only when the current user has `notify_in_app=True`; with it off, the inbox reads empty and the UI hides the bell. (Rows may still exist to feed the email digest — a deliberate, documented simplification, §12.)
- **Email digest**: `EmailChannel._deliver_to_recipient` skips a recipient whose `notify_email_digest=False` (a one-line filter; the digest already groups by `user_id`, email.py line 147).
- **HTTP/MQTT** dispatch is untouched (global, fires once per notification).

### 4.6 Rate limiting + exponential backoff

`RateLimiter` (in-memory, per-process) tracks, per `(scope, key)`, a recent failure count and a lockout-until timestamp:

- **Key**: `client_ip` for `login`/`setup`/token-accept; `(client_ip, user_id)` for `change-password`. Client IP = `request.client.host` (X-Forwarded-For handling behind a trusted proxy is a documented deployment note — §12).
- **Policy**: allow up to `N` failures (default 5) per rolling window (default 5 min); beyond that, a **lockout that doubles per subsequent violation** — `base_lockout * 2^(violations - 1)`, capped (e.g. base 30s, cap 30 min). A **success clears** the counter.
- **Enforcement**: a `auth_rate_limit(scope)` dependency calls `check()` *before* the handler; on lockout it raises `auth.rate_limited`/429 with a `Retry-After` header and `params={"retry_after_seconds": n}`. The handler calls `register_failure()` on a bad credential / bad token and `clear()` on success.
- Failed logins also write `auth.login_failed` to the audit log.

### 4.7 SSRF guard (outbound) — and the Home-Assistant analysis

**Where the only outbound fetch is.** The app makes exactly one kind of outbound HTTP request: the **webhook notifier** (`app/notifications/channels/http.py`, POST at line 228). MQTT *connects* to a broker (different protocol, currently disabled). The inbound HA state endpoint is inbound, not SSRF.

**Why a guard at all (honest scope).** Classic SSRF bites when *untrusted users* submit URLs the server fetches. Omniventory has **none** of those — the only fetched URL is the **admin-configured** webhook, and after M6 only `MANAGE_SETTINGS` (admin) can set it. So the practical risk is **low**; the guard is **defense-in-depth**, valuable mainly for (a) cloud deployments (block the `169.254.169.254` metadata endpoint → no instance-credential theft), (b) catching misconfiguration, (c) limiting a compromised admin session's blast radius. This is why M6 ships a **lightweight, always-on** guard rather than strict modes.

**Home Assistant on LAN is a first-class target, so the guard does NOT block private ranges.** The author reaches HA via a **public IP** (hairpin NAT loops the public hostname back into the LAN) — to the app, the webhook resolves to a *public* IP and passes trivially; even a strict private-block wouldn't affect that setup. But the open-source audience commonly points straight at `192.168.x.x`, so **private LAN is allowed**.

**`validate_outbound_url(url)`**: scheme ∈ `{http,https}` and host present → else reject; resolve the host (`socket.getaddrinfo`) and reject if **any** resolved IP is loopback / link-local (incl. `169.254.169.254`) / unspecified / multicast / reserved (Python `ipaddress` classification); **allow** private LAN. The webhook call adds `follow_redirects=False` (a 302→internal can't sneak past) and keeps the existing hard timeout. `validate_broker_host(host)` applies the same block-list to the MQTT broker host (light; MQTT is disabled). The residual DNS-rebinding TOCTOU (resolve-then-connect) is accepted given admin-configured static config (§12).

### 4.8 `/media` authorization & session sliding-window

- **`/media`**: add `Depends(get_current_user)` to the existing `GET /media/{shard}/{digest}` route (main.py ~382) → 401 without a session. Same-origin `<img>`/download requests carry the cookie automatically; galleries are unaffected. `include_in_schema=False` stays.
- **Sliding window**: `sessions.verify()` (auth/sessions.py) — on a valid session, if `expires_at - now < TTL/2`, set `expires_at = now + TTL` and `last_seen_at = now`, then `flush`. Throttled by the half-life test so the vast majority of requests don't write. The refresh must be **committed** for the request even on read-only routes (handled in the dependency / `get_db` teardown — flagged as easy-to-get-wrong in §5).

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0–M5 §5 (`make check` all green; build; `make codegen` no-drift). The **logic that must have unit tests** in M6:

**Backend**
- **Permission matrix & enforcement** (§4.2 — required): for each role × each gated route class, allowed/denied is exactly the matrix; `viewer` is rejected (403 `auth.forbidden`) on **every** data mutation and allowed on every read; `member` allowed on data, rejected on user/settings/audit; `admin` allowed everywhere; self routes (`/auth/me`, change-password, own inbox) allowed for all roles.
- **Last-admin guard** (§4.1): demoting / deactivating / deleting the **last active admin** → `user.last_admin`; the same op succeeds when a second active admin exists; a self-demote that would orphan the household is blocked.
- **Invitations & tokens** (§4.3): invite for an existing email → `user.email_exists`; a fresh invite revokes the prior pending one for that email; accept creates the user with the invited role + a usable password; a **consumed**, **expired**, or **unknown** token → `auth.invalid_token`; tokens are stored **hashed** (the raw token never hits the DB); `purge_expired` drops stale rows.
- **Passwords**: self change with a wrong current password → `auth.password_incorrect`; a correct change updates the hash and revokes *other* sessions but keeps the current; admin reset issues a working token; accept-reset sets the new hash and consumes the token.
- **Responsible-party routing** (§4.4 — required): instance override wins over definition default; definition default applies when the lot is unset; **unassigned → all active users** (M4 parity); an **inactive/deleted** responsible user collapses to the fallback (no dropped reminder); low-stock routes by the definition's party; lead-time resolution still applies after recipient selection.
- **Per-user notify prefs** (§4.5): `notify_email_digest=False` removes only that user's digest (in-app row intact); `notify_in_app=False` empties that user's inbox; a user with **both** off gets no notification row; defaults (both true) reproduce M4 exactly.
- **Rate limiting** (§4.6): N failures within the window are allowed, the next is locked out; lockout **doubles** per violation and is capped; a success **clears** the counter; the lock returns 429 + `Retry-After`; the limiter is per-`(scope, key)` (one IP's lockout doesn't affect another).
- **SSRF guard** (§4.7): `http://127.0.0.1`, `http://169.254.169.254`, `http://[::1]`, a hostname resolving to loopback, an `ftp://` scheme, and a missing host are **rejected**; a public host **and** a private-LAN host (`192.168.x`, `10.x`) are **allowed**; the webhook call sets `follow_redirects=False`.
- **`/media` auth**: a request without a session → 401; with a valid session → the file (correct `content_type` + `nosniff`).
- **Session sliding-window**: a request within the refresh threshold extends `expires_at`/`last_seen_at` (and the change is committed); a request with plenty of TTL left does **not** write; an expired session still 401s.
- **Audit log**: each covered event writes exactly one row with the right `event_type`, `actor_email` snapshot, target, and params; failed login records with `actor_user_id=NULL` but `actor_email` set; the list endpoint filters + paginates; admin-only.
- **Migration round-trip**: `0028`–`0032` upgrade clean on a DB at `0027` and downgrade cleanly; existing rows unaffected (booleans default true, FKs NULL).

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): role-gated nav/routes render per role (mock `user.role`); write-action buttons are hidden/disabled for `viewer`; the **Users** page lists, changes role, toggles active, deletes (with the last-admin error surfaced), and creates an invite (link shown + copy + optional "send email"); the **invite-accept** and **reset-accept** public pages set a password and route to login; **change-password** validates and surfaces `auth.password_incorrect`; the **responsible-party picker** sets/clears assignment on definition + instance forms; the **notification-pref** toggles round-trip via `PATCH /auth/me`; the **audit** page lists + filters; the **rate-limit** 429 surfaces a clear "try again in N s" message; all new strings present in **en + zh**; dates/numbers via M1.5 `formatDate`/`formatQuantity`.

## 7. Frontend design

### 7.1 Auth context + role gating (`AuthContext`)
Lift the `user` that `App.tsx` currently passes by prop into a small **`AuthContext`** exposing `{ user, role, can(permission), refresh, logout }`, with a client-side mirror of the permission matrix (`can('EDIT')`, `can('MANAGE_USERS')`, …) so the UI can hide what the backend will refuse. Gating points:
- **Nav** (`AppShell`): `Configuration`, `Users`, `Audit` items render only for `admin`; everything else for all roles.
- **Routes**: admin-only routes (`/users`, `/audit`, `/configuration`) redirect non-admins to the dashboard; the existing data routes stay open (read).
- **Write actions**: create/edit/delete buttons and the intake/consume/move controls are hidden (or disabled with a tooltip) when `!can('EDIT')` (viewer). The backend is still the source of truth (defense in depth).
- The profile/user button shows the current **role**.

### 7.2 Users admin page (`pages/Users.tsx`, admin)
A table of users (email, role, active, created) with: change-role (select), activate/deactivate toggle, delete (confirm) — all surfacing `user.last_admin` clearly — and an **Invite** action: a modal taking email + role, then showing the returned **accept link** with a copy button and, when SMTP is configured, a **"Send email"** confirmation. A **pending invitations** list with revoke. A per-user **"Reset password"** action that shows the reset link (+ optional email).

### 7.3 Public token pages (`pages/AcceptInvite.tsx`, `pages/ResetPassword.tsx`)
**Pre-auth** pages (like Login/Setup). `App.tsx`'s gate must route `/invite/accept` and `/password-reset/accept` **before** the `me` check (they work with no session): read the `token` from the query, call the `GET …/accept` validator, render a set-password form, POST, then route to Login. This is the one change to the outer auth gate (a small path check in the `loading`/`anon` branches, or lifting a minimal public router).

### 7.4 Change-password (profile)
A "Change password" form (current + new + confirm) in a profile/account area, calling `POST /auth/change-password`, surfacing `auth.password_incorrect`.

### 7.5 Responsible-party assignment
A **user picker** (autocomplete over `GET /users`, clearable) on the **definition** form (`DefinitionFormModal`) and the **instance** form (`InstanceFormModal`), plus a read-only "Responsible: …" line on the detail pages. Clearing it = inherit (instance → definition) / unassigned (definition).

### 7.6 Per-user notification preferences
Two toggles — **In-app inbox** and **Email digest** — in the user's preferences area (alongside the M1.5/M4 language + lead-time prefs), persisted via `PATCH /auth/me`.

### 7.7 Audit log page (`pages/Audit.tsx`, admin)
A read-only, paginated table (time, event, actor, target, detail) with filters by event type / actor / date range.

### 7.8 i18n
New namespaces: **`users`**, **`invitations`**, **`account`** (change-password), **`audit`**, **`roles`** (role labels), **`responsible`** (assignment) (en + zh), registered in `src/i18n/index.ts`; additions to `nav` (Users, Audit) and `errors` (the eight new codes). The 429 message uses `params.retry_after_seconds`. Tests pinned to `en` (M1.5).

1. **Invite a second user:** as admin, open **Users → Invite**, enter an email + role `member` → copy the **accept link** (and, if SMTP is set, send it). Open the link in a fresh browser, **set a password**, land on login, and log in as the new member.
2. **Role enforcement:** as the **member**, confirm you can create/edit/consume stock but **cannot** see Users / Configuration / Audit and get refused if you hit those URLs directly. Invite/create a **viewer**; confirm the viewer sees everything **read-only** (no create/edit/delete/intake controls).
3. **Last-admin guard:** try to demote or deactivate the **only** admin → blocked with a clear message; add a second admin, then it's allowed.
4. **Responsible-party routing:** assign a durable's **instance** (or its definition) to the member; set its `warranty_expires` near the lead window; **run the scan** → only the **member** gets that reminder. Leave another item **unassigned** → its reminder reaches **everyone**. Assign a consumable's **definition** to the viewer, drop it below `min_stock` → only the viewer gets the low-stock reminder.
5. **Per-user preferences:** as the member, turn **off the email digest** (keep in-app) → next digest skips them but the in-app bell still shows; turn **off in-app** → the bell empties.
6. **Passwords:** change your own password (wrong current → clear error; right → works, other sessions logged out); as admin, **reset** the member's password via the link.
7. **Audit log:** as admin, open **Audit** → see the logins (incl. a deliberate failed one), the invite issued + accepted, the role change, and the settings change; filter by event type and actor.
8. **Rate limiting:** hammer the login with a wrong password several times → responses **back off exponentially** (429 + "try again in N s"); a correct login clears it.
9. **SSRF + media:** set the webhook URL to `http://169.254.169.254/x` → the notifier refuses it (a LAN/public URL is accepted); open a `/media/...` URL in a **logged-out** tab → **401**; logged in → the image loads.
10. **Sliding sessions:** stay active past the old fixed window → you remain logged in; go idle past the TTL → you're logged out.
11. **CI green:** the same gates pass in GitHub Actions, including the no-drift contract gate; migrations `0001`–`0032` apply cleanly to a fresh DB in the docker smoke test.

## 12. Open questions / deferred

- **Custom roles / fine-grained permissions:** the three-role code matrix is the ratified granularity. A `custom_role` + per-resource ACLs (DB-driven) is a contained follow-up if real use demands it.
- **Per-user cadence / per-channel routing:** M6 ships two opt-outs (in-app, email digest). Full "per user × channel × schedule" (and making HTTP/MQTT per-user) stays parked (M4 §12).
- **In-app vs email decoupling:** a user with in-app **off** but email **on** still gets a `notifications` row (the email source) that the inbox query hides — a deliberate simplification. A cleaner split (separate inbox-visibility flag per row, or an outbox table) is a refinement.
- **SSRF strict modes:** M6 allows private LAN and accepts the resolve-then-connect DNS-rebinding TOCTOU (admin-configured URLs). An optional "block private targets" toggle and IP-pinning are available later if the threat model changes.
- **Rate-limit store & proxy trust:** in-memory, single-process, resets on restart, keys on `request.client.host`. A persistent/shared store and a documented `X-Forwarded-For` trust config land if/when multi-process or a fronting proxy is in play.
- **OIDC / SSO / 2FA / email-verified signup:** parking lot; the invite link is the trust path in M6.
- **Audit scope & retention:** security/admin events only; no generic data-change audit (the movement ledger covers stock), no retention/rotation policy yet (the table grows unbounded — a pruning job is a later refinement).
- **Session controls:** sliding-window only; an explicit "active sessions" list + "log out everywhere" UI and an absolute-lifetime cap are refinements.
- **Account self-service:** M6 has change-password + prefs; self-service email change and account deletion are later.
