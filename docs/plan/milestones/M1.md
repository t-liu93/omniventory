## 1. Goal & non-goals

**Goal (the roadmap's M1 promise):** register and browse **durable goods** in a real, nested location hierarchy. M1 is where the **definition / instance split** — the spine the whole project hangs off (roadmap §2.1) — first lands in the schema. Every later milestone (M2 ledger, M3 best-before, M4 reminders) extends the tables M1 creates.

**Done when (🟢, expanded in §11):** you can create a nested location tree; register a serialized durable into it (with serial / model / manufacturer / warranty / purchase value); edit it; search for it; mark a container location as "this place *is* a tracked asset" (container-as-item); and the `serial ⇒ quantity = 1` constraint rejects bad input at both the API and DB layers. CI stays green, including the no-drift contract gate.

**Non-goals (explicitly out of M1 — they belong to later milestones):**
- **No movement ledger.** `quantity` is a directly-stored column in M1; the typed in/out/move/adjust ledger and "quantity derived from movements" arrive in **M2** (which backfills an initial-intake movement per existing instance — see §2 decision Q2).
- **No best-before / expiry.** `best_before_date` and `default_best_before_days` (auto-compute on intake) are **M3**.
- **No reminders.** `warranty_expires` is **stored but never scanned** in M1; the unified reminder engine is **M4**.
- **No cross-cutting features.** Photos / attachments, tags, notes, custom fields, barcodes are **M5** — so **M1 durables have no photos**.
- **No multi-unit conversion.** Definitions carry a single `unit` string; purchase/stock/consume units + factors are **M8**.
- **No min-stock / low-stock.** `min_stock` lands on the Definition in **M2** (its consumer), not now.
- **No multi-user / roles beyond M0.** All M1 endpoints require a valid session (the single admin); roles/permissions are **M6**.

## 2. Locked decisions (settled in M1 planning; rationale in roadmap §2/§3 + investigation ch.3)

| Area | Decision |
|---|---|
| **Spine** | **Definition / instance split** (InvenTree `Part`/`StockItem`). `item_definitions` = "what kind of thing" (name, category, kind, unit, defaults). `stock_instances` = "this specific lot/unit" (quantity, location, serial, warranty, value). Never collapse them (roadmap §2.1). |
| **Serial constraint** | **`serial ⇒ quantity = 1`** enforced in **both** the DB (a `CHECK (serial IS NULL OR quantity = 1)`) and the application service layer (roadmap §2.2). One instance table holds both per-unit durables and (later) in-bulk consumables. |
| **Container-as-item** (Q1) | **Bridge, not merge.** `locations` is its own self-referential tree; a `locations.item_instance_id` **nullable, unique FK → stock_instances** expresses "this location physically *is* that tracked durable" (the toolbox). Keeps the two-layer model clean; container-as-item is an *add-on*, not a Homebox-style single-table merge. |
| **Quantity in M1** (Q2) | **Stored directly** as a `Numeric` column (default `1`). The ledger does not exist yet (M2). When M2 lands, `quantity` becomes **derived from the movement ledger**, and an M2 migration backfills one `intake` movement per existing instance equal to its current quantity. M1 does **not** over-design for a ledger that isn't here. |
| **Decimal & money** | `quantity` and `purchase_price` are SQLAlchemy `Numeric` (→ Python `Decimal`), never float (roadmap §2.9). Money **inherits `household.currency`**; instances store **no** per-instance currency in v1. No money arithmetic happens in M1, so rounding rules are deferred to the milestone that first computes with money. |
| **Definition defaults timing** | `min_stock` (M2) and `default_best_before_days` (M3) are **added by the milestone that consumes them**, each via its own migration. M1's Definition carries only what M1 uses. Pre-1.0, no data — adding a column later is cheap and keeps each milestone's schema change cohesive. |
| **`kind`** (author's call) | A **FK → `item_kinds` lookup table**, *not* a baked-in string enum — decided now (M1) so future references never need a breaking contract change. M1 keeps it **minimal**: the table is seeded with three system kinds (`durable` / `consumable` / `perishable`), exposed **read-only** (`GET /kinds`); the Definition's `kind_id` defaults to `durable` when omitted. A UI hint + default driver only — **no hard business logic** branches on it yet. Kinds CRUD + per-kind behaviour flags are deferred (§12). |
| **`unit`** | Single free-text unit string on the Definition (e.g. `pcs`, `m`, `kg`), default `pcs`. Multi-unit conversion is M8. |
| **Search** | Minimal: list endpoints accept a `q` query param doing a **case-insensitive substring** match over the relevant text columns (definition name; instance serial / model / manufacturer). No full-text index. |
| **Serial uniqueness** | A **partial unique index on `(definition_id, serial)` where `serial IS NOT NULL`** — the same physical serial can't be registered twice for the same product. (Global uniqueness is rejected: two different products may legitimately share a serial string.) Listed in §12 to revisit. |
| **Tree delete semantics** | Deleting a location/category is **blocked (HTTP 409)** if it still has child nodes, and a location is also blocked if any instance is assigned to it or it is referenced as a container (`item_instance_id`). No cascade in M1 — safe default, avoids orphaning. |
| **Auth** | **Every M1 business endpoint requires a valid session** (via `get_authenticated_context` / `get_current_user`, the M0 dependencies). No public domain endpoints. |
| **Frontend routing** | Adopt **`react-router-dom`** (M0 deliberately shipped no router). The M0 `App.tsx` auth gate stays the outer shell; routing lives *inside* the authenticated `AppShell`. Tree UI uses Mantine's **`Tree`** component. |

> These extend M0's foundations; they are **not** "tech stack" changes, so `AGENTS.md` does **not** change for M1 — except `react-router-dom` is a genuinely new frontend dependency (note it in the M1 report; the AGENTS "Tech stack" line may get a one-word addition at milestone wrap-up if the author wants).

## 3. Data model (the heart of M1)

Five new tables. All use the M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `DeclarativeBase`, `created_at` via `server_default=func.now()`, access **only** through a repository (no raw queries in routes), business rules in a service layer.

### 3.1 `locations` — self-referential tree (+ container-as-item bridge)

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer PK | no | autoincrement |
| `name` | String(255) | no | |
| `description` | String(1000) | yes | |
| `parent_id` | FK → `locations.id` | yes | `NULL` = root node; arbitrary depth |
| `item_instance_id` | FK → `stock_instances.id` | yes | **unique**; the durable instance this location *is* (container-as-item). **Added in step 4** (see §3.6). |
| `created_at` | DateTime(tz) | no | `server_default=func.now()` |

- **Cycle prevention** (app layer, roadmap §2.11 — no SQL triggers): on reparent, the new parent must not be the node itself nor any of its descendants.
- A location's *own* place in the world is its `parent_id`; the toolbox-instance separately sits in some location via `stock_instances.location_id`. The bridge (`item_instance_id`) is the back-link. No cycle is possible between the two because they model different relationships (containment vs. identity).

### 3.2 `categories` — self-referential tree

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer PK | no | |
| `name` | String(255) | no | |
| `description` | String(1000) | yes | |
| `parent_id` | FK → `categories.id` | yes | `NULL` = root; arbitrary depth |
| `created_at` | DateTime(tz) | no | |

Same cycle-prevention + delete-guard rules as locations (it reuses the tree pattern from step 1).

### 3.3 `item_kinds` — kind lookup table (seeded)

A small reference table so `kind` is a **FK, not a baked-in string enum** — chosen now (M1) so future references never need a breaking contract change. M1 keeps it **minimal**: seeded with the three system kinds, exposed read-only over the API; user-managed kinds CRUD and per-kind behaviour flags (e.g. "perishable kinds have best-before") are deferred (§12).

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer PK | no | |
| `code` | String(32) | no | **unique**; stable machine key (`durable` / `consumable` / `perishable`) |
| `name` | String(64) | no | display label |
| `is_system` | Boolean | no | default `true`; seeded system kinds (reserves room for user kinds later) |
| `created_at` | DateTime(tz) | no | |

Seeded by its migration (M0's `INSERT OR IGNORE` pattern) with `durable` / `consumable` / `perishable` (all `is_system = true`).

### 3.4 `item_definitions` — "what kind of thing"

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer PK | no | |
| `name` | String(255) | no | |
| `description` | String(1000) | yes | |
| `category_id` | FK → `categories.id` | yes | |
| `kind_id` | FK → `item_kinds.id` | no | the service resolves the `durable` kind when omitted |
| `unit` | String(32) | no | default `pcs` |
| `default_location_id` | FK → `locations.id` | yes | suggested location for new instances |
| `created_at` | DateTime(tz) | no | |

> `min_stock` (M2) and `default_best_before_days` (M3) are intentionally **absent** — added by their consuming milestone.

### 3.5 `stock_instances` — "this specific lot / unit"

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer PK | no | |
| `definition_id` | FK → `item_definitions.id` | no | what this instance is |
| `location_id` | FK → `locations.id` | yes | where it physically sits |
| `quantity` | Numeric(18,6) | no | default `1`; **directly stored in M1** (ledger-derived in M2) |
| `serial` | String(255) | yes | |
| `model_number` | String(255) | yes | durable identity |
| `manufacturer` | String(255) | yes | durable identity |
| `warranty_expires` | Date | yes | **stored only**; reminders are M4 |
| `warranty_details` | String(1000) | yes | |
| `purchase_price` | Numeric(18,2) | yes | value; currency = `household.currency` |
| `purchase_date` | Date | yes | |
| `purchase_source` | String(255) | yes | where it was bought |
| `created_at` | DateTime(tz) | no | |

Constraints (DB-level, schema invariants — permitted by roadmap §2.1/§2.11):
- **`CHECK (serial IS NULL OR quantity = 1)`** — the serial constraint, also enforced in the service layer.
- **Partial unique index** on `(definition_id, serial) WHERE serial IS NOT NULL`.

> **Identity vs. uniqueness (important).** The table PK stays the surrogate `id`; `(definition_id, serial)` is a *uniqueness constraint*, **not** a composite PK — `serial` is nullable (bulk lots have none, and PK columns can't be NULL) and one definition can legitimately hold **many** serial-less lots/batches (the whole point of the definition/instance split). So the pair uniquely identifies a *serialized* unit, while bulk instances are intentionally many-per-definition and are told apart by `id`.

### 3.6 The cross-FK ordering problem (why the bridge lands in step 4)

`locations.item_instance_id → stock_instances.id` and `stock_instances.location_id → locations.id` reference each other. Migration order:

1. **Step 1 (migration 0004)** creates `locations` **without** `item_instance_id` (instances don't exist yet).
2. **Step 4 (migration 0008)** creates `stock_instances` (its `location_id` FK is valid — `locations` already exists), then **adds `locations.item_instance_id` + its FK** via Alembic **batch mode** (`with op.batch_alter_table("locations") as batch: ...`), because SQLite cannot `ALTER TABLE ... ADD CONSTRAINT` directly (batch mode rebuilds the table).

This also matches the domain: you can't link a location to an instance before any instance exists.

### 3.7 Migration list

| Rev | Step | Creates |
|---|---|---|
| `0004` | 1 | `locations` (tree only) |
| `0005` | 2 | `categories` |
| `0006` | 3 | `item_kinds` (seeded) |
| `0007` | 3 | `item_definitions` |
| `0008` | 4 | `stock_instances` + batch-alter `locations` to add `item_instance_id` FK |

All reversible (downgrade drops in reverse; 0008 downgrade batch-drops the column first, then drops the table).

## 4. Backend design

### 4.1 Repository + service layering (extends M0)

- **Repositories** (`app/repositories/`): one per table — `LocationRepository`, `CategoryRepository`, `ItemKindRepository` (read-only: `list` + `get_by_code` for default resolution), `ItemDefinitionRepository`, `StockInstanceRepository`. Pure data access (get / list / create / update / delete + tree-specific reads like `get_children`, `get_descendants`). **No business rules** here; **no raw queries in routes**.
- **Services** (`app/services/` — first real use of this layer): hold the easy-to-get-wrong logic:
  - `TreeService` (shared base or mixin reused by location + category): cycle prevention on reparent; delete-guard (block non-empty); building the nested tree DTO.
  - `LocationService`: tree logic + container-as-item linking (set/clear `item_instance_id`, enforce its uniqueness and that the target instance exists).
  - `StockInstanceService`: enforce `serial ⇒ quantity = 1`; default `quantity = 1`; resolve `default_location_id` from the definition when a new instance omits a location.
- Routes depend on `get_authenticated_context` (carries household + user) and call services/repositories. The `RequestContext` is the seam for the future multi-tenant `household_id` scope.

### 4.2 API surface (all under `settings.api_prefix`, default `/api`; all authenticated)

Each resource: `GET` (list, with filters/search), `POST` (create), `GET /{id}`, `PATCH /{id}`, `DELETE /{id}`. Trees additionally expose a nested read.

| Method + path | Purpose |
|---|---|
| `GET /locations?q=&parent_id=` · `GET /locations/tree` | flat list / filtered / full nested tree |
| `POST/GET/PATCH/DELETE /locations[/{id}]` | CRUD (delete-guarded); `PATCH` can set `parent_id` (reparent, cycle-checked) and `item_instance_id` (container-as-item) |
| `GET /categories?q=&parent_id=` · `GET /categories/tree` · CRUD | category tree |
| `GET /kinds` | **read-only** list of the seeded kinds (for the Definition kind picker); no write endpoints in M1 |
| `GET /definitions?q=&category_id=` · CRUD | item definitions |
| `GET /instances?q=&definition_id=&location_id=` · CRUD | stock instances; `serial⇒qty=1` rejected with 422 |

Error contract: `409` for delete-guard / cycle / uniqueness conflicts; `422` for `serial⇒qty=1` and validation; `404` for missing ids; `401` for no session.

### 4.3 Pydantic schemas (`app/schemas/`)

Thin wire DTOs per resource: `*Create`, `*Update` (all-optional for PATCH), `*Response`, and `*TreeNode` (recursive, for the nested tree reads). `from_attributes = True` like M0's `UserResponse`. `item_kinds` gets a `KindResponse` only (read-only — no `Create`/`Update`).

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0 §5 (`make check` = ruff + mypy + pytest + eslint + tsc + vitest, all green; build passes). The **logic that must have unit tests** in M1 (roadmap DoD):

- **Tree cycle prevention**: reparenting a node under itself or a descendant is rejected.
- **Tree delete-guard**: deleting a non-empty location/category → 409; deleting a location with assigned instances or a container link → 409.
- **`serial ⇒ quantity = 1`**: rejected at the **service layer** (422) *and* proven at the **DB layer** (the `CHECK` raises `IntegrityError` on a direct bad insert).
- **Serial partial-uniqueness**: a duplicate `(definition_id, serial)` is rejected; the same serial under a *different* definition is allowed; two `NULL` serials coexist.
- **Container-as-item integrity**: `item_instance_id` is unique (one instance ↔ one location); linking a non-existent instance fails.
- **Default-location resolution**: creating an instance without a location falls back to the definition's `default_location_id` (and stays `NULL` if the definition has none).
- **Default-kind resolution**: creating a definition without a `kind_id` falls back to the seeded `durable` kind; an invalid `kind_id` is rejected.
- **Kind seed**: the `item_kinds` migration leaves exactly the three system kinds (`durable` / `consumable` / `perishable`) after `upgrade head`.
- **Migration round-trip**: `0004`–`0008` upgrade clean on an empty DB and downgrade cleanly (incl. the 0008 batch-alter on `locations`).

## 6. Contract-first codegen & no-drift gate

Unchanged mechanism from M0 §6: every step that touches the API **re-runs `make codegen`** (regenerates repo-root `openapi.json` + `frontend/src/api/schema.d.ts`) and commits both. The CI **contract** job (`make codegen` + `git diff --exit-code`) fails on drift. M1 simply grows the schema with the new resources; the frontend consumes the new typed paths via the existing `openapi-fetch` client.

## 7. Frontend design

### 7.1 Routing (new)

- Add **`react-router-dom`**. The M0 `App.tsx` gate (setup → login → authed) stays as the **outer** decision; when `authed`, render a `<BrowserRouter>` inside `AppShell`.
- Routes: `/` (dashboard placeholder), `/locations`, `/categories`, `/items` (definition list), `/items/:id` (definition detail + its instances), `/instances/:id` (instance detail). Nav links live in the existing `AppShell` sidebar/drawer.

### 7.2 Tree browse

- Mantine **`Tree`** for locations and categories: expand/collapse, select a node to see/edit it, create child / rename / reparent / delete (delete shows the 409 guard message when non-empty).
- A location node visibly indicates when it is container-as-item (links to the instance).

### 7.3 Definitions & instances

- **Definition**: list (with `q` search + category filter), create/edit form (name, description, category picker, kind, unit, default location), detail page listing the instances of that definition.
- **Instance**: create/edit form (definition picker, location picker, quantity, serial, model, manufacturer, warranty_expires, warranty_details, purchase price/date/source). **Client-side `serial ⇒ quantity = 1`** validation (disable/auto-set quantity to 1 when a serial is entered) **plus** surfacing the server's 422 if it slips through. Search by `q`.

### 7.4 Tests (vitest + Testing Library)

Tree render + node CRUD (client mocked); definition form submit; instance form `serial⇒qty=1` client rule; list search renders; routing renders the right page. Keep to the M0 "mock the typed client" style.

## 9. Step breakdown (atomic, ordered)

Each step is independently testable, test-backed, lands exactly one commit (one per-step autosquash in orchestrator mode), re-runs `make codegen` if it touched the API, and inherits the global DoD (§5) + the **anti-freelancing rule**: implement *only* this step — no other steps, no opportunistic refactors, no touching the host's real environment.

> **Ordering rationale:** FKs dictate order — `categories`/`locations`/`item_kinds` before `item_definitions` (it FKs all three) before `stock_instances` (FKs definition + location); the container-as-item bridge waits for `stock_instances` (§3.6). Backend (1–4) lands the typed contract the frontend (5–6) consumes.

### Step 1 — Location self-referential tree (backend)
- **Goal:** a nested location tree with CRUD, cycle prevention, and delete-guard — establishing the reusable **tree repo/service pattern**.
- **Build:** `models/location.py` (`id, name, description, parent_id, created_at` — **no** `item_instance_id` yet); migration `0004`; `LocationRepository` + tree-aware `LocationService` (cycle check on reparent, delete-guard); `schemas/location.py` (`Create/Update/Response/TreeNode`); `api/routes/locations.py` (CRUD + `/tree` + `?q=&parent_id=`); register router; `make codegen`.
- **Tests:** cycle rejected (self + descendant); reparent OK; delete-guard 409 on non-empty; tree DTO shape; `q` search; migration 0004 up/down.
- **Out of scope:** no container-as-item bridge (step 4); no other tables.
- **Commit:** `feat(backend): location self-referential tree with CRUD`

### Step 2 — Category self-referential tree (backend)
- **Goal:** the category tree, reusing step 1's tree pattern.
- **Build:** `models/category.py`; migration `0005`; `CategoryRepository` + service (cycle + delete-guard via the shared tree logic); `schemas/category.py`; `api/routes/categories.py` (CRUD + `/tree` + `?q=&parent_id=`); `make codegen`.
- **Tests:** mirror step 1 (cycle, delete-guard, tree, search, migration).
- **Out of scope:** definitions/instances.
- **Commit:** `feat(backend): category self-referential tree with CRUD`

### Step 3 — Item kinds lookup + Item Definition (backend)
- **Goal:** the seeded `item_kinds` lookup + "what kind of thing" CRUD.
- **Build:**
  - `models/item_kind.py` (`id, code unique, name, is_system, created_at`); migration `0006` (creates `item_kinds` and **seeds** `durable`/`consumable`/`perishable` via `INSERT OR IGNORE`); `ItemKindRepository` (read-only `list` + `get_by_code`); `schemas/item_kind.py` (`KindResponse` only); `api/routes/kinds.py` (**`GET /kinds`** read-only).
  - `models/item_definition.py` (`name, description, category_id, kind_id FK, unit, default_location_id, created_at`); migration `0007`; `ItemDefinitionRepository` + service (resolve the `durable` kind when `kind_id` omitted); `schemas/item_definition.py`; `api/routes/definitions.py` (CRUD + `?q=&category_id=`); `make codegen`.
- **Tests:** `GET /kinds` returns exactly the three seeded kinds; definition create/read/update/delete; default-kind resolution when `kind_id` omitted; invalid `kind_id` rejected; FK to category/location; `q` search; migrations 0006 (incl. seed) + 0007 up/down.
- **Out of scope:** `min_stock`/`default_best_before_days` (M2/M3); kinds CRUD / behaviour flags (deferred §12); instances.
- **Commit:** `feat(backend): item kinds lookup and item definition CRUD`

### Step 4 — Stock Instance + container-as-item bridge (backend)
- **Goal:** "this specific unit/lot" CRUD with the serial constraint, plus wiring container-as-item.
- **Build:** `models/stock_instance.py` (all §3.5 fields; `CHECK(serial IS NULL OR quantity=1)`; partial unique `(definition_id, serial)`); migration `0008` (create `stock_instances`, then **batch-alter `locations`** to add `item_instance_id` FK, unique); add `item_instance_id` to the `Location` model; `StockInstanceRepository`; `StockInstanceService` (`serial⇒qty=1`, default-location resolution); extend `LocationService` with container-as-item link/unlink; `schemas/stock_instance.py`; `api/routes/instances.py` (CRUD + `?q=&definition_id=&location_id=`); extend the location route/schema to set `item_instance_id`; `make codegen`.
- **Tests (the easy-to-get-wrong logic — required):** `serial⇒qty=1` rejected at **service (422)** and **DB (IntegrityError)**; partial-uniqueness (dup rejected, cross-definition allowed, NULLs coexist); default-location fallback; container-as-item uniqueness + non-existent-instance failure + location delete-guard when linked/occupied; migration 0008 up/down incl. the batch-alter.
- **Out of scope:** no ledger/quantity-derivation (M2); no best_before (M3); no warranty scanning (M4).
- **Commit:** `feat(backend): stock instance registry with serial⇒qty=1 and container-as-item`

### Step 5 — Frontend foundation: router + nav + tree browse
- **Goal:** routing + the locations & categories tree UIs.
- **Build:** add `react-router-dom`; mount `<BrowserRouter>` inside the authed `AppShell`; real nav links; `pages/Locations.tsx` + `pages/Categories.tsx` using Mantine `Tree` (browse + create child / rename / reparent / delete with 409 messaging); show a location's container-as-item link.
- **Tests:** tree renders + node CRUD (client mocked); routing shows the right page; delete-guard message renders.
- **Out of scope:** definition/instance pages (step 6).
- **Commit:** `feat(frontend): router, nav, and location/category tree browse`

### Step 6 — Frontend: definitions + instances (list / detail / CRUD / search)
- **Goal:** register & browse durables end-to-end in the UI.
- **Build:** `pages/Items.tsx` (definition list + `q` + category filter), definition create/edit form + detail (lists its instances); instance create/edit form with **client-side `serial⇒qty=1`** + server-422 surfacing; instance detail; search.
- **Tests:** definition form submit; instance `serial⇒qty=1` client rule; list search; instance detail renders (client mocked).
- **Out of scope:** photos/tags/custom fields (M5).
- **Commit:** `feat(frontend): durable definitions and instances with search`

> Step 6 may split into **6a (definitions)** / **6b (instances)** if it runs large; keep each independently green.

## 10. Blind-review checkpoints (per step)

Reviewer gets **only**: this doc + roadmap, the step's impl brief, and the step's diff. Check:

- **Step 1:** cycle prevention is in the **service layer** (no SQL trigger); delete-guard blocks non-empty; no raw queries in routes; migration reversible; `/tree` DTO is correct; codegen committed.
- **Step 2:** genuinely reuses the step-1 tree pattern (no copy-paste divergence); same guards hold.
- **Step 3:** `item_kinds` seeded with exactly the three system kinds; `kind_id` is a real FK (no leftover string-enum/CHECK); `GET /kinds` is read-only (no write endpoints); default-kind resolution works; other FKs correct; `min_stock`/`default_best_before_days` **absent** (not smuggled in early); codegen committed.
- **Step 4:** `serial⇒qty=1` in **both** DB `CHECK` and service; partial-unique index correct; **batch-alter** used for the `locations` FK (no broken SQLite ALTER); container-as-item uniqueness enforced; quantity is a **plain stored column** (no premature ledger); codegen committed.
- **Step 5:** router mounted **inside** the authed shell (M0 gate intact); one tree component reused; cookie-only auth unchanged (nothing in localStorage).
- **Step 6:** client-side `serial⇒qty=1` **and** server-422 both handled; search wired; typed client used (no hand-written fetch).

## 12. Open questions / deferred

- **Serial uniqueness scope:** M1 uses **per-definition** partial-unique `(definition_id, serial)`; the PK stays the surrogate `id` (see §3.5). Revisit if a cross-product global serial registry is ever wanted (unlikely for a household).
- **Tree delete semantics (cascade is wanted later — confirmed):** M1 **blocks** delete of non-empty nodes (safe default). A future iteration **will** support deletion of populated nodes — but the chosen direction is **soft-delete / archive** and/or **cascade *with confirmation*** (showing "this will delete N tracked assets"), **not** a silent hard cascade, because cascading a location delete would wipe every asset inside it. Revisit with M5/M6.
- **Qualitative stock levels (forward requirement — affects M2 design):** some items (esp. small parts) can't be counted precisely; track them as `high` / `medium` / `low` instead of a number. Plan a per-**definition** *stock-tracking mode* when designing M2: `exact` (Decimal + the M2 ledger; low-stock = below `min_stock`) · `level` (qualitative enum, manually set, **no ledger**; low-stock = level is `low`) · `none` (presence-only, no quantity). The M2 ledger applies to `exact` mode only. **M1 needs no change** — `quantity` stays a plain stored column; this column/mode is added in M2.
