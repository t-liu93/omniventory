# M1.5 — 国际化(i18n)基础

> 🌐 **语言:** [English](./M1.5.md) · 中文(当前)

> **里程碑设计文档 — 自包含。** 与 `docs/plan/roadmap.md`(总图,尤其 §5 M1.5 + §2.6 线缆/展示分离)一起读;“我们为何存在”见 `docs/inspiration/investigation.md`。本文是 *M1.5 构建什么、如何验证* 的唯一事实来源,**不要**从 roadmap 重新推导范围。进度**只**在 roadmap §4 表中追踪。

> 内建约定:原子步骤(§9)、盲评检查点(§10)、🟢 部署自测点(§11),使手动与编排两种执行方式都能挂靠本文。

## 1. 目标与非目标

**目标(roadmap 的 M1.5 承诺):** 在更多界面堆积**之前**就把 UI 做成**中英双语(ZH + EN)**——一项跨切面的基础能力,提早完成,好让后续每个里程碑(M2 流水账、M3 保质期、M4 提醒、……)**从一开始就通过 i18n 写文案**,而不是硬编码英文、事后回填。M1.5 还把**线缆/展示分离**钉死:后端只说**稳定错误码**,前端**独占**所有面向人的文本。

**完成判据(🟢,§11 展开):** 运行时切换语言(ZH ⇄ EN),整个 UI 跟随;选择**随账号持久化**、跨会话/跨设备;登录/初始化页在鉴权**之前**就以解析出的语言渲染;后端错误经由其**错误码本地化显示**;日期/数字按当前 locale 渲染。CI 保持绿色,包含无漂移契约门禁。

**非目标(明确排除 M1.5——推迟或归属后续里程碑):**
- **不加新业务功能。** M1.5 是跨切面的*能力*层;只新增**一个** DB 列(`users.preferred_language`),**零**新业务表。
- **后端不做本地化。** 后端绝不向终端用户返回面向人的展示串;只返回**码**(外加给日志用的、面向开发者的英文 `message`)。对 Pydantic 校验信息逐字段本地化**不在**范围内(M1.5 把*整个*校验失败映射成一个码;见 §4.1)。
- **不加第三种及以上语言。** 恰好 **EN + ZH**。目录/结构必须让“加一种语言”是*内容*任务(丢进一个 locale 文件),而非*代码*任务——但 M1.5 不交付任何其他语言。
- **不做翻译管理工具/抽取 CI。** 没有 Crowdin/Weblate,没有未用 key 的自动 lint(§12 记录)。
- **不做家庭级默认语言。** 解析是账号绑定 + 客户端的(§4.3);工作区级默认推迟(§12)。
- **不做用户输入的 locale 感知*解析***(例如输入 `1,5` 当 `1.5`)。M1.5 只做 locale 感知**渲染**(用 Intl 输出);输入保持规范形。

## 2. 锁定的决策(M1.5 规划期确定;理据见 roadmap §2.6 + 规划讨论)

| 方面 | 决策 |
|---|---|
| **库** | **`react-i18next`**(roadmap 已锁)基于 **`i18next`**,加 **`i18next-browser-languagedetector`** 实现登录前探测链。新前端依赖(收尾时记录;§12)。 |
| **语言与码** | 恰好 **`en`** 与 **`zh`**(规范短码)。`navigator.language` 形如 `zh-CN`/`zh-TW`/`zh-Hans` 归一为 `zh`;其余回退 `en`。存储列足够宽(`String(16)`)以便日后**无需迁移**即可容纳 BCP-47 子标签——但 M1.5 只存 `en`/`zh`。 |
| **错误契约**(重头戏) | **彻底干净重构,而非加性兜底。** 1.0 前无生产数据 → 我们把 FastAPI 默认的 `{"detail": ...}` 体替换为**统一的扁平错误信封** `{ "code", "message", "params"? }`,覆盖**每一条**错误路径(领域 `AppError`、漏网的 `HTTPException`、Pydantic `RequestValidationError`)。`code` 是前端用于本地化的稳定机器键;`message` 是**面向开发者的英文**(日志/curl/未知码兜底),**绝不**原样给终端用户看。扁平(不嵌在 `detail`/`error` 下),让调用处用 `openapi-fetch` 直接读 `result.error.code`。 |
| **文本归属** | **所有**面向人的串都在**前端** i18n 目录里。后端不发任何展示文本。两者之间唯一的接缝是**错误码表**(§4.2)——码即契约,改名只能通过 FE+BE 协同改动 + codegen。 |
| **`preferred_language` 存储** | 可空的 `users.preferred_language`(`String(16)`)。**`NULL` = “从未显式选择”** → 前端走登录前解析链(于是在 `zh` 浏览器里完成初始化的新管理员仍看到 ZH)。具体取值是**权威的**,随账号跨设备。不强制 `'en'` 默认(那会覆盖浏览器意图)。 |
| **读/写端点** | 读经由既有 **`GET /api/auth/me`**(现含 `preferred_language`);写经由 **`PATCH /api/auth/me`**,体 `{ preferred_language }`,返回更新后的 `MeResponse`。一个通用的“当前用户偏好”补丁——M6 扩展更多字段,不必新端点。 |
| **语言校验** | 支持集在**应用层**强制(`SUPPORTED_LANGUAGES = ("en", "zh")`,位于 `app/core/languages.py`);不支持的码 → `AppError("validation.unsupported_language", 422)`。不设 DB CHECK(集合会增长;按 roadmap §2.11 留在应用代码里)。 |
| **解析分层** | **已登录:** `me.preferred_language` 非空时权威;为 `NULL` 时落到客户端链。**登录前:** localStorage 显式选择 → `navigator` 探测 → 回退 **`en`**。(roadmap §5 M1.5。)详见 §4.3。 |
| **格式化范围** | M1.5 **包含**经由薄 **`Intl`** 包装、绑定当前语言的日期与数字 locale 感知**渲染**(`src/i18n/format.ts`)。既有 `formatQuantity`(去尾零——见提交 `18f9ec3`)并入此包装,**保留**该行为同时变为 locale 感知。 |
| **测试语言固定** | 测试环境把 i18n **固定为 `en`**(确定性),让抽取后既有的“匹配英文文案”断言仍可用;语言切换行为通过强制 `zh` 显式测试。 |

> 这些扩展 M0/M1 基础。**错误契约重构是对 API 形状的真正跨切面改动**,故受无漂移契约门禁约束(§6),并按重构标准评审(§10)。`react-i18next` 等是新前端依赖——在 M1.5 报告中记录;收尾时若作者想让清单详尽,`AGENTS.md` “技术栈”行可加 `react-i18next`。

## 3. 数据模型(极简——一列)

M1.5 新增**一**列;无新表。

### 3.1 `users.preferred_language`(迁移 `0009`)

| 列 | 类型 | 可空 | 说明 |
|---|---|---|---|
| `preferred_language` | `String(16)` | **是** | `NULL` = 从未显式选择(→ 客户端解析)。M1.5 的具体取值:`'en'` / `'zh'`。在**应用层**对 `SUPPORTED_LANGUAGES` 校验;无 DB CHECK(集合会增长)。 |

- 经 Alembic 加到既有 `users` 表。SQLite 的 `ALTER TABLE ... ADD COLUMN` 加可空列**无需**批量重建——`op.add_column` 即可;降级是 `op.drop_column`(SQLite 上对 drop 用 batch 模式)。
- 迁移**不回填**(既有行保持 `NULL` = 继承客户端解析)。1.0 前本就无数据。

> M1.5 不做家庭级 `default_language`(推迟,§12)。解析链(§4.3)以“浏览器探测 → `en`”覆盖未配置情形。

## 4. 后端设计

### 4.1 统一错误码契约(重构)

**现状**每条错误都是 `raise HTTPException(detail="<英文串>")`,被 FastAPI/Starlette 序列化为 `{"detail": ...}`。M1.5 整体替换之。

**新核心模块 `app/core/errors.py`:**
- `class AppError(Exception)` —— 携带 `status_code: int`、`code: str`、`params: dict | None = None`、`message: str | None = None`(面向开发者的英文;默认按码取通用串)。今后服务/路由都 raise 它。
- `class ErrorResponse(BaseModel)` —— 线缆模式:`code: str`、`message: str`、`params: dict | None = None`。这是**被记录**的错误体(在 OpenAPI 中声明,使契约被类型化并被门禁检查,§6)。
- **三个异常处理器**在 `app.main.create_app` 注册,全部输出**同一** `ErrorResponse` 形状:
  1. `AppError` → `{code, message, params}`,状态码取 `exc.status_code`。
  2. `StarletteHTTPException`(漏网的 `HTTPException`,如 `main.py` 的 SPA 回退 404,或尚未迁移者)→ `{code: "http.<status>", message, params: null}`,使未迁移路径也遵守信封。
  3. `RequestValidationError`(Pydantic 422)→ **单一**码 `validation.invalid_input`,`message` 取通用串,`params: { "fields": [ {loc, type} … ] }`(机读字段列表;前端可提示“检查这些字段”,但**不**逐条本地化 Pydantic 信息)。另有 `500` 安全网处理器输出 `internal.error`,不泄露内部细节。

**迁移每一处既有 raise**(`api/deps.py`、`api/routes/auth.py`、`services/{tree,location,category,item_definition,stock_instance}.py`)从 `HTTPException(detail=…)` 改为 `AppError(code=…, params=…)`,把旧英文串保留为 `message`。这是 Step 1 的主体。

### 4.2 错误码表(FE↔BE 契约)

码为 **`<resource>.<reason>`**,稳定、机读。该表是唯一事实来源(以枚举/常量形式留在 `app/core/errors.py`;前端在 `errors` 命名空间下镜像为 i18n key)。代表性集合(实现者须给**每一处**既有 raise 赋码;本表是骨架,非穷举):

| 码 | 状态 | 当前由谁 raise | `params` |
|---|---|---|---|
| `auth.not_authenticated` | 401 | `deps`(无会话) | — |
| `auth.session_invalid` | 401 | `deps`(会话过期/无效) | — |
| `auth.account_inactive` | 401 | `deps`(用户不存在/停用) | — |
| `auth.invalid_credentials` | 401 | `auth.login` | — |
| `auth.account_disabled` | 401 | `auth.login` | — |
| `auth.setup_already_complete` | 409 | `auth.setup` | — |
| `validation.invalid_input` | 422 | Pydantic 请求校验 | `{fields}` |
| `validation.unsupported_language` | 422 | `PATCH /auth/me`(新,Step 2) | `{value, supported}` |
| `tree.cycle` | 409 | `tree` 改父(自身/后代) | `{kind}` |
| `tree.delete_has_children` | 409 | `tree` 删非空 | `{kind}` |
| `location.not_found` | 404 | `location`、`definition`、`instance` | `{id}` |
| `location.parent_not_found` | 404 | `location` | `{id}` |
| `location.delete_in_use` | 409 | `location` 删除时仍有实例或容器链接 | `{id}` |
| `location.container_link_conflict` | 409 | `location` 容器即物品唯一性 | `{id}` |
| `category.not_found` / `category.parent_not_found` | 404 | `category`、`definition` | `{id}` |
| `item_kind.not_found` | 422 | `definition`(坏 `kind_id`) | `{id}` |
| `item_definition.not_found` | 404 | `definition`、`instance` | `{id}` |
| `item_definition.name_conflict`* | 409 | `definition`(若存在唯一性规则) | `{name}` |
| `stock_instance.not_found` | 404 | `instance`、`location` 链接 | `{id}` |
| `stock_instance.serial_requires_qty_one` | 422 | `instance` 服务(`serial⇒qty=1`) | — |
| `stock_instance.serial_duplicate` | 409 | `instance`(重复 `(definition_id, serial)`) | `{serial}` |
| `http.<status>` | * | 未迁移 `HTTPException` 兜底 | — |
| `internal.error` | 500 | 安全网处理器 | — |

\* 在 Step 1 对照 `services/item_definition.py` 的实际 raise 核实;按那里既有 409 的含义赋码。规则是**每个不同失败一个码**,而非每个调用点。

### 4.3 `preferred_language` 端点与解析

- **`GET /api/auth/me`** —— `UserResponse` 增 `preferred_language: str | None`;`MeResponse` 原样包裹。(避免二次往返:gate 随用户一并拿到语言。)
- **`PATCH /api/auth/me`** —— 体 `UserPreferencesUpdate { preferred_language: str | None }`(全可选,PATCH 风格)。对 `SUPPORTED_LANGUAGES` 校验;经 `UserRepository` 持久化(无裸查询);返回更新后的 `MeResponse`。需有效会话(`get_current_user`)。允许置 `null`(显式“取消设置” → 重新继承客户端解析)。
- **`SUPPORTED_LANGUAGES = ("en", "zh")`** 位于 `app/core/languages.py`;校验器(及日后若暴露的发现端点)都读它。M1.5 **不**经 API 暴露该集合——前端硬编码同一清单;保持同步是 2 行的事,§12 标注若清单增长则加 `/api/i18n/languages`。

> **解析(值在何处生效——前端,§7.3):** 已登录 → `me.preferred_language` 非空则权威,否则客户端链;登录前 → localStorage 显式选择 → `navigator` 探测(归一)→ `en`。

## 5. 质量门禁与易错逻辑(完成定义)

继承 M0 §5 / M1 §5(`make check` = ruff + mypy + pytest + eslint + tsc + vitest 全绿;构建通过;`make codegen` 无漂移)。M1.5 中**必须有单测**的逻辑:

**后端**
- **错误信封一致性:** 代表性 `AppError`(如 `location.not_found`)在正确状态返回 `{code, message, params}`;Pydantic 校验失败返回 `validation.invalid_input` 带 `fields` 列表;漏网 `HTTPException`(SPA 404)返回 `http.404`。**没有**端点再返回旧的裸字符串 `detail`。
- **码稳定性:** 迁移后的 auth 错误带准确码(`auth.invalid_credentials`、`auth.setup_already_complete` 等);保留防用户枚举(登录“邮箱不存在”与“密码错误”都 → `auth.invalid_credentials`)。
- **`preferred_language`:** 默认 `NULL`;`me` 返回它;`PATCH` 持久化 `en`/`zh`/`null`;不支持的码 → `validation.unsupported_language`(422)且**不**写入。
- **迁移往返:** `0009` 在 `0008` 之上干净升级、干净降级(列被删)。

**前端**
- **解析优先级:** localStorage 选择 > `navigator` > `en`;`zh-CN`/`zh-TW`/`zh-Hans` → `zh`;未知浏览器 locale → `en`。
- **切换即时且彻底:** 改语言后各类界面样本(一个登录前页 + 一个已登录页)重渲染为目标语言;`<html lang>` 更新。
- **账号持久化:** 已登录切换调用 `PATCH /api/auth/me`;gate 在加载时应用非空 `me.preferred_language`。
- **错误本地化:** 已知码 → 带 `params` 插值的本地化串;**未知**码 → 通用本地化兜底(绝不把原始英文 `message` 给用户);`serial⇒qty=1`(422)与删除守卫(409)都本地化渲染。
- **目录完整性:** `en` 与 `zh` 拥有**相同 key 集**(一个单测断言两者无缺失/多余 key——廉价的防漂移)。
- **格式化:** 日期/数字按当前 locale 渲染;`formatQuantity` 仍去尾零(保留 `18f9ec3`)且 locale 感知。

## 6. 契约优先 codegen 与无漂移门禁

机制不变(M0 §6)。M1.5 改了**错误响应形状**,并增 `preferred_language` + `PATCH /api/auth/me`,所以**每个触及 API 的步骤都重跑 `make codegen`** 并提交 `openapi.json` + `frontend/src/api/schema.d.ts`。因错误信封现在是**已声明的 `ErrorResponse` 组件**(经共享 `responses=` 助手挂到路由上),契约会记录它,前端可类型化错误分支(`result.error` → `ErrorResponse`)。CI **contract** 作业一如既往在漂移时失败。

## 7. 前端设计

### 7.1 i18n 架构(`src/i18n/`)

```
src/i18n/
├── index.ts          # i18next 初始化:detector、fallback、namespaces、<html lang> 同步
├── languages.ts      # SUPPORTED_LANGUAGES = ['en','zh'];normalize(navigatorLang) → 'en'|'zh'
├── format.ts         # Intl 包装:formatDate / formatNumber / formatQuantity(绑定 locale)
├── errors.ts         # mapApiError(error) → 经 errors 命名空间本地化的串
└── locales/
    ├── en/{common,auth,nav,locations,categories,items,instances,errors}.json
    └── zh/{common,auth,nav,locations,categories,items,instances,errors}.json
```

- **初始化**(`index.ts`):`i18next.use(LanguageDetector).use(initReactI18next).init({ supportedLngs:['en','zh'], fallbackLng:'en', ns:[…], detection:{ order:['localStorage','navigator'], lookupLocalStorage:'omniventory_lang', caches:['localStorage'] }, load:'languageOnly' })`。`load:'languageOnly'` 把 `zh-CN`→`zh`。订阅 `languageChanged` → 设 `document.documentElement.lang`。
- **Provider:** i18next 全局;在根处包裹一次(`main.tsx` 里 `<I18nextProvider>`,或依赖单例 + `useTranslation`)。测试环境引同一初始化但**固定 `lng:'en'`**(§2)。
- **命名空间**让目录内聚,也让后续里程碑按功能加命名空间而非一个巨文件。

### 7.2 字符串抽取(Step 4 —— UI 工作主体)

把**每一个**硬编码的人读串替换为 `t(...)`,并填充 `en` 与 `zh`,覆盖既有界面:
`pages/{Login,Setup,Dashboard,Locations,Categories,Items,InstanceDetail}.tsx`、`shell/AppShell.tsx`(导航标签、登出、配色切换)、`components/{TreeBrowser,InstanceFormModal,EmptyState,ErrorState,LoadingState,PageShell}.tsx`。含按钮文案、占位符、标题、`aria-label`、空/错/加载文案、确认对话框。`en` 串就是既有文案的逐字(让测试断言存活);`zh` 是忠实翻译。

### 7.3 语言解析与切换

- **Gate 接线(`App.tsx`):** `GET /auth/me` 解析后,若 `user.preferred_language` 非空 → `i18n.changeLanguage(它)` **并**写 localStorage(使 `me` 重解析前的刷新仍显示正确语言)。为空则保留 detector 的选择。登录前页本就经 detector 链渲染(localStorage → navigator → `en`)。
- **`<LanguageSwitcher>`**(一个共享组件):紧凑的 EN/中文 切换。
  - **登录前**(Setup/Login):仅 `changeLanguage` + localStorage(无会话可持久化)。
  - **已登录**(AppShell 头部):`changeLanguage` + localStorage **并** `PATCH /api/auth/me { preferred_language }`,使其随账号到其他设备。PATCH 失败不致命(本地改动仍生效;经错误映射器弹 toast)。

### 7.4 错误本地化(`errors.ts`,Step 5)

`mapApiError(error: ErrorResponse | unknown): string`:
1. 防御性读取 `error.code`(+ `error.params`)。
2. 若码存在于 `errors` 命名空间 → `t('errors.'+code, params)`。
3. 否则 → `t('errors.generic')`(本地化的“出错了”);在 `import.meta.env.DEV` 下追加原始 `message` 便于调试。

把它接进当前显示英文文本或裸 `error.detail` 的调用处:`Login`(凭据无效)、`Setup`(409)、`TreeBrowser`(环/删除守卫/容器冲突)、`InstanceFormModal`(`serial⇒qty=1` 422)。它们不再读 `error.detail`,改调 `mapApiError(error)`。

### 7.5 Locale 格式化(`format.ts`,Step 7)

绑定 `i18n.language` 的薄 `Intl` 包装:`formatDate(d)`(`Intl.DateTimeFormat`)、`formatNumber(n)`(`Intl.NumberFormat`),以及 `formatQuantity(q)`——**保留**既有去尾零(提交 `18f9ec3`),同时应用该 locale 的小数分组/分隔符。替换当前 `utils.ts` 的 `formatQuantity` 与各处零散的 `created_at`/日期渲染(如 `InstanceDetail`、`Items`)。

### 7.6 测试(vitest + Testing Library)

## 9. 步骤拆解(原子、有序)

每步可独立测试、有测试支撑、恰好落一个提交(编排模式下每步一次 autosquash),触及 API 则重跑 `make codegen`,并继承全局 DoD(§5)+ **反夹带规则**:只实现*本步*——不做其他步骤、不顺手重构、不碰宿主真实环境。

> **排序理据:** 后端契约(1–2)必须先落,前端才能消费新的类型化错误形状 + `preferred_language` 路径。前端内部:i18n 基建(3)→ 字符串抽取(4)→ 错误映射(5,需 Step 1 的码 + Step 3 的基建)→ 切换器/持久化(6,需 Step 2 的端点 + Step 3)→ locale 格式化(7,需 Step 3)。

### Step 1 — 后端:统一错误码契约(重构)
- **目标:** 把处处的 `{"detail": str}` 替换为扁平 `ErrorResponse { code, message, params }` 信封,经 `AppError` + 三个异常处理器;把每一处既有 raise 迁移到稳定码。
- **构建:** `app/core/errors.py`(`AppError`、`ErrorResponse`、§4.2 码表);在 `app/main.create_app` 注册 `AppError` / `StarletteHTTPException` / `RequestValidationError`(+ 500 安全网)处理器;迁移 `api/deps.py`、`api/routes/auth.py`、`services/{tree,location,category,item_definition,stock_instance}.py` 中所有 raise;经共享 `responses={401,404,409,422: ErrorResponse}` 助手挂到路由,使 OpenAPI 记录该信封;`make codegen`。
- **测试:** 代表性 `AppError` → 正确 `{code,status,params}`;Pydantic 422 → `validation.invalid_input` + `fields`;漏网 `HTTPException` → `http.404`;auth 码准确 + 防用户枚举保留;不再有遗留裸串 `detail`(对各状态样本断言形状)。
- **范围外:** `preferred_language`(Step 2);任何前端改动。
- **提交:** `refactor(backend): uniform error-code contract for i18n`
- *(若过大可拆成 1a 基建+处理器 / 1b 迁移 raise 点;各自保持绿。)*

### Step 2 — 后端:`preferred_language` 字段 + `me` 读/写
- **目标:** 每用户语言偏好,`GET /auth/me` 读,`PATCH /auth/me` 写。
- **构建:** 模型上 `users.preferred_language`(可空 `String(16)`);迁移 `0009`(加列;可逆);`app/core/languages.py` 的 `SUPPORTED_LANGUAGES`;`UserResponse` 增 `preferred_language`;`UserPreferencesUpdate` 模式;`PATCH /api/auth/me`(对集合校验 → `validation.unsupported_language`;经 `UserRepository` 持久化;返回 `MeResponse`);`make codegen`。
- **测试:** 默认 `NULL`;`me` 返回该字段;`PATCH` 持久化 `en`/`zh`/`null`;不支持的码 → 422 `validation.unsupported_language` 且不写;迁移 0009 升/降。
- **范围外:** 任何前端消费(Step 3–6)。
- **提交:** `feat(backend): per-user preferred_language with me read/update`

### Step 3 — 前端:i18n 基建 + 语言解析
- **目标:** 立起 `react-i18next`,含探测链、命名空间、`<html lang>` 同步、支持语言助手——尚不抽取字符串。
- **构建:** 加 `i18next`、`react-i18next`、`i18next-browser-languagedetector`;`src/i18n/{index.ts,languages.ts}`;初始化(detector 顺序 localStorage→navigator、`fallbackLng:'en'`、`load:'languageOnly'`、命名空间);接入 `main.tsx`;播种空/近空的 `en`+`zh` 命名空间文件(结构);测试环境固定 `en`。
- **测试:** 解析优先级(localStorage > navigator > `en`);`zh-CN`/`zh-TW`→`zh`;未知 → `en`;`changeLanguage` 更新 `<html lang>`。
- **范围外:** 翻译既有串(Step 4);错误映射(Step 5);切换器(Step 6);格式化(Step 7)。
- **提交:** `feat(frontend): react-i18next scaffolding and language resolution`

### Step 4 — 前端:抽取全部 M0–M1 UI 字符串(en + zh)
- **目标:** 每个既有硬编码人读串走 `t(...)`;`en`+`zh` 目录填满。
- **构建:** 替换所有页面/外壳/组件中的串(§7.2);填充 `locales/en/*`(逐字既有文案)+ `locales/zh/*`(忠实翻译);加**目录 key 对齐**测试。

- **范围外:** 错误码映射(Step 5)、切换器/持久化(Step 6)、格式化(Step 7)。

### Step 5 — 前端:经稳定码本地化后端错误
- **目标:** 后端错误码渲染为本地化文本;不让原始英文 `detail`/`message` 触达用户。

### Step 6 — 前端:语言切换器 + 账号绑定持久化
- **目标:** 一个运行时切换器(登录前 + 已登录),持久化到账号,并被 gate 应用。

- **测试:** 实时切换让一个登录前页 + 一个已登录页重渲染为 `zh`;已登录切换 PATCH(client mock);gate 应用 `me.preferred_language`;登录前切换仅写 localStorage。

### Step 7 — 前端:locale 感知的日期与数字格式化
- **目标:** 日期/数字按当前 locale 渲染;`formatQuantity` 保去尾零并变 locale 感知。

- **范围外:** 输入的 locale 感知*解析*(§1 非目标)。

## 11. 🟢 部署自测点(并入 M1.5 里程碑手动走查)

作者在里程碑末运行的手动走查(展开 roadmap M1.5 🟢 项)。假定 M0/M1 运行流程(compose up;migrate 服务已应用 `0001`–`0009`):

1. **登录前语言:** 用主语言为中文的浏览器(或无先前选择)打开应用 → **Setup/Login 页以 ZH 渲染**(浏览器探测 → `zh`)。该页可见 `<LanguageSwitcher>`。
2. **鉴权前切换:** 在登录页切 EN ⇄ 中文 → 整页即时跟随;刷新 → 选择**持久**(localStorage)。`document.documentElement.lang` 匹配。
3. **账号绑定持久化:** 登录;在头部切语言 → 即时跟随**并**存到账号(`PATCH /api/auth/me`)。登出,**换一个浏览器**(全新 localStorage)登录 → UI 以**账号的**语言出现(对浏览器探测有权威性)。
4. **整个 UI 跟随:** ZH 激活时,浏览 Locations/Categories 树、Items、某实例详情、打开实例表单 → **所有**标签、占位符、按钮、空/错状态都已翻译。
5. **后端错误经码本地化:** 触发删除守卫(删非空 location → 409 `tree.delete_has_children`)与 `serial⇒qty=1` 规则(给序列化实例设数量 3 → 422 `stock_instance.serial_requires_qty_one`)→ 都显示**本地化**信息(ZH 激活时为中文),而非英文服务端文本。一次错误登录显示本地化的 `auth.invalid_credentials`。
6. **Locale 格式化:** 日期(`created_at`、保修/购买日期)与数量按当前 locale 格式渲染;小数数量仍无尾零。
7. **CI 绿:** 同套门禁在 GitHub Actions 通过,含无漂移契约门禁;迁移 `0001`–`0009` 在 docker 冒烟测试中干净应用到全新 DB。

## 12. 待定问题 / 推迟

- **输入的 locale 感知解析:** M1.5 经 `Intl` 渲染但**不**解析本地化输入(如 `1,5` → `1.5`、本地化日期)。当表单接受 locale 格式数字时再议(很可能随 M2 数量录入或 M8 多单位)。
- **逐字段校验本地化:** M1.5 把整个 Pydantic 422 映射成单一 `validation.invalid_input`(+ `fields` 列表)。逐字段/逐约束信息本地化(如“邮箱必填”)推迟——表单获得更丰富的内联校验时再议。
- **家庭级默认语言:** M1.5 无工作区级默认(解析为账号绑定 + 客户端)。若多用户(M6)让其值得,可加 `households.default_language`(为新账号播种 / 登录前默认)。
- **经 API 暴露 `SUPPORTED_LANGUAGES`:** FE+BE 各自硬编码 `['en','zh']`。若清单增长,加 `GET /api/i18n/languages`(或并入 `/api/health`),让前端动态发现而非重复。
- **第三种及以上语言 / 翻译工具:** 加语言按设计是*内容*任务(丢入一个 locale 目录),但未设 TMS(Crowdin/Weblate)或未用/缺失 key 的抽取 CI。有贡献者翻译时再加。
- **`AGENTS.md` 技术栈行:** ✅ 收尾时已处理——前端行现已列出 `react-i18next`(+ `i18next-browser-languagedetector`),并记录了 wire/display 切分。
- **数字/日期 locale 边角:** `zh` 的 `Intl` 输出(如 CJK 日期顺序、M4 显示金额后的货币符号位置)——金额/日期变显著时再核。
