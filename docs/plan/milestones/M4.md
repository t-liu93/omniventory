> **Size note (read first).** M4 is the largest milestone so far. The author ratified the **maximal scope**: lead times configurable at **global + per-item + per-user**; channels = **in-app + Email/SMTP + HTTP + full MQTT bridge** (the MQTT bridge — reminder publish + state topics + Home Assistant auto-discovery + inbound commands — was pulled forward from M9). ## 1. Goal & non-goals

**Goal (the roadmap's M4 promise — the product differentiator):** **one** reminder engine, many sources, proactive "N days ahead" alerts across all of ①②③. Concretely:

- **Three trigger sources unified**: `best_before_date` (perishables ①), `warranty_expires` (durables ②), and **low-stock** (consumables ③). (Maintenance-due is added in M7; the engine is built source-pluggable so it slots in.)
- **Configurable lead times at three levels** — **global default per source**, **per-item override** (on the definition), **per-user override** (on the account) — with a deterministic resolution chain (§4.3).
- **Two firing paths (double safety, roadmap §2.6)**: a **daily scheduled scan** (APScheduler, at a configurable household-local time) evaluates every source idempotently; **event-triggering** fires low-stock the instant a movement drops a definition below its threshold (in-app immediacy without waiting for the next scan).
- **A persistent notification model**: an in-app **inbox** (the primary channel), with **idempotent de-duplication** so the daily scan never re-creates the same reminder, and a **low-stock episode** model so a persisting shortage re-reminds on a configurable cadence (immediate, then e.g. day 1/3/7) instead of either spamming daily or going silent.
- **Pluggable channels** delivering each new notification: **in-app** (always on), **Email/SMTP** (daily digest), **HTTP** (outbound webhook **and** an inbound state endpoint for Home Assistant's RESTful sensor), and the **full MQTT bridge** (reminder publish + stock/expiry/low-stock state topics + Home Assistant auto-discovery + inbound command topics).
- **A new KV settings table** holding all of the above configuration, surfaced through a new frontend **Configuration page** (the seed of the future general settings surface).

**Done when (🟢, expanded in §11):** set a global best-before lead of 3 days and a warranty lead of 30; override one item to a longer lead and one user to a shorter one; register an expiring perishable, an expiring warranty, and drive a consumable low; **run the scan** (on demand and on the daily timer) and watch a **consolidated set of in-app notifications** appear in the bell/inbox, an **email digest** arrive covering all three, the **MQTT state topics + reminder events** publish (and a Home-Assistant-discovered sensor reflect the counts), the **HTTP webhook** fire and the **HA state endpoint** return the live counts; let the low-stock persist and watch the **repeat reminder** fire on schedule; replenish and watch the **episode close** (no more repeats). CI stays green, including the no-drift contract gate.

**Non-goals (explicitly out of M4 — deferred or later milestones):**
- **No new domain data beyond reminders.** M4 *reads* the existing `best_before_date` / `warranty_expires` / low-stock signals (M2/M3); it does not change perishable/durable/consumable modelling. The only domain-table deltas are the two **lead-time override** columns (§3.4/§3.5).
- **No maintenance-due source.** That source (recurring durable maintenance) is **M7**; the engine's source interface is built to accept it without rework, but M7 implements it.
- **No multi-user delivery routing / roles.** Reminders fan out to **all active users** as the recipient set (M4 effectively has the single admin). **Responsible-party assignment** and **role-scoped visibility** are **M6**; the `notifications.user_id` recipient column is the forward-compatible seam.
- **No general settings surface beyond reminders + channels.** The Configuration page ships the **reminders + channels** sections only. Household name/currency/timezone editing and other settings come later; the KV table + page are built to extend.
- **No LLM / public third-party REST API / generic event webhooks.** Those stay **M9**. M4's HTTP is scoped to *reminder webhook out* + *integration state read in*; M4's MQTT is the HA bridge. (M9 retains a broader public API + generic webhooks + the LLM features.)
- **No retroactive replay.** Turning a channel on does **not** back-deliver historical notifications; changing a lead time affects **future** scan evaluations, not already-created notifications.

## 2. Locked decisions (settled in M4 planning; rationale in roadmap §2/§3 + investigation ch.3)

| Area | Decision |
|---|---|
| **One engine, source-pluggable** | A single `ReminderEngine.run_scan()` evaluates all sources in one pass and is **idempotent** (safe to re-run any number of times a day). Each source is a small evaluator (`best_before`, `warranty`, `low_stock`) behind a uniform interface so **maintenance-due (M7)** drops in without touching the others. |
| **Recipients = all active users (M4)** | Every notification carries a **`user_id`** (the recipient). In M4 the engine fans out to **all `is_active` users** (typically the one admin). **Responsible-party narrowing + roles = M6**; the column is the seam, so M6 is a filter change, not a schema change. |
| **Lead time: global → per-item → per-user, resolved most-specific-first** | Date-source lead days resolve as **per-item override (definition) → per-user override (account) → global default (settings)** — first non-null wins (§4.3). Per-item expresses the *item's* nature (passport warranty = 90 days, milk = 2); per-user is the account's blanket fallback; global is the floor. Low-stock has **no lead** (it's a state, not a date); its cadence is the repeat schedule. |
| **Date sources fire once per (recipient, lot, target-date)** | best_before / warranty fire a **single** notification when `today_local ≥ target_date − lead`. Dedup key = `"{source}:u{uid}:i{lot}:{target_date}"`. Editing the lot's date ⇒ a new key ⇒ a fresh reminder (correct). No daily re-fire as the date nears (a deliberate simplicity; an "also re-fire on the day-of / when overdue" refinement is noted §12). |
| **Low-stock = immediate + repeat *episodes*** | A definition going low **opens an episode**: an opener notification (offset 0) fires at once. While it **stays low** and **unresolved**, repeats fire when `days_since_episode_start ≥ offset` for each configured offset (default `[1, 3, 7]`), each once. Replenishing **closes the episode** (`resolved_at` set). Going low again later opens a **new** episode (new anchor date ⇒ new dedup namespace). The opener notification row *is* the episode record (no separate table) — see §3.3. |
| **Two firing paths (double safety)** | **Daily APScheduler scan** evaluates **all** sources at a configurable **household-local** time. **Event-triggering** runs the **low-stock** evaluator the instant a `consume`/`discard`/`adjust` movement drops a definition below its threshold (immediate in-app + instant channels; the email **digest** still waits for the daily scan). Date sources need no event hook (a date crossing into its window only happens at date rollover, which the daily scan owns). |
| **"Today" / scan time honour `household.timezone`** | The scan fires at the household-local time, and all date comparisons (`today_local`) use `household.timezone` (resolving the M3 §12 deferral). One household, one clock. |
| **New KV settings table** | A new **user-facing `settings`** table (`key` PK, `value` TEXT/JSON). **Distinct** from `app_config` (server-managed secrets — never user-editable) and from `household.settings` (a JSON blob, left untouched). All reminder + channel config lives here, dot-namespaced (`reminders.*`, `channels.*`). A typed `SettingsService` reads/writes with **defaults baked in code** (an unset key returns its default; the table only stores overrides). |
| **Channel secrets are write-only on the wire** | SMTP/MQTT passwords and the integration token are **set** via the settings API but **never echoed** in GET (a `*_is_set: bool` flag is returned instead). User-entered secrets live in `settings`; only the auto-generated server `secret_key` stays in `app_config`. |
| **In-app stays code+params; email/MQTT/HTTP carry a server-rendered message** | Per the wire/display split (M1.5), the **in-app** channel stores a **`message_code` + `params`** and the **frontend** localizes (the `notifications` namespace). Email/MQTT/HTTP payloads originate **server-side with no frontend in the loop**, so the backend renders human text from a **small bilingual server catalog** (ZH+EN, keyed by the same codes) in the **recipient's `preferred_language`** (household default if null). This is a **scoped, deliberate exception** to "backend emits no display text", justified because these artifacts leave the system without ever passing through the SPA. The external payload is `{code, params, message}` so machine consumers (HA) can use the structured part and humans read `message`. |
| **Channels pluggable behind a dispatcher** | A `NotificationDispatcher` iterates **enabled** channel adapters (`InAppChannel` is implicit = the row; `EmailChannel`, `HttpChannel`, `MqttChannel`). Per-notification **instant** channels (HTTP, MQTT) fire on every new notification (scan + event paths); **Email** is **digest-only** (bundled at the end of the daily scan). Each external attempt is logged in `notification_deliveries` and is **idempotent** (a channel skips notifications it already delivered). Network I/O happens **after** the notification rows commit — never inside a DB transaction; failures are best-effort (logged, recorded), never crash a scan or a movement. |
| **MQTT = full Home Assistant bridge (pulled forward from M9)** | A long-lived `paho-mqtt` connection managed in the FastAPI lifespan (connect on startup if enabled; clean disconnect on shutdown; auto-reconnect). It **publishes** reminder events + **state topics** (`low_stock_count`, `expiring_count`, `expired_count`, …), emits **Home Assistant MQTT discovery** configs so HA auto-creates the sensors, and **subscribes** to command topics to accept a **bounded** operation set (`consume`, `intake`, `adjust`). Commands execute through the existing services as a configured system actor; unknown/malformed commands are dropped + logged. |
| **HTTP = outbound webhook + inbound HA state endpoint** | (a) An **outbound** `HttpChannel` POSTs each new notification (`{code, params, message}`) to a configured URL with an optional auth header. (b) An **inbound** `GET /api/integrations/state` returns the live counts as JSON for Home Assistant's **RESTful sensor** to poll; it is authorized by a **static integration token** (header/query), *not* the session cookie (HA can't hold a cookie). A basic outbound-URL **SSRF note** is recorded; the hardened guard is **M6** (roadmap §5 M6). |
| **New deps** | Backend adds **`apscheduler`** (scheduler), **`paho-mqtt`** (MQTT), and **`httpx`** promoted to a **runtime** dependency (outbound webhook + integration tests already use it). Email uses the **stdlib** `smtplib`/`email`. These are stack additions ⇒ `AGENTS.md` "Tech stack & commands" backend line gains them (a foundations touch, done with Step 1). |
| **Error codes (minimal additions)** | New: `notification.not_found` (mark-read on a missing/foreign notification, 404) and `integration.invalid_token` (bad/missing token on the state endpoint, 401). Settings validation rides Pydantic ⇒ existing `validation.invalid_input`. MQTT command failures map to existing `stock.*` codes (logged, not HTTP). So M4 adds **two** `ErrorCode` entries. |

> These extend M0–M3 foundations. The two new error codes, the new tables/columns, every new schema, and the new endpoints flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope. The new backend deps + the scheduler/MQTT runtime wiring are the one "foundations" touch (AGENTS.md backend line + a `scheduler`/`mqtt` settings gate).

## 3. Data model

Three new tables + two additive column deltas. All use M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `Base`, rules in the service layer, DB access **only** through repositories. The two column deltas are nullable plain adds (no table rebuild).

### 3.1 `settings` — new table (migration `0015`)

User-facing key/value configuration. **Distinct** from `app_config` (server secrets) and `household.settings` (JSON blob).

| Column | Type | Null | Notes |
|---|---|---|---|
| `key` | String(128) | no | PK. Dot-namespaced (`reminders.best_before.lead_days`, `channels.mqtt.host`, …). |
| `value` | Text | no | Stored as text; JSON-encoded for structured values (e.g. the repeat list). Callers (de)serialize via the typed `SettingsService`. |
| `updated_at` | DateTime(tz) | no | `server_default=now()`, refreshed on upsert. |

Only **overridden** keys are stored; unset keys return their **code-defined default** through `SettingsService`. Upsert via `Session.merge` (the portable pattern already used by `AppConfigRepository`).

### 3.2 `notifications` — new table (migration `0018`)

The in-app inbox + the dedup ledger + the low-stock episode record (the opener row *is* the episode).

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `user_id` | FK→users.id | no | **Recipient** (the §2 seam). `ondelete=CASCADE`. |
| `source` | String(32) | no | `best_before` / `warranty` / `low_stock`. |
| `subject_type` | String(32) | no | `instance` (date sources) / `definition` (low-stock). |
| `subject_id` | Integer | no | The referenced lot or definition id (no hard FK — survives deletion as a historical record; existence is the engine's concern). |
| `dedup_key` | String(255) | no | Idempotency key (§4.4/§4.5). **Unique with `user_id`** (`uq_notifications_user_dedup`). |
| `message_code` | String(64) | no | i18n code (`reminder.best_before`, `reminder.warranty`, `reminder.low_stock`, `reminder.low_stock_repeat`). |
| `params` | Text | yes | JSON render params (name, date, days_remaining, qty, threshold, offset, location…). |
| `episode_started_on` | Date | yes | Low-stock only: the episode anchor date. NULL for date sources. |
| `offset_days` | Integer | yes | Low-stock only: which scheduled offset this row is (`0` = opener). NULL for date sources. |
| `resolved_at` | DateTime(tz) | yes | Low-stock only: set on the **opener** when the definition recovers (closes the episode). NULL = open / not applicable. |
| `read_at` | DateTime(tz) | yes | In-app read state. NULL = unread. |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |

Indexes: `uq_notifications_user_dedup` (unique idempotency); a non-unique `(user_id, read_at)` to make unread-count / inbox queries cheap.

### 3.3 Low-stock episode model (no extra table)

An **episode** = "this definition has been continuously low since date D, for this recipient." It is represented by its **opener notification** (`source=low_stock`, `offset_days=0`, `episode_started_on=D`, `resolved_at` NULL while open). Repeats are sibling rows (`offset_days ∈ repeat_schedule`) sharing the same `episode_started_on`. The engine:
- **opens** an episode (insert opener) when a definition is low and **no open opener** exists for `(user, def)`;
- **repeats** by inserting offset rows whose offset `≤ today_local − episode_started_on` and that don't yet exist;
- **closes** by stamping `resolved_at` on the open opener (and, for tidiness, its open repeats) when the definition is no longer low.

This keeps one table doing inbox + dedup + episode state; the unique `(user_id, dedup_key)` makes every insert idempotent. (A dedicated `episodes` table was considered and rejected as over-engineering for one state-based source — §12 notes the trade-off.)

### 3.4 `item_definitions` — addition (migration `0016`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `reminder_lead_days` | Integer | yes | **Per-item lead override** (`≥ 0`, Pydantic-validated). Applies to whichever date source this definition's lots carry (best_before for perishables, warranty for durables). NULL = inherit (§4.3). |

> A single override field (not one per source) is deliberate: a definition's lots are in practice one kind. Per-source per-item overrides are a noted refinement (§12).

### 3.5 `users` — additions (migration `0017`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `reminder_best_before_lead_days` | Integer | yes | **Per-user** best-before lead override (`≥ 0`). NULL = inherit global. |
| `reminder_warranty_lead_days` | Integer | yes | **Per-user** warranty lead override (`≥ 0`). NULL = inherit global. |

> Mirrors the existing nullable `preferred_language` lifecycle. Per-user **channel** preferences and per-user **low-stock** cadence are out of M4 (noted §12).

### 3.6 `notification_deliveries` — new table (migration `0019`)

The roadmap §3 "delivery log". Written by external channels (in-app is implicit).

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `notification_id` | FK→notifications.id | no | `ondelete=CASCADE`. |
| `channel` | String(32) | no | `email` / `http` / `mqtt`. |
| `status` | String(16) | no | `sent` / `failed`. |
| `detail` | String(1024) | yes | Error text on failure (truncated). |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |

Idempotency: a channel skips a notification that already has a `sent` row for that channel (a failed row may be retried on the next pass). Index `(notification_id, channel)`.

## 4. Backend design

### 4.1 Layering (extends M2/M3)

- **Repositories** (`app/repositories/`): `SettingsRepository` (get/set/get_all — `merge` upsert like `AppConfigRepository`); `NotificationRepository` (create-if-absent by dedup, list for user, unread count, mark read/all, find open low-stock opener, list undelivered-for-channel, mark resolved); `NotificationDeliveryRepository` (record, exists-sent). `ItemDefinitionRepository` / `UserRepository` thread the new override columns. **No raw queries** outside repositories.
- **Services** (`app/services/`):
  - `SettingsService` — typed accessors with code-defined defaults; validation; **write-only secret** handling.
  - `ReminderEngine` — the orchestrator: `run_scan()` (all sources, all recipients), `evaluate_low_stock(definition_id)` (the event-trigger scoped path), lead resolution (§4.3), dedup, episode lifecycle, then dispatch.
  - `NotificationService` — in-app inbox reads/writes (list, unread count, mark read) for the API.
  - `NotificationDispatcher` + channel adapters (`app/notifications/channels/`): `EmailChannel`, `HttpChannel`, `MqttChannel`; the server message catalog (`app/notifications/messages.py`, ZH+EN).
  - `IntegrationStateService` — computes the counts JSON for the inbound HA endpoint (reuses `LowStockService` / `ExpiryService`).
- **Engine reuses existing signals**: low-stock via `LowStockService.compute()`; expiry candidates via a best_before lot query; warranty via a warranty lot query. No re-implementation of those rules.
- **Scheduler** (`app/scheduler.py`): a thin wrapper starting an APScheduler `BackgroundScheduler` in the lifespan, registering the daily job from `reminders.scan_time`. Gated by a `scheduler_enabled` setting (off in tests).

### 4.2 Recipients & "today"

`run_scan()`: `today_local = now(household.timezone).date()`. `recipients = UserRepository.list_active()`. The engine loops recipients × sources. (One admin ⇒ one pass; the structure scales and M6 narrows `recipients` to the responsible party.)

### 4.3 Lead-time resolution (the easy-to-get-wrong chain — required tests)

For a date source `s ∈ {best_before, warranty}`, a definition `d`, and a recipient `u`:

```
resolve_lead(s, d, u):
    if d.reminder_lead_days is not None:        # 1. per-item override (both date sources)
        return d.reminder_lead_days
    per_user = u.reminder_best_before_lead_days if s == best_before
               else u.reminder_warranty_lead_days
    if per_user is not None:                    # 2. per-user override (per source)
        return per_user
    return settings.lead_days(s)                # 3. global default (per source)
```

First non-null wins; **per-item > per-user > global**. All values are `≥ 0` (Pydantic). A lead of `0` ⇒ fire on the target date itself.

### 4.4 Date sources (best_before, warranty)

For each recipient `u`, for each **live** lot carrying the source's date (`best_before_date` / `warranty_expires` NOT NULL; live = `quantity IS NULL OR quantity > 0`):

```
lead   = resolve_lead(source, lot.definition, u)
window = lot.target_date - timedelta(days=lead)
if today_local >= window:
    dedup = f"{source}:u{u.id}:i{lot.id}:{lot.target_date.isoformat()}"
    NotificationRepository.create_if_absent(
        user_id=u.id, source=source, subject_type="instance", subject_id=lot.id,
        dedup_key=dedup, message_code=f"reminder.{source}",
        params={ "name": lot.definition.name, "date": lot.target_date,
                 "days_remaining": (lot.target_date - today_local).days,
                 "location_id": lot.location_id })
```

Fires once per `(u, lot, date)`; re-running the scan is a no-op; editing the date makes a new key.

### 4.5 Low-stock source (immediate + repeat episodes)

Repeat offsets: `settings.low_stock_repeat_days()` (default `[1, 3, 7]`, sorted, deduped, each `≥ 1`). Let `low_now = { LowStockItem.definition_id }` from `LowStockService.compute()`.

```
for u in recipients:
    for def_id in low_now:
        opener = NotificationRepository.open_low_stock_opener(u.id, def_id)   # offset 0, resolved_at NULL
        if opener is None:
            create opener: offset_days=0, episode_started_on=today_local,
                dedup=f"low_stock:u{u.id}:d{def_id}:{today_local.isoformat()}:o0",
                message_code="reminder.low_stock", params={name, current, threshold, mode}
        else:
            elapsed = (today_local - opener.episode_started_on).days
            for o in repeat_offsets where o <= elapsed:
                dedup=f"low_stock:u{u.id}:d{def_id}:{opener.episode_started_on.isoformat()}:o{o}"
                create_if_absent: offset_days=o, episode_started_on=opener.episode_started_on,
                    message_code="reminder.low_stock_repeat", params={name, current, threshold, mode, offset:o}
    # close recovered episodes
    for opener in NotificationRepository.open_low_stock_openers(u.id) where opener.subject_id not in low_now:
        mark_resolved(opener)   # stamp resolved_at on opener (+ its open repeats)
```

`create_if_absent` relies on the unique `(user_id, dedup_key)`. `elapsed ≥ o` (not `==`) so a missed scan day still catches up; each offset fires once (dedup). **Event-trigger** calls `evaluate_low_stock(def_id)` — the same body scoped to one definition across recipients — right after a movement, in the same transaction (atomic with the movement); the daily scan is the fallback.

### 4.6 Dispatch & channels

After a scan/event commits new notification rows, `NotificationDispatcher.dispatch(new_notifications, *, include_email_digest)`:
- **In-app**: implicit (the rows exist).
- **HTTP / MQTT (instant)**: for each new notification not yet `sent` on that channel, render `{code, params, message}` (message via the server catalog in the recipient's language) and deliver; record a `notification_deliveries` row.
- **Email (digest)**: only when `include_email_digest` (daily scan): group the day's new notifications **per recipient**, render one digest email in that recipient's language to that user's email via SMTP; record a delivery row per included notification.

Event path → `dispatch(new, include_email_digest=False)`; scan → `dispatch(new, include_email_digest=True)`. All channel I/O is **post-commit**, best-effort, wrapped so a channel error is logged + recorded but never propagates.

Channel adapters read their config from `SettingsService`; a disabled/unconfigured channel is a **no-op** (skipped). `EmailChannel` uses stdlib `smtplib` (+`use_tls`); `HttpChannel` uses `httpx` (short timeout); `MqttChannel` publishes via the shared `paho-mqtt` client (§4.8).

### 4.7 Scheduler (APScheduler)

`app/scheduler.py` starts a `BackgroundScheduler` in `_lifespan` when `settings.scheduler_enabled` (default True; **False under `environment == "test"`** and in unit tests). It registers a daily `CronTrigger(hour, minute, timezone=household.timezone)` from `reminders.scan_time`; the job opens its own DB session, runs `ReminderEngine.run_scan()`, commits, then dispatches. Re-registration on config change is a noted refinement (§12); for M4 a `scan_time` change applies on next restart, and the on-demand endpoint covers immediate needs. **Single-worker assumption** (the Docker image runs one uvicorn worker); a multi-worker lock is noted (§12). The **engine is unit-tested directly** (call `run_scan()` synchronously) — APScheduler is thin wiring, not in the hot test path.

`POST /api/reminders/run` (authenticated) triggers `run_scan()` on demand and returns a small summary (created counts per source) — this is what the 🟢 walkthrough uses instead of waiting for the timer.

### 4.8 MQTT bridge (full)

A single `MqttBridge` (paho-mqtt) owned by the lifespan, started when `channels.mqtt.enabled`:
- **Connect** to `host:port` (optional TLS / username / password); `loop_start()` in a background thread; auto-reconnect via paho. Topic prefix from `channels.mqtt.topic_prefix` (default `omniventory`).
- **Reminder publish**: `MqttChannel` publishes each new notification to `{prefix}/notifications/{source}` (`{code, params, message}`, retained=false).
- **State topics**: after each scan (and after low-stock events) publish counts to `{prefix}/state/low_stock_count`, `/expiring_count`, `/expired_count` (retained=true so HA sees last value on reconnect).
- **Home Assistant discovery**: on connect (and on config change) publish HA MQTT-discovery configs to `homeassistant/sensor/omniventory_<metric>/config` so HA auto-creates the sensors bound to the state topics. Gated by `channels.mqtt.discovery_enabled`.
- **Inbound commands**: subscribe to `{prefix}/command/#`; accept a **bounded** set — `consume {definition_id, quantity}`, `intake {instance_id, quantity}`, `adjust {instance_id, counted_quantity}` — execute through `StockMovementService` in a fresh session as a configured **system actor** (a reserved user id / null acting user), publish a result to `{prefix}/command_result`. Unknown topic/op or malformed payload ⇒ drop + log. Gated by `channels.mqtt.commands_enabled` (default **false** — opt-in, since it mutates stock). A note on command auth/SSRF-equivalent trust boundary is in §12 (broker is operator-trusted; hardening is M6).

### 4.9 HTTP integration

- **Outbound** `HttpChannel` (instant): POST `{code, params, message}` to `channels.http.webhook_url` with optional `channels.http.auth_header`; `httpx` short timeout; record delivery. Disabled/unset ⇒ no-op. **SSRF note** (§12): M4 does a basic public-URL sanity check; the full guard is M6.
- **Inbound** `GET /api/integrations/state` → `{ low_stock_count, expiring_count, expired_count, generated_at }` for HA's RESTful sensor. **Auth = static integration token** (`X-Omniventory-Token` header or `?token=`), compared to `settings` `channels.http.integration_token` (auto-generated on first read if enabled; regenerate-able). Missing/wrong ⇒ `integration.invalid_token` (401). It is **not** behind the session-cookie dependency.

### 4.10 API surface (delta; all under `settings.api_prefix`, default `/api`)

| Method + path | Auth | Purpose |
|---|---|---|
| `GET /settings` | session | Read the reminders + channels config group (secrets masked → `*_is_set` flags). |
| `PATCH /settings` | session | Update config (validated; write-only secret fields). |
| `GET /notifications?unread_only=&limit=` | session | Current user's inbox (newest-first). |
| `GET /notifications/unread-count` | session | Badge count. |
| `POST /notifications/{id}/read` | session | Mark one read (404 `notification.not_found` if missing/foreign). |
| `POST /notifications/read-all` | session | Mark all current-user notifications read. |

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0–M3 §5 (`make check` all green; build; `make codegen` no-drift). The **logic that must have unit tests** in M4 (roadmap DoD — *expiry/lead-time date computation, threshold/low-stock triggers, idempotency*):

**Backend**
- **Lead resolution chain** (§4.3): per-item wins over per-user wins over global; NULLs fall through; `0` lead fires on the date; both sources pick the right per-user field.
- **Date-source firing & dedup** (§4.4): fires exactly when `today_local ≥ date − lead`, not before; **boundary** (`== window` fires, `window − 1 day` doesn't); a second scan creates nothing (idempotent); editing the date yields a new notification; an expired (past-date) lot fires; a depleted `exact` lot (`quantity 0`) does not; a `level`/`none` lot (qty NULL) with a date does; uses `household.timezone` for `today`.
- **Low-stock episodes** (§4.5 — required): opener fires once on going low; **repeats** fire at each offset (`elapsed ≥ offset`), each once; a missed scan day **catches up** (still fires, not skipped); **recovery closes** the episode (`resolved_at` set, no further repeats); going low **again** opens a **new** episode (new anchor, fresh dedup); event-trigger and daily scan produce the **same** rows (no double-insert).
- **Settings**: defaults returned when unset; upsert overrides; validation (lead `< 0`, repeat item `< 1`, bad `scan_time`, bad port) → `validation.invalid_input`; secrets never echoed (only `*_is_set`); set/clear a secret.
- **Dispatcher idempotency**: a channel skips notifications already `sent`; a channel error is recorded (`failed`) and never raised; email **digest** groups per recipient; disabled channel = no-op.
- **In-app API**: list (newest-first, unread filter), unread count, mark-one (404 on foreign id ⇒ `notification.not_found`), mark-all; a user sees only their own rows.
- **Integration token**: valid token ⇒ counts; missing/wrong ⇒ `integration.invalid_token` (401); counts match `LowStockService`/`ExpiryService`.
- **MQTT** (paho client mocked — no real broker in tests): publish builds the right topic+payload for reminders/state/discovery; a command payload maps to the right service call; malformed/unknown command is dropped (no exception, no mutation); bridge no-op when disabled.
- **Migration round-trip**: `0015`–`0019` upgrade clean on a DB at `0014` and downgrade cleanly; existing rows unaffected (new columns NULL).

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): the **bell** shows the unread count and the dropdown lists notifications localized from `message_code`+`params`; **mark-read** clears the badge; the **/notifications page** lists/filters; the **Configuration page** loads current settings, edits leads/repeat/scan-time/channel fields, masks secrets (shows "set"/"not set"), and PATCHes; the **per-item lead** field on the item form; the **per-user lead** fields on the profile/settings; dates/numbers via M1.5 `formatDate`/`formatQuantity`; all strings in **en+zh**.

## 12. Open questions / deferred

- **Per-item vs per-user precedence:** M4 locks **per-item > per-user > global** (first non-null). A "**max-of-applicable**" rule (warn at the earliest any policy would) is a plausible alternative if real use shows the override semantics surprise users — revisit.
- **Date-source re-fire:** M4 fires date reminders **once** at the lead boundary. Re-firing **on the day-of** and/or **when newly overdue** (a second, escalated notification) is a noted refinement — pairs with a per-source per-item lead if added.
- **Repeat cadence default:** `[1, 3, 7]` is a tunable default (the author mused 1/3/5/7 but "too frequent isn't good"); change in settings. A **back-off** cadence (widening gaps) is a candidate refinement.
- **Per-user channel + low-stock prefs:** M4 keeps per-user to **lead times**. Per-user **channel** selection ("email me but not MQTT") and per-user **low-stock cadence** are deferred (pair with M6 multi-user + roles).
- **Responsible-party routing:** M4 fans out to **all active users**. Narrowing to the **assigned responsible party** + role-scoped inbox visibility is **M6** (the `notifications.user_id` seam is in place).
- **Scheduler: multi-worker + live re-schedule:** single uvicorn worker assumed (no distributed lock); a `scan_time` change applies on **restart** (or via the on-demand endpoint) — live re-registration + a multi-worker lock are refinements if the deploy ever scales.
- **Channel security hardening:** the **SSRF guard** on the outbound webhook and **rate limiting** are **M6** (roadmap §5 M6); M4 does a basic public-URL sanity check. The MQTT broker is **operator-trusted**; `commands_enabled` defaults **false**; tighter command auth/allow-listing is a follow-up.
- **Episode model as one table:** the opener-row-as-episode (no dedicated `episodes` table) is deliberate for the single state-based source. If maintenance-due (M7) or future state sources need richer episode state, promoting it to its own table is a contained refactor.
- **Notification retention:** M4 keeps all notification rows (household scale). A retention/auto-prune policy (e.g. archive read+resolved after N days) is a later refinement.
- **Digest vs per-event email:** M4 makes email a **daily digest**; an "instant email per reminder" option (per-user preference) is a candidate when per-user channel prefs land.
