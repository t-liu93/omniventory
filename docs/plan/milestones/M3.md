## 1. Goal & non-goals

**Goal (the roadmap's M3 promise):** give every lot a **best-before date** and make the app *surface what is expiring*. Concretely: a per-lot **`best_before_date`** (batch-level precision — roadmap §2.4, the Grocy bar), a per-definition **`default_best_before_days`** shelf-life default that **auto-computes** a lot's best-before on intake when none is given, **FEFO consumption** (the M2 FIFO walk now orders **nearest-expiry-first** — the M2 §12 forward-promise: "M3 only changes the `ORDER BY`"), and a **computed expiring/expired read** (`GET /expiring`) surfaced as a **dashboard tile** + a dedicated **/expiring page**, mirroring M2's low-stock pattern.

**Done when (🟢, expanded in §11):** create a perishable definition with `default_best_before_days = 7`; register a lot **without** typing a date and watch `best_before_date` auto-compute to *today + 7*; register a second lot with an explicit nearer date; **consume** the definition and watch FEFO take the **nearest-expiry lot first** (not merely the oldest-received); the **dashboard expiry tile** lists lots expiring within the window and **flags expired** ones; the **/expiring page** lists the same with an adjustable horizon. CI stays green, including the no-drift contract gate.

**Non-goals (explicitly out of M3 — deferred or later milestones):**
- **No reminders / notifications.** Expiring/expired is a **computed read** (an endpoint + a dashboard tile + a page), exactly like M2's low-stock. The proactive "N days ahead" engine that *consumes* best-before (per-item **and** per-user lead times, daily scan, channels) is **M4** (roadmap §2.6 / §5 M4). M3 ships **no** scheduler, no per-user config, no email.
- **No opened / frozen / thawed "+N days" adjustments.** The Grocy-style perishable-state refinement (roadmap §5 M3 "optional", parking lot) is **deferred** — see §12. M3 has a single `best_before_date` per lot with no state machine.
- **No new tracking modes or mode changes.** The three M2 modes (`exact`/`level`/`none`) are unchanged. `best_before_date` is **mode-independent** (like `warranty_expires`): any lot may carry one.
- **No retroactive recompute.** Editing a definition's `default_best_before_days` affects **future** intakes only; existing lots keep their stored `best_before_date` (§2). Best-before is a **lot** property, set once at intake, editable per-lot.
- **No multi-unit / cross-cutting / multi-user changes.** Units stay M1's single string; photos/tags/barcodes are M5; roles are M6. Untouched here.
- **No warranty-as-expiry merge.** `warranty_expires` (M1, durable) and `best_before_date` (M3, perishable) stay **separate fields**; M4 is where both feed one reminder engine (roadmap §2.6/§2.7). M3 does not unify them.

## 2. Locked decisions (settled in M3 planning; rationale in roadmap §2/§3 + investigation ch.3)

| Area | Decision |
|---|---|
| **Best-before lives on the lot** | A new nullable **`stock_instances.best_before_date`** (`Date`). Per-**lot/batch**, never on the definition (roadmap §2.4 — the same-item-multiple-expiries case the def/instance split exists for). **Mode-independent**: settable for `exact`/`level`/`none` lots alike (a perishable can be tracked any way), mirroring `warranty_expires`. Nullable = "no expiry tracked" (non-perishable). |
| **Shelf-life default lives on the definition** | A new nullable **`item_definitions.default_best_before_days`** (`Integer`, `≥ 0`). The M2 doc deliberately left this absent "until its consumer (M3)" — M3 is that consumer. `NULL` = no default shelf life. |
| **Auto-compute on intake** | On **create** (`POST /instances`), when `best_before_date` is **omitted** *and* the definition has a `default_best_before_days`, the service sets `best_before_date = today + default_best_before_days`. **Precedence:** an explicit `best_before_date` always wins (even past-dated — you may register already-expired or known-expiry stock); omitted + a default → computed; omitted + no default → `NULL`. The reference "today" is the **server date** (`date.today()`); honouring `household.timezone` is a noted refinement (§12). Subsequent `intake` movements into an **existing** lot do **not** touch its `best_before_date` (one lot = one batch = one expiry; a different-expiry receipt is a **new lot**). |
| **FEFO consumption** (M2 §12 promise) | `consume` still walks the definition's active `exact` lots, but the FIFO ordering key gains best-before as the **primary** key: **`(best_before_date ASC NULLS LAST, received_at ASC, id ASC)`** — nearest-expiry-first, dated lots before never-expiring ones, `received_at`/`id` as tie-breakers. **Only the repository `ORDER BY` changes** (M2 wrote `consume_fifo` so M3 touches nothing else). The method/endpoint **names are unchanged** (`consume` / `consume_fifo`); the docstrings say FEFO. |
| **NULLS-LAST is explicit & portable** | SQLite orders `NULL` **first** under plain `ASC`; we want never-expiring lots **last**. The order key leads with **`best_before_date IS NULL`** (a `0/1` boolean: non-null sorts before null) rather than relying on a dialect `NULLS LAST` clause — portable across SQLite/Postgres (roadmap §2.11 portability), logic in the app/query layer. |
| **Expiring/expired is a per-lot computed read** | `GET /expiring?within_days=N` returns the **lots** (instances) whose `best_before_date ≤ today + N` (so the set is *expired ∪ expiring-within-N*), each tagged `status ∈ {expired, expiring}` with `days_remaining` (negative = past). Granularity is **per-lot** (unlike M2's per-**definition** low-stock) because expiry is a **batch** property (roadmap §2.4). Pure read, no persistence — M4 *consumes* this signal. |
| **What counts as "live" stock for the expiring list** | Include a lot iff it has a `best_before_date`, that date `≤ today + N`, **and** the lot is not an empty `exact` lot — i.e. `best_before_date IS NOT NULL AND best_before_date ≤ cutoff AND (quantity IS NULL OR quantity > 0)`. The `quantity IS NULL` arm keeps `level`/`none` lots (present-but-unquantified); the `> 0` arm drops fully-consumed `exact` lots so a depleted batch doesn't haunt the list. |
| **Horizon `within_days`** | A query param, **default 30**, **clamped `≥ 0`**. `0` ⇒ only already-expired + expiring-today. This is a *display* horizon, **not** a reminder lead time (those are per-item/per-user and land in M4). |
| **No new error codes** | `default_best_before_days ≥ 0` is enforced with a Pydantic field bound (`Field(ge=0)`) → the existing `validation.invalid_input` envelope; `best_before_date` accepts any valid date (incl. past). `within_days` is clamped, not rejected. So M3 adds **zero** `ErrorCode` entries — the FE↔BE error contract is unchanged. |
| **No retroactive recompute** | Changing `default_best_before_days` on a populated definition never rewrites existing lots' `best_before_date`. Best-before is captured at the lot's intake and only changes via an explicit `PATCH /instances/{id}`. |

> These extend M0/M1/M1.5/M2 foundations; they are **not** "tech stack" changes, so `AGENTS.md` does **not** change for M3. The two new columns and the `ExpiringItem` schema flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope (no new codes).

## 3. Data model

Two additive column deltas (one per existing table) + the expiry-read semantics. No new tables. All use the M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `Base`, business rules in the service layer, DB access **only** through a repository. Both columns are **nullable plain adds** — no SQLite batch rebuild needed (contrast M2's `0012`).

### 3.1 `item_definitions` — addition (migration `0013`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `default_best_before_days` | Integer | yes | Default shelf life in days; `≥ 0` (Pydantic-validated). `NULL` = no default. Consumed by auto-compute on intake (§4.2). Editing it is **non-retroactive** (§2). |

### 3.2 `stock_instances` — addition (migration `0014`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `best_before_date` | Date | yes | Per-lot/batch best-before. `NULL` = no expiry tracked. Mode-independent. Set explicitly or auto-computed on create (§4.2); editable via `PATCH`. The FEFO primary sort key (§4.3) and the expiring-read filter (§4.4). |

> Mirrors the existing nullable `warranty_expires` (`Date`) exactly in shape and lifecycle — a clean precedent for storage, serialization (`InstanceResponse`), and the `set_*`-flag update pattern (§4.1). No CHECK, no index in M3 (the expiring scan is a full read like low-stock; an index on `best_before_date` is a noted scaling refinement, §12).

### 3.3 Migration list

| Rev | Step | Does |
|---|---|---|
| `0013` | 1 | `item_definitions`: add `default_best_before_days` (nullable Integer). |
| `0014` | 2 | `stock_instances`: add `best_before_date` (nullable Date). |

Both reversible with a plain `op.drop_column` on downgrade. No data backfill (new nullable columns default to `NULL`; existing lots simply have "no expiry tracked", which is correct).

## 4. Backend design

### 4.1 Repository + service layering (extends M2)

- **Repositories** (`app/repositories/`):
  - `StockInstanceRepository`: **change the `ORDER BY`** in `list_active_lots_for_definition` to the FEFO key (§4.3) — the *only* change to that method. Add `list_expiring(cutoff_date)` (the §4.4 filter; pure data access). `create`/`update` thread the new `best_before_date` field (the `update` keeps the M2 `set_*`-flag pattern so "clear to NULL" is expressible).
  - `ItemDefinitionRepository`: `create`/`update` thread `default_best_before_days`. (No new query methods.)
- **Services** (`app/services/`):
  - `ItemDefinitionService`: store/echo `default_best_before_days` (the Pydantic `ge=0` bound does the validation; no service-level guard needed).
  - `StockInstanceService`: **auto-compute** `best_before_date` on create (§4.2); thread it through all three mode branches; allow it on `PATCH`. **No mode coupling** — best-before validation is independent of `exact`/`level`/`none` (unlike `quantity`/`stock_level`).
  - **`ExpiryService`** (new, small, pure-read — the sibling of M2's `LowStockService`): `compute(within_days)` → `list[ExpiringItem]` (§4.4).
- **No change** to `StockMovementService` beyond the FEFO ordering it inherits from the repo (consume already calls `list_active_lots_for_definition`). Auto-compute lives in *create*, not in the movement layer (best-before is a lot attribute, not a ledger event).

### 4.2 Auto-compute on intake (the headline convenience)

In `StockInstanceService.create`, **before** the mode branch (best-before is mode-independent), resolve the effective best-before once:

```
resolve_best_before(data, definition):
    if 'best_before_date' in data.model_fields_set:   # explicit wins, even past/None
        return data.best_before_date
    if definition.default_best_before_days is not None:
        return date.today() + timedelta(days=definition.default_best_before_days)
    return None
```

- **Explicit beats default:** a provided `best_before_date` (including an explicit past date) is stored verbatim; auto-compute only fills an **omitted** value.
- **Reference date:** `date.today()` (server date; household-tz refinement noted §12). Computed in the service so it needs no post-flush DB read.
- Then `repo.create(..., best_before_date=resolved)` in every mode branch (`exact`/`level`/`none`).
- `PATCH /instances/{id}` sets/clears `best_before_date` directly (no auto-compute on update — update is an explicit correction).

### 4.3 FEFO consumption (only the `ORDER BY` changes)

`StockInstanceRepository.list_active_lots_for_definition` — the FIFO→FEFO swap M2 §12 pre-authorised:

```python
stmt = (
    select(StockInstance)
    .where(
        StockInstance.definition_id == definition_id,
        StockInstance.quantity > 0,
    )
    .order_by(
        StockInstance.best_before_date.is_(None),  # 0=dated first, 1=NULL last (portable NULLS LAST)
        StockInstance.best_before_date,             # nearest expiry first
        StockInstance.received_at,                  # then oldest received (M2 tie-break)
        StockInstance.id,                           # then stable id
    )
)
```

`StockMovementService.consume_fifo` is **untouched** — it walks whatever order the repo returns, so it now consumes **nearest-expiry-first** automatically (dated lots before never-expiring lots; same-expiry lots oldest-received-first). The endpoint stays `POST /definitions/{id}/consume`; docstrings updated to say FEFO.

### 4.4 Expiring / expired (computed read)

`GET /expiring?within_days=N` → a per-lot list. The service computes the cutoff and delegates the filter to the repo:

```
compute(within_days):
    within_days = max(within_days, 0)           # clamp, never reject
    today  = date.today()
    cutoff = today + timedelta(days=within_days)
    lots   = repo.list_expiring(cutoff)          # best_before NOT NULL AND <= cutoff
                                                 #   AND (quantity IS NULL OR quantity > 0)
                                                 # ORDER BY best_before_date, id
    for lot in lots:
        days = (lot.best_before_date - today).days
        yield ExpiringItem(
            instance_id   = lot.id,
            definition_id = lot.definition_id,
            name          = lot.definition.name,
            location_id   = lot.location_id,
            best_before_date = lot.best_before_date,
            quantity      = lot.quantity,          # Decimal | None (mode-dependent)
            days_remaining = days,                 # negative = expired
            status        = 'expired' if days < 0 else 'expiring',
        )
```

- **Ordering:** soonest/most-overdue first (`ORDER BY best_before_date, id`) — most-urgent at the top, expired naturally leads (their date is earliest).
- **`name`** comes from the lot's definition; the repo eager-loads or the service resolves it (the definition `name` is already loaded for low-stock; reuse the same access pattern — no N+1 in the small household dataset, with a note in §12 if it ever grows).
- Pure read; **no writes**. M4 will consume this exact signal for proactive reminders.

### 4.5 API surface (delta from M2; all under `settings.api_prefix`, default `/api`; all authenticated via `get_authenticated_context`)

| Method + path | Change |
|---|---|
| `POST /definitions` · `PATCH /definitions/{id}` | now also accept **`default_best_before_days`** (`int ≥ 0`, nullable) |
| `POST /instances` | now accepts **`best_before_date`**; **auto-computed** when omitted and the definition has a default (§4.2) |
| `PATCH /instances/{id}` | **`best_before_date`** settable / clearable (nullable, `set_*`-flag pattern) |
| `GET /instances/{id}` · `GET /instances` | `InstanceResponse` now carries **`best_before_date`** |
| `POST /definitions/{id}/consume` | unchanged shape; now **FEFO** (nearest-expiry first) — §4.3 |
| `GET /expiring?within_days=N` | **new** — computed expired ∪ expiring-within-N lot list (§4.4); `within_days` default 30, clamped `≥ 0` |

Error contract (unchanged envelope, no new codes): `422 validation.invalid_input` for `default_best_before_days < 0`; `401` no session. `GET /expiring` has no 4xx beyond auth.

### 4.6 Pydantic schemas (`app/schemas/`)

- `DefinitionCreate/Update/Response` gain `default_best_before_days: int | None` (create/update with `Field(default=None, ge=0)`; `Update` keeps it optional-nullable).
- `InstanceCreate/Update/Response` gain `best_before_date: date | None`. `Update` uses the existing optional-nullable convention so the service's `model_fields_set` check distinguishes "omit" from "clear to NULL".
- New `ExpiringItem` (the §4.4 shape): `instance_id`, `definition_id`, `name`, `location_id: int | None`, `best_before_date: date`, `quantity: Decimal | None`, `days_remaining: int`, `status: str` (`"expired"`/`"expiring"`). `Decimal` for quantity (roadmap §2.9); `date` for the date.

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0–M2 §5 (`make check` = ruff + mypy + pytest + eslint + tsc + vitest, all green; build passes; `make codegen` no-drift). The **logic that must have unit tests** in M3 (roadmap DoD — *expiry / lead-time date computation*, *consumption order*):

**Backend**
- **Auto-compute precedence:** omitted `best_before_date` + definition default `N` ⇒ `best_before_date == today + N`; **explicit date wins** (incl. an explicit past date and an explicit `None` left as `NULL`); omitted + **no** default ⇒ `NULL`; works for **all three modes** (`exact`/`level`/`none`). Non-retroactive: changing the definition default does **not** alter an existing lot.
- **FEFO ordering (the consumption order — required):** two lots, the **nearest-expiry** one consumed first **even when it was received later** (proves expiry beats `received_at`); a **NULL**-best-before lot consumed **last** (NULLS-LAST); **same** best-before falls back to `received_at` then `id`; multi-lot span and partial-lot `Decimal` precision still hold (M2 invariants unbroken); insufficient-total still rejected with nothing written.
- **Expiring read:** a lot with `best_before < today` ⇒ `status='expired'`, `days_remaining < 0`; `today ≤ best_before ≤ today+N` ⇒ `status='expiring'`, `days_remaining ≥ 0`; **boundary** `best_before == today+N` included, `today+N+1` excluded; `within_days=0` ⇒ only expired+today; **clamp** negative `within_days` to 0; a depleted `exact` lot (`quantity=0`) **excluded**; a `level`/`none` lot (`quantity=NULL`) with a date **included**; ordering soonest-first; empty result when nothing qualifies.
- **Schema bound:** `default_best_before_days < 0` ⇒ 422 `validation.invalid_input`; `0` accepted (expires same day).
- **Migration round-trip:** `0013` and `0014` upgrade clean on a DB at `0012` and downgrade cleanly (plain add/drop column); existing rows get `NULL` for both new columns and displayed data is unchanged.

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): definition form shows/stores `default_best_before_days`; instance form has a `best_before_date` picker and (optional) shows the would-be-computed hint when a default exists; instance/lot detail renders an **expiry badge** (expired = red, expiring-soon = amber, otherwise none) computed from `best_before_date`; the **dashboard expiry tile** renders count + short list from `GET /expiring` with an empty state; the **/expiring page** renders the list and the horizon control; dates render via `formatDate` (M1.5 locale-aware).

## 9. Step breakdown (atomic, ordered)

Each step is independently testable, test-backed, lands exactly one commit (one per-step autosquash in orchestrator mode), re-runs `make codegen` if it touched the API, and inherits the global DoD (§5) + the **anti-freelancing rule**: implement *only* this step — no other steps, no opportunistic refactors, no touching the host's real environment.

> **Ordering rationale:** the definition default (`0013`) and the instance column (`0014`) are independent additive schema; the instance column must exist before auto-compute writes it, so `0014` + auto-compute land together (step 2). FEFO (step 3) only re-orders the existing consume query once `best_before_date` exists. The expiring read (step 4) sits on the same column. Frontend (5–6) consumes the typed contract the backend lands.

### Step 1 — Definition: `default_best_before_days` (backend)
- **Goal:** the per-definition shelf-life default, validated and on the contract.
- **Build:** migration `0013` (add nullable Integer); extend `ItemDefinition` model + `ItemDefinitionRepository` create/update; `DefinitionCreate/Update/Response` gain `default_best_before_days` (`Field(ge=0)`, nullable); `ItemDefinitionService` stores/echoes it; `make codegen`.
- **Tests:** stored/echoed; default `None`; `< 0` rejected (422 `validation.invalid_input`); `0` accepted; non-retroactive is moot here (no instance column yet); migration `0013` up/down.
- **Out of scope:** instance column, auto-compute (step 2); FEFO/expiring (3/4).
- **Commit:** `feat(backend): per-definition default best-before days`

### Step 2 — Instance: `best_before_date` + auto-compute (backend)
- **Goal:** the per-lot best-before column, explicit-or-auto on create, editable on update.
- **Build:** migration `0014` (add nullable Date); extend `StockInstance` model + `StockInstanceRepository` create/update (`set_best_before_date` flag); `Instance{Create,Update,Response}` gain `best_before_date`; `StockInstanceService.create` resolves best-before (explicit-wins → definition default → NULL — §4.2) and threads it through all three mode branches; `PATCH` sets/clears it; `make codegen`.
- **Tests (the easy-to-get-wrong date math — required, per §5):** auto-compute `today + N`; explicit date (incl. past) wins; explicit `None` stays NULL; omitted + no default → NULL; all three modes; non-retroactive (changing the def default leaves an existing lot's date unchanged); PATCH set/clear; migration `0014` up/down.
- **Out of scope:** FEFO (step 3); the expiring endpoint (step 4); frontend.
- **Commit:** `feat(backend): per-lot best-before date with intake auto-compute`

### Step 3 — FEFO consume ordering (backend)
- **Goal:** consume nearest-expiry-first (the M2 §12 promise).
- **Build:** change **only** the `ORDER BY` in `StockInstanceRepository.list_active_lots_for_definition` to `(best_before_date IS NULL, best_before_date, received_at, id)` (§4.3); update that method's + `consume_fifo`'s docstrings to say FEFO. **No schema change → no codegen.**

## 12. Open questions / deferred

- **Opened / frozen / thawed "+N days" (Grocy-style):** deferred (roadmap §5 M3 "optional" + parking lot). Would add a per-lot perishable-state + a "+N days on open / +M days on freeze" adjustment to `best_before_date`. Revisit once real use shows the single-date model is too coarse.
- **"Today" timezone:** M3 uses the **server** `date.today()`. The singleton `Household` already carries a `timezone`; honouring it for the expiry boundary (so "expires today" flips at the household's midnight, not the server's) is a clean refinement — pairs with M4's scheduled scan, which will need the same notion of "now".
- **Best-before index:** the expiring scan is a full read (fine for a household). If the dataset ever grows, add an index on `stock_instances.best_before_date` (and possibly a composite for the FEFO key). Noted, not done.
- **`within_days` default / per-user horizon:** M3 fixes the display default at 30 (clamped `≥ 0`) and offers a horizon control. Truly **configurable, per-item *and* per-user lead times** are an **M4** concern (roadmap §2.6) — M3's horizon is a display filter, not a reminder policy.
- **Definition `name` resolution in the expiring read:** M3 resolves the lot's definition name inline (small dataset). If list sizes grow, batch-load definitions to avoid N+1 — same note as any other list endpoint.
- **Warranty ↔ best-before unification:** kept **separate** (M1 durable vs M3 perishable). **M4** is where both (plus low-stock and, later, maintenance-due) feed **one** reminder engine; M3 deliberately doesn't merge them.
- **Expose `received_at` on intake / back-dated auto-compute:** M3 auto-computes from `date.today()`, and a user back-dating a perishable simply types the known date manually (explicit-wins). Wiring `received_at` into `POST /instances` so auto-compute can reference a back-dated receipt (the M2 "received_at granularity" open item) is deferred — revisit if manual entry proves friction.
