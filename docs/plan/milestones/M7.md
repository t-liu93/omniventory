> **Scope note (read first).** M7 closes the two loops the core model left open: the **consumable loop** (low stock → a real, persisted **shopping list** you tick off, optionally feeding stock back via the M2 intake ledger) and the **durable-care loop** (recurring **maintenance schedules** on durables that fire through the M4 reminder engine as its long-promised **maintenance-due source**). It delivers, in **three phases** (A shopping list, B maintenance + the reminder source, C frontend) of **8 atomic steps**: a `shopping_list_items` table auto-reconciled from the live low-stock signal plus free-text manual items, check-off that can intake purchased stock through the **existing** ledger path, a `maintenance_schedules` table with calendar-correct recurrence and a "mark done → advance" flow, the additive **maintenance source** in `ReminderEngine` (routed by the durable's responsible party, M6), and the shopping-list / maintenance / dashboard UI. M7 also **reserves — but does not build — the external task-sync seam (TickTick)**: see **§12**.

## 1. Goal & non-goals

**Goal (the roadmap's M7 promise — close the consumable loop and the durable-care loop):**

- **Shopping list** — a persisted, **household-shared** list (single-tenant, all users share data — roadmap §1.2). It is **auto-populated from the live low-stock signal** (`LowStockService`, M2) and also accepts **free-text manual items**. Items can be **checked off**; checking off a definition-linked item can **intake the purchased quantity into stock through the existing M2 ledger** (no new stock path — roadmap §2.3). Auto rows **self-prune** when their definition recovers above threshold.
- **Maintenance schedules** — recurring maintenance tasks attached to a **durable stock instance** (the specific physical unit: "service the car every 6 months", "replace the AC filter every 3 months"). Each schedule carries a **calendar-correct recurrence** (every N days/weeks/months/years), a `next_due_date`, an advance-notice **lead**, and a **"mark done → advance"** action that records the completion and rolls `next_due_date` forward.
- **Maintenance-due reminder source** — the source M4 was built to accept (M4 §1/§2 "maintenance-due is added in M7; the engine is built source-pluggable so it slots in"). It is an **additive** evaluation pass in `ReminderEngine.run_scan()` — it fires "N days ahead" of `next_due_date`, **routes to the durable's effective responsible party** (M6 §4.4), de-duplicates per occurrence, and renders in-app + through every external channel (email digest / HTTP / MQTT) exactly like the other three sources. **No change to best_before / warranty / low-stock.**

**Done when (🟢, expanded in §11):** drop a consumable below its `min_stock` and watch it **auto-appear on the shopping list**; add a free-text item by hand; **check an item off and intake** the purchased quantity, which lands in stock via a real `intake` movement and (because the definition is no longer low) **drops off the list**; create a maintenance schedule on a durable with a `next_due_date` inside its lead window, **run the scan**, and receive a **maintenance reminder** in the bell/inbox (and the email digest / MQTT / HTTP), routed to the durable's responsible party; **mark the maintenance done** and watch `next_due_date` roll forward by the interval and the reminder clear. CI stays green, including the no-drift contract gate; migrations `0001`–`0034` apply cleanly to a fresh DB.

**Non-goals (explicitly out of M7 — deferred or later milestones):**
- **No external task-list integration (TickTick et al.).** M7 **reserves the seam** (§12) but ships **nothing TickTick-specific**. The actual OAuth2 + two-way sync build belongs with the integrations milestone family (M9-era) — see §12 and the roadmap §6/§7.
- **No usage-/meter-based maintenance** ("every 10 000 km"). M7's recurrence is **time/calendar-based only**. Usage meters (odometer, run-hours) are a noted refinement (§13) — they need a meter-reading concept M7 doesn't model.
- **No maintenance-completion history ledger.** A schedule stores only its **last** completion (`last_completed_date` + recompute). A full per-completion audit table (like the M2 stock ledger) is a noted deferral (§13).
- **No per-definition maintenance schedules.** Schedules attach to a **stock instance** (the specific durable). A definition-level "applies to every unit" schedule is deferred (§13); it would mirror the M6 responsible-party "on both" pattern when needed.
- **No new stock semantics.** Check-off intake **delegates to the existing M2 `StockMovementService` / instance-create path** — it never re-implements quantity math or the ledger (roadmap §2.3). Partial split-move and lot-scoped consume stay as M2 left them.
- **No multi-list / categories / store-aisle ordering / sharing-out.** One household, one shopping list (roadmap §1.2). Multiple named lists, per-store grouping, and assignment of shopping to a person are parking-lot (§13).
- **No per-user shopping list or per-user maintenance routing knobs beyond what M6 already gives.** Maintenance routing reuses the M6 responsible-party chain + fallback-to-all verbatim; no new routing model.

## 2. Locked decisions (settled in M7 planning; rationale in roadmap §2 + M4 §2 + M6 §2)

| Area | Decision |
|---|---|
| **Shopping list is a *persisted* table, not a computed view** | A computed "show me what's low" list cannot hold **manual items** or **check-off state**, which the roadmap M7 bullet requires. So a real `shopping_list_items` table holds both **`manual`** rows (user-entered) and **`auto`** rows (materialised from low-stock). This mirrors Grocy's real shopping-list table and lets the row carry desired-quantity, note, and purchased state. |
| **Auto rows are *reconciled*, not live** | `auto` rows are inserted/removed by an **idempotent reconcile pass** (`ShoppingListService.reconcile_auto_items()`) that reads `LowStockService.compute()` — the same signal M4 consumes. It **opens** an auto row for every currently-low definition: if none exists, it creates one; if a checked auto row exists (check-off without restock), it **reopens it** (`purchased_at` cleared) so the suggestion resurfaces (§4.3). It **removes** any **open, unchecked** auto row whose definition has recovered (a **checked** auto row with a recovered definition survives in "done" until *clear-purchased*). A **partial-unique index** `(definition_id) WHERE source='auto'` is the DB backstop for **exactly one auto row per definition, in any purchased state** — making it state-independent means check-off (which only stamps `purchased_at`) can never produce a second auto row to collide with on uncheck. Gated by a `shopping_list.auto_add_low_stock` setting (default **true**). |
| **Reconcile runs on the same hooks low-stock already fires on (double safety)** | `reconcile_auto_items()` is called by (a) the **daily scan** job, (b) the on-demand `POST /reminders/run`, and (c) the **low-stock event hook** in `StockMovementService` — in **all three** movement paths that already call `evaluate_low_stock` (`consume_fifo` / `discard` / `adjust`), **best-effort + savepoint-isolated** so a reconcile failure can never roll back the movement — so a consumable dropping low **immediately** surfaces on the list, matching the reminder engine's event + scheduled double-safety. The **`ReminderEngine` itself stays decoupled** from the shopping list (callers invoke both; the engine has no shopping-list dependency). |
| **Check-off → intake delegates to the M2 ledger (no new stock path)** | Checking off a definition-linked item with intake params **creates a new stock lot via the existing `StockInstanceService` create path** (which records the initial `intake` movement, M2 §2) — quantity stays **ledger-derived, never blind-set** (roadmap §2.3). Without intake params, check-off only stamps `purchased_at`. Both happen in **one transaction**. Intake is meaningful only for an **`exact`** definition, so `check_off` **pre-checks the mode** and rejects intake on a `level`/`none` definition with the existing `stock.movement_not_applicable` (a small mode guard — the only check-off-specific stock logic; the lot creation itself is pure delegation). |
| **Maintenance schedule attaches to a *stock instance* (the durable unit)** | `maintenance_schedules.instance_id` → `stock_instances.id`, `ondelete=CASCADE` (the schedule dies with the durable). The specific-unit anchor matches "durable goods" (serialized instances) and lets the reminder reuse the M6 **per-lot** responsible-party chain (`instance → definition → fallback-all`). Per-definition schedules are deferred (§13). |
| **Recurrence = calendar-correct interval (unit + count), computed in the app layer** | A schedule stores `interval_unit ∈ {day, week, month, year}` (String, app-validated against `MAINTENANCE_INTERVAL_UNITS`; no DB CHECK — §2.11) + `interval_count ≥ 1`. The **next due date** is computed by a small, unit-tested **`add_interval(d, unit, count)`** helper with **calendar-correct month/year arithmetic and end-of-month clamping** (e.g. Jan-31 + 1 month = Feb-28/29) — stdlib only, **no new dependency**. This is exactly the "date computation that's easy to get wrong" the DoD demands tests for (§5). |
| **Completion model = `last_completed_date` + advance (no extra table)** | "Mark done" sets `last_completed_date = completed_on` (default today, back-datable) and **advances `next_due_date = add_interval(completed_on, unit, count)`**. The schedule keeps only its **last** completion; a full completion-history ledger is deferred (§13), consistent with the M4 "opener-row-as-episode, no extra table" minimalism. A schedule can be **paused** (`is_active=false`) instead of deleted; paused schedules are skipped by the engine. |
| **Maintenance is an *additive* reminder source (the M4 seam)** | `ReminderEngine` gains a dedicated `_evaluate_maintenance` pass (parallel to the two `_DateSource` evaluators and the low-stock pass) — **the existing sources are untouched**. It fires when `today_local ≥ next_due_date − lead`, dedups per occurrence (`maintenance:u{uid}:s{sid}:{next_due_date}`), routes via the **instance's** effective responsible party (`_effective_responsible_for_lot(schedule.instance)`, M6), and writes a notification with `source="maintenance"`, `subject_type="maintenance_schedule"`, `subject_id=schedule.id`, `message_code="reminder.maintenance"`. `ScanSummary` / `ReminderRunSummary` gain a `maintenance` count. |
| **Maintenance lead resolution: per-schedule → global (no per-user)** | `resolve_maintenance_lead = schedule.lead_days if not None else settings.maintenance_lead_days()` (global default `7`). Per-user maintenance lead is intentionally **not** modelled (the M4 per-user leads are for best_before/warranty only); a maintenance task's advance notice is a property of the *task*, not the *person*. Per-user maintenance lead is a noted refinement (§13). |
| **External task-sync (TickTick) is a *reserved seam*, not a feature** | M7 ships **no** TickTick code. It keeps `ShoppingListService` the **single mutation choke-point** and keeps the `NotificationChannel` protocol the **single delivery extension point**, so a future outbound shopping sync and a future "reminder → TickTick task" channel are **additive, not a refactor**. Full rationale, the two seams, and the technical realities (OAuth2, no change-webhooks → polling, two-way reconciliation) are in **§12**; the timing decision (build it in the integrations milestone) is recorded in the roadmap §6/§7. |
| **Permissions (M6)** | Reads (`GET /shopping-list`, `GET /maintenance-schedules`) require `VIEW`; every mutation (add / edit / check / intake / delete / clear; schedule create / edit / complete / delete) requires `EDIT`; both ride the existing M6 `require_permission` sweep. Acting user is recorded on shopping-list rows and (optionally) on schedules. Nothing here is `MANAGE_*`. |
| **Error codes (additions)** | New: `shopping_list.not_found` (404 — operate on a missing list item), `maintenance.not_found` (404 — operate on a missing schedule), `validation.unsupported_interval_unit` (422 — bad `interval_unit`, mirroring `validation.unsupported_tracking_mode`). **Three** new `ErrorCode` entries, all through the M1.5 uniform error envelope. Intake errors on check-off reuse the existing `stock.*` codes; missing instances reuse `instance.not_found`. |

> These extend M0–M6 foundations. The two new tables, every new schema, and the new endpoints flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope. **No new runtime dependencies** (calendar arithmetic uses the stdlib; reconcile reuses `LowStockService`; the reminder source reuses the M4 engine + dispatcher). `AGENTS.md` is **unchanged** — M7 is not a "foundations" touch.

## 3. Data model

Two new tables + the maintenance source's notification rows (which reuse the existing `notifications` columns — **no schema delta there**). All use M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `Base`, rules in the service layer, DB access **only** through repositories. **Current head = `0032`; M7 adds `0033`–`0034`.**

### 3.1 `shopping_list_items` — new table (migration `0033`)

The persisted, household-shared shopping list (auto + manual rows).

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `source` | String(16) | no | `auto` (materialised from low-stock) / `manual` (user-entered). App-validated against `SHOPPING_LIST_SOURCES` (no DB CHECK — §2.11). |
| `definition_id` | FK→item_definitions.id | yes | `ondelete=CASCADE`. Set for auto rows and for definition-linked manual rows. **NULL** for free-text manual items. |
| `name` | String(255) | yes | Free-text label for manual items **without** a definition. For definition-linked rows the display name is read **live** from the definition (kept fresh, not snapshotted). A row must have `definition_id` **or** `name` (app-layer cross-field check). |
| `desired_quantity` | Numeric(18,6) | yes | How much to buy (Decimal, never float — §2.9). NULL = unspecified (e.g. level-mode items, or "just buy some"). |
| `unit` | String(32) | yes | Unit label for free-text manual items (definition-linked rows show the definition's unit). |
| `note` | String(1000) | yes | Free-text note. |
| `purchased_at` | DateTime(tz) | yes | Check-off state. NULL = open/unchecked; set on check-off. |
| `created_by` | FK→users.id | yes | `ondelete=SET NULL`. The user who added/auto-created the row. |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |
| `updated_at` | DateTime(tz) | no | `server_default=now()`, refreshed on update. |

Indexes: **partial unique** `uq_shopping_list_one_auto_per_def` on `(definition_id) WHERE source='auto'` (**exactly one auto row per definition, in any purchased state** — the reconcile idempotency backstop; making it state-independent means a check-off/uncheck round-trip can never create a colliding second auto row, `sqlite_where` like the M2 serial index); non-unique `(purchased_at)` for the open/done split.

### 3.2 `maintenance_schedules` — new table (migration `0034`)

Recurring maintenance tasks on a durable stock instance.

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `instance_id` | FK→stock_instances.id | no | `ondelete=CASCADE`. The durable this schedule maintains. |
| `name` | String(255) | no | What the maintenance is (e.g. "Replace AC filter"). |
| `interval_unit` | String(8) | no | `day` / `week` / `month` / `year`. App-validated against `MAINTENANCE_INTERVAL_UNITS` (no DB CHECK — §2.11). |
| `interval_count` | Integer | no | `≥ 1` (Pydantic-validated). "every `count` `unit`s". |
| `next_due_date` | Date | no | The next scheduled due date (the reminder target date). |
| `lead_days` | Integer | yes | Advance-notice override (`≥ 0`). NULL = inherit the global `reminders.maintenance.lead_days`. |
| `last_completed_date` | Date | yes | When last completed (NULL = never). The recompute anchor for "mark done". |
| `notes` | String(1000) | yes | Free-text. |
| `is_active` | Boolean | no | `server_default=true`. False = paused (skipped by the engine, kept for history). |
| `created_by` | FK→users.id | yes | `ondelete=SET NULL`. |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |
| `updated_at` | DateTime(tz) | no | `server_default=now()`, refreshed on update. |

Indexes: `(instance_id)` (instance-detail listing + cascade), `(next_due_date)` (the engine's "due soon" scan), `(is_active)`.

### 3.3 Maintenance notifications reuse the existing `notifications` table (no delta)

The maintenance source writes ordinary `notifications` rows — the M4 columns already fit:
- `source = "maintenance"`, `subject_type = "maintenance_schedule"`, `subject_id = schedule.id` (no hard FK; `subject_*` are free string/int by M4 design, surviving deletion as history).
- `message_code = "reminder.maintenance"`, `params = {name, instance_name, next_due_date, days_remaining, location_id}`.
- `dedup_key = "maintenance:u{uid}:s{sid}:{next_due_date}"` — fires **once per (recipient, schedule, due-date)**; advancing `next_due_date` on completion yields a fresh key for the next occurrence (the exact date-source dedup discipline, M4 §4.4). The low-stock-only columns (`episode_started_on`, `offset_days`, `resolved_at`) stay **NULL**.

So M7 needs **no `notifications` migration** — only the engine and the server message catalog grow.

## 4. Backend design

### 4.1 Layering (extends M2/M4/M6)

- **Constants** (`app/core/stock.py` or a small `app/core/maintenance.py`): `SHOPPING_LIST_SOURCES = ("auto", "manual")`; `MAINTENANCE_INTERVAL_UNITS = ("day", "week", "month", "year")`.
- **Date helper** (`app/core/dates.py`, new — or extend an existing core module): `add_interval(d: date, unit: str, count: int) -> date` with **calendar-correct** month/year addition + end-of-month clamping (§4.4). Pure, stdlib-only, heavily unit-tested.
- **Repositories** (`app/repositories/`):
  - `ShoppingListRepository` (new): `create` (catches a unique-index `IntegrityError` → no-op, mirroring `NotificationRepository.create_if_absent`, so a concurrent reconcile can't raise), `get`, `list` (open / include-purchased, ordered), `update`, `delete`, `get_auto_item(definition_id)` (the def's auto row in **any** state — open or checked), `list_open_auto_items()` (open unchecked, for pruning), `clear_purchased()`.
  - `MaintenanceScheduleRepository` (new): `create`, `get`, `list` (filter by instance / active), `list_for_instance`, `list_active()` (all `is_active` schedules, joinedload instance→definition for routing/naming — the engine applies the due-window in Python, exact parity with `list_live_with_best_before`; **no scalar "horizon"**, because a per-schedule `lead_days` is unbounded and any fixed horizon would silently drop a long-lead schedule whose window is already open — §4.5), `update`, `delete`.
  - **No raw queries** outside repositories (roadmap §2.10).
- **Services** (`app/services/`):
  - `ShoppingListService` (new) — the **single mutation choke-point** (the TickTick seam, §12): `add_manual`, `edit`, `check_off(item_id, intake?)`, `uncheck`, `remove`, `clear_purchased`, and `reconcile_auto_items()` (§4.3). `check_off` with intake **delegates** to `StockInstanceService` (no new stock logic).
  - `MaintenanceScheduleService` (new) — `create`, `edit`, `delete`, `complete(schedule_id, completed_on?, note?)` (sets `last_completed_date`, advances `next_due_date` via `add_interval`), validation (interval unit/count, instance exists).
  - `ReminderEngine` (extended) — the additive `_evaluate_maintenance` pass + `resolve_maintenance_lead` (§4.5). The notification model is unchanged.
  - The **server message catalog** (`app/notifications/messages.py`) gains the `reminder.maintenance` renderer (ZH+EN) for email/HTTP/MQTT.

### 4.2 Shopping-list CRUD & check-off

- **Add manual** (`POST /shopping-list`, `EDIT`): `{definition_id?, name?, desired_quantity?, unit?, note?}` → a `manual` row. Cross-field: at least one of `definition_id` / `name` (else `validation.invalid_input`). A `definition_id` must exist (`item_definition.not_found`/existing 404).
- **Edit** (`PATCH /shopping-list/{id}`, `EDIT`): `desired_quantity` / `name` / `note` (PATCH semantics via `model_fields_set`). 404 `shopping_list.not_found`.
- **Check off** (`POST /shopping-list/{id}/check`, `EDIT`): stamp `purchased_at = now` (uniform for auto + manual rows — auto rows are **not** deleted, so they remain the single auto row for their definition, §3.1). If the body carries `intake` (`{location_id?, quantity?}`) **and** the row has a `definition_id`: (1) pre-check the definition is `exact` (else `stock.movement_not_applicable`); (2) resolve the intake quantity as `intake.quantity ?? desired_quantity` — if **both** are NULL → `validation.invalid_input` (you must say how many you bought, since auto rows carry no `desired_quantity`); (3) **create a new lot** for that definition through `StockInstanceService.create` (records the initial `intake` movement, M2 — that movement currently records `user_id=None` per the M2 create path; attributing it is a pre-existing gap, §13). All in **one transaction** (an intake failure rolls back the check-off). Returns the updated item (+ the created instance id when intake ran).
- **Uncheck** (`POST /shopping-list/{id}/uncheck`, `EDIT`): clear `purchased_at` (does **not** reverse any intake — that's a separate stock action; documented). Safe for auto rows because the per-def auto-row uniqueness is **state-independent** (§3.1), so clearing `purchased_at` can never collide with a second auto row. 404 if missing.
- **Remove** (`DELETE /shopping-list/{id}`, `EDIT`): hard delete. 404 if missing.
- **Clear purchased** (`POST /shopping-list/clear-purchased`, `EDIT`): delete all `purchased_at IS NOT NULL` rows; returns the deleted count.
- **Refresh** (`POST /shopping-list/refresh`, `EDIT`): force `reconcile_auto_items()` and return the current list (the in-UI way to demo auto-population without a scan).

### 4.3 Auto-reconcile (the easy-to-get-wrong idempotency — required tests)

```
reconcile_auto_items():
    if not settings.shopping_list_auto_add():      # gate: shopping_list.auto_add_low_stock (default true)
        return
    low = LowStockService(db).compute()            # reuse the M2/M4 signal — never re-derive the rule
    low_def_ids = { item.definition_id for item in low }

    # 1. open: ensure ONE open auto row per currently-low definition
    for item in low:
        existing = repo.get_auto_item(item.definition_id)
        if existing is None:
            repo.create(source='auto', definition_id=item.definition_id,
                        desired_quantity=None, created_by=None)   # create catches IntegrityError -> no-op (DB backstop)
        elif existing.purchased_at is not None:
            repo.clear_purchased_at(existing)      # re-low after check-off without restock: reopen (fail-safe)

    # 2. prune: drop OPEN, UNCHECKED auto rows whose definition recovered
    for row in repo.list_open_auto_items():
        if row.definition_id not in low_def_ids:
            repo.delete(row)
```

- "Open" = `purchased_at IS NULL`. A **checked** auto row whose definition is **still low** is **reopened** (`purchased_at` cleared) so the suggestion resurfaces without requiring the user to "clear purchased" first. A genuine restock raises stock above `min_stock`, so a truly-recovered definition is never low and its checked row is never reopened.
- Intentional consequence: if a user checks off an auto row without actually restocking (definition is still below `min_stock`), the next reconcile/refresh will reopen it. This is accepted fail-safe behaviour.
- The partial-unique index `(definition_id) WHERE source='auto'` makes "**exactly one auto row per definition, any state**" a DB invariant; `create` catches the `IntegrityError` and no-ops, so re-running reconcile — or any check-off/uncheck round-trip — can never produce a duplicate (this is the B-class fix that closes the uncheck-collision hole).
- **Recovery prunes the suggestion** (an *open, unchecked* auto row whose definition is back above threshold disappears) — but never a manual row, and never a checked row (a checked auto row whose definition has recovered stays in "done" until clear-purchased).
- Called from the **daily scan**, the **on-demand `POST /reminders/run`**, the **low-stock event hook** in all three of `consume_fifo` / `discard` / `adjust` (best-effort + savepoint-isolated; it can reuse the `LowStockService.compute()` the hook already ran), and **`POST /refresh`** — all idempotent.

### 4.4 `add_interval` — calendar-correct recurrence (required tests)

```
add_interval(d, unit, count):
    if unit == 'day':   return d + timedelta(days=count)
    if unit == 'week':  return d + timedelta(weeks=count)
    if unit == 'month': return _add_months(d, count)
    if unit == 'year':  return _add_months(d, count * 12)

_add_months(d, n):
    m = d.month - 1 + n
    year  = d.year + m // 12
    month = m % 12 + 1
    day   = min(d.day, _days_in_month(year, month))   # end-of-month clamp
    return date(year, month, day)
```

- **Clamping** is the trap: Jan-31 + 1 month → Feb-28 (or Feb-29 in a leap year); Aug-31 + 6 months → Feb-28/29. Tested across leap years and every month boundary.
- Stdlib only (`calendar.monthrange` for `_days_in_month`); **no `dateutil`** dependency.

### 4.5 Maintenance reminder source (the additive engine pass)

A new `_evaluate_maintenance(active_users, active_by_id, today_local)` pass in `ReminderEngine`, invoked from `run_scan()` after the date sources and low-stock (the existing sources are **not** modified):

```
schedules = MaintenanceScheduleRepository(db).list_active()
                # ALL active schedules, joinedload instance→definition; the due-window
                # below is applied in Python (parity with list_live_with_best_before).
                # No scalar horizon: a per-schedule lead_days is unbounded, so any fixed
                # horizon would silently drop a long-lead schedule whose window is open.
for s in schedules:
    lead   = s.lead_days if s.lead_days is not None else settings.maintenance_lead_days()
    window = s.next_due_date - timedelta(days=lead)
    if today_local < window:
        continue
    recipients = _recipients_for(_effective_responsible_for_lot(s.instance), active_users, active_by_id)
    for u in recipients:
        if not u.notify_in_app and not u.notify_email_digest:   # M6 pref gate
            continue
        dedup = f"maintenance:u{u.id}:s{s.id}:{s.next_due_date.isoformat()}"
        params = { "name": s.name,
                   "instance_name": s.instance.definition.name,   # the durable's product name
                   "next_due_date": s.next_due_date.isoformat(),
                   "days_remaining": (s.next_due_date - today_local).days,
                   "location_id": s.instance.location_id }
        notification, created = notification_repo.create_if_absent(
            user_id=u.id, source="maintenance",
            subject_type="maintenance_schedule", subject_id=s.id,
            dedup_key=dedup, message_code="reminder.maintenance", params=params)
        # count + collect new rows like the other sources
```

- **Routing** reuses M6 verbatim: the durable instance's `responsible_user_id` → its definition's → fallback to all active users. The instance + definition are joinedloaded so no N+1.
- **Dedup** fires once per occurrence; **completion advances `next_due_date`**, so the next occurrence gets a fresh key. Editing `next_due_date` directly also yields a new occurrence (correct).
- **Lead** is per-schedule → global; a lead of `0` fires on the due date itself; an **overdue** schedule (today past `next_due_date`) still fires (window already passed) and `days_remaining` goes negative (renderers show "overdue").
- `ScanSummary.maintenance` and `ReminderRunSummary.maintenance` carry the created count; new rows ride the existing **post-commit dispatch** (in-app + email digest + HTTP + MQTT) with the new `reminder.maintenance` server-catalog text. **No event-trigger path** (a maintenance date crossing into its window only happens at date rollover, which the daily scan owns — same reasoning as the date sources, M4 §2).

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0–M6 §5 (`make check` all green; build; `make codegen` no-drift). The **logic that must have unit tests** in M7 (roadmap DoD — *date computation, threshold triggers, idempotency, stock in/out*):

**Backend**
- **`add_interval` calendar math** (§4.4 — required): day/week add; month/year add; **end-of-month clamp** (Jan-31 +1mo → Feb-28/29; Aug-31 +6mo → Feb-28/29); leap-year February; year rollover; `count ≥ 2`; year = ×12 months.
- **Shopping-list reconcile idempotency** (§4.3 — required): a low definition with no auto row → one created; **re-running creates nothing** (idempotent / state-independent partial-unique); a recovered definition's **open unchecked** auto row is **pruned**; a **manual** row is never auto-pruned; a **checked** auto row whose definition is **still low** is **reopened** (`purchased_at` cleared, suggestion resurfaces); a checked auto row whose definition has **recovered** survives in "done" (not pruned); a **check-off → uncheck round-trip on an auto row never collides** (the B-class regression — state-independent per-def uniqueness); the `auto_add_low_stock=false` gate makes reconcile a no-op; level-mode low item gets an auto row with NULL quantity.
- **Check-off → intake** (§4.2): check-off without intake only stamps `purchased_at`; check-off **with** intake creates a lot via the existing service and the quantity is **ledger-derived** (an `intake` movement exists; `quantity == SUM(deltas)`); intake quantity = `intake.quantity ?? desired_quantity`, with **both NULL → `validation.invalid_input`**; intake on a non-`exact` definition → `stock.movement_not_applicable` (the `check_off` mode pre-check); the whole op is atomic (intake failure rolls back the check-off); after intake the definition (now above min) **drops off the list** on the next reconcile.
- **Maintenance complete → advance** (§4.5/§4.4): completing sets `last_completed_date` and advances `next_due_date = add_interval(completed_on, unit, count)`; back-dated `completed_on` advances from that date; the advanced date yields a new dedup namespace.
- **Maintenance reminder firing & routing** (§4.5 — required): fires exactly when `today_local ≥ next_due_date − lead`, **boundary** (`== window` fires, `window − 1` doesn't); **overdue** fires with negative `days_remaining`; **per-schedule lead beats global**; lead `0` fires on the due date; a **long-lead schedule whose `next_due_date` is far out but whose window is already open still fires** (the all-active candidate query — no horizon to drop it, the B-class fix); a **paused** (`is_active=false`) schedule never fires; re-running the scan creates nothing (dedup); routing = instance→definition→fallback-all (reuse M6 helper) and respects the pref gate; **the three existing sources are unaffected** (a regression test asserts best_before/warranty/low-stock counts unchanged with maintenance present).
- **Settings**: `maintenance_lead_days` default `7`, `auto_add_low_stock` default `true`; both round-trip; `ge=0` validation; `ReminderRunSummary.maintenance` returned.
- **Permissions** (M6): `viewer` blocked from every shopping-list / maintenance mutation (403 `auth.forbidden`); `member`/`admin` permitted; reads open to all.
- **Migration round-trip**: `0033`–`0034` upgrade clean on a DB at `0032` and downgrade cleanly; existing rows unaffected.

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): the **Shopping list page** lists auto + manual rows (source badge), adds a manual item, edits quantity, checks off (with the optional **intake** modal that reuses the existing intake form), clears purchased, and refreshes; a **low-stock** drop surfaces an auto row after refresh; the **maintenance section** on the durable detail lists schedules with a due/overdue/ok status, creates/edits/deletes, and "marks done" (next-due advances); the **dashboard upcoming-maintenance tile** renders the count + nearest items; the **Configuration page** edits `maintenance_lead_days` + `auto_add_low_stock`; the **notification bell** renders `reminder.maintenance` localized from `message_code`+`params`; all new strings present in **en + zh**; dates/numbers via M1.5 `formatDate`/`formatQuantity`.

## 12. Reserved integration seam — external task / shopping sync (TickTick)

> **M7 builds none of this.** This section records *why the seam is reserved now* and *what a future build hangs off*, so the later integration is **additive, not a refactor**. The **timing decision** — build it in the integrations milestone family (M9-era), not in M7 — is recorded in the roadmap §6 (parking lot) and §7 (open questions).

**Why reserve now, build later.** A TickTick-style sync is an **external-SaaS, OAuth2, two-way** integration — exactly the theme of the integrations milestone (M9: public REST API + generic webhooks + OAuth/token infrastructure), which has to build the token-store + OAuth plumbing anyway. Doing it inside M7 would couple a core domain loop to a third-party API's stability and balloon a milestone whose job is to *close the loops*. The project's established discipline is "**reserve the abstraction now, implement later**" (roadmap §1.3 / §2.12 — the LLM backend is the precedent). The MQTT/HA bridge was pulled *forward* into M4 only because it is core to the reminder differentiator; an external to-do mirror is not — it is convenience on top.

**The two seams a future build hangs off (both already exist after M7 — by design, at zero extra cost):**

1. **Outbound shopping-list sync.** `ShoppingListService` is the **single mutation choke-point** for every list change (add / edit / check / uncheck / remove / reconcile). A future `ShoppingListSyncProvider` (no-op default) observes those mutations and mirrors them to a TickTick **project/list** (and, on inbound poll, marks items purchased when ticked in TickTick). The natural future addition is a nullable **`external_ref`** column on `shopping_list_items` for id-mapping — **noted, not added now** (don't pay for it until the sync ships).
2. **Reminder → task.** A future **`TickTickChannel`** implements the **existing `NotificationChannel` protocol** (the M4 dispatcher is already pluggable: in-app / email / HTTP / MQTT). It would create a TickTick **task with a due date** for each new reminder — "a perishable is about to expire / a warranty is ending / **maintenance is due** → a task on your to-do list you already check daily." This maps the author's "expiring → add it to TickTick" intent **directly onto the channel seam** — and because every M7/M4 notification already carries structured `{message_code, params}`, the task title/body/due-date render with **no new server text**.

**Technical realities to handle in the future build (deferred — all belong with the M9 OAuth/token work):**
- **OAuth2** authorization-code flow + **token storage & refresh** (per-household, or per-user if accounts map 1:1 to TickTick accounts) — reuse the M9 credential infrastructure; do **not** invent a parallel secret store.
- **Inbound sync likely needs polling.** TickTick's Open API exposes tasks/projects but (per current knowledge) **no change-webhooks**, so "ticked off in TickTick → mark purchased here" must be **polled** on an interval — verify the exact API capabilities at design time.
- **Two-way reconciliation** — conflict resolution (ticked in both places), de-duplication, and **idempotency** (don't recreate a task every scan; map by `external_ref`). This distributed-sync surface is precisely why it is *not* an M7 concern.
- **Scope toggle** — make each direction independently opt-in (outbound shopping mirror; reminder→task channel), defaulting off, like every other M4 channel.

**What M7 deliberately does to keep the seam cheap:** keeps `ShoppingListService` the sole mutation path (no route writes directly to the repo); keeps the `NotificationChannel` protocol the sole delivery extension point; emits structured `{message_code, params}` for the new maintenance source like every other source. Nothing TickTick-specific ships, and nothing later requires reworking M7.

## 13. Open questions / deferred

- **Usage-/meter-based maintenance** ("every 10 000 km / 200 run-hours"): M7 is calendar-only. A usage-meter concept (a reading you log, intervals in meter units) is a contained later addition — it needs a meter model M7 doesn't have.
- **Maintenance completion history:** M7 keeps only `last_completed_date`. A per-completion ledger (date, note, cost, who) — a durable-care analogue of the M2 stock ledger — is a natural follow-up if real use wants the history/audit.
- **Per-definition maintenance schedules:** M7 attaches to a specific instance. A definition-level "applies to every unit" schedule (mirroring the M6 responsible-party "on both definition and instance" shape) is deferred until a real multi-unit-same-product case appears.
- **Per-user maintenance lead / per-schedule recurrence richness:** M7 leads resolve per-schedule → global (no per-user). "nth weekday of the month", "skip weekends", and cron-like recurrence are parking-lot.
- **Shopping-list ergonomics:** one shared list, no per-store aisle grouping, no multiple named lists, no "assign the shopping to a person". All parking-lot until the single list proves limiting (roadmap §1.2 keeps it one household).
- **Auto-row quantity suggestion:** auto rows carry NULL `desired_quantity`. Suggesting a reorder quantity (e.g. `min_stock − current`, or a per-definition reorder amount) is a refinement once the reorder-point UX settles (relates to the M2 §12 `min_stock` semantics).
- **Check-off intake target:** M7 check-off intake **creates a new lot**. Topping up an **existing** lot, or choosing the lot, is a later option (M2 already exposes per-lot `intake`).
- **Check-off intake attribution + fields:** the initial intake movement created on check-off records `user_id=None` (a pre-existing `StockInstanceService.create` limitation, **not** introduced by M7 — the shopping-list row itself records `created_by`); and `ShoppingListIntake` omits `occurred_at`/`note` because the M2 create path doesn't thread them. Wiring the acting user + back-date/note through the create path is a small follow-up.
- **Re-low before clearing purchased (fixed):** if a definition recovers while its auto row is still **checked** (sitting in the "purchased" section) and then goes low again, reconcile now **reopens the checked row** (`purchased_at` cleared) so the suggestion resurfaces immediately — no need for the user to hit "clear purchased" first. Intentional consequence: checking off an auto row without an actual restock (definition still below `min_stock`) will also be reopened by the next reconcile/refresh — this is the fail-safe intent (a genuine restock raises stock above `min_stock`, so a truly-recovered definition is never re-low and never reopened).
- **TickTick / external task sync:** reserved seam only (§12); the actual OAuth2 + two-way build is scheduled into the integrations milestone family (roadmap §6/§7).
