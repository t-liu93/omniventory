> **Scope note (read first).** The roadmap's M5 bullet lists eight capabilities. In M5 planning the author **trimmed and re-shaped** that list (rationale below + §2): **backup/restore is deferred** (a self-hosted user backs up the single `DATA_DIR` directory by hand for now), and **CSV is export-only** (import is deferred to the LLM era, where free-form column mapping is the natural fit). What ships is **six cross-cutting capabilities + barcode + data export**, split into **four phases** (A generic metadata, B barcode + search, C export, D frontend) of **13 atomic steps**, each independently deployable.

## 1. Goal & non-goals

**Goal (the roadmap's M5 promise — daily-use ergonomics + data portability):** turn the bare data model into something that feels like a real self-hosted app. Concretely:

- **Generic attachments / photos** — upload images and documents (receipts, warranty PDFs) and attach them to any item definition, stock instance, or location through **one** polymorphic mechanism (roadmap §2.8). Files live on the filesystem under `DATA_DIR`, content-addressed (sha256) so identical uploads de-duplicate; **reference-counted** so deleting an owner never leaves an orphan file.
- **Tags** — flat, colour-able labels attachable to multiple entity types (definitions, instances, locations), for cross-cutting grouping that the category tree can't express.
- **Notes** — free-text notes attachable to the same entity types.
- **Custom fields** — per-entity user-defined key/value pairs (a JSON map on the definition and the instance) for the attributes the built-in schema doesn't cover (voltage, vintage, dosage…). Searchable as text.
- **Global search** — one `GET /search` across item definitions, stock instances (serial / barcode / custom fields), locations, categories, and tags, app-layer LIKE (portable, roadmap §2.11), behind a **provider seam** so an LLM-assisted semantic search can slot in later.
- **Barcode scanning + product lookup** — the browser camera decodes 1D (EAN/UPC) and 2D (QR) codes client-side; the value hits `GET /barcodes/lookup`, which resolves it through a **`ProductLookupProvider` seam**. M5 ships the **internal** provider only (a scanned code that's already bound to a definition jumps straight to it — fast re-intake); the external product-database / LLM providers are a reserved hook (§2.12, M9).
- **CSV / JSON export** — download item definitions, stock instances, and locations for reporting, an insurance inventory list, or off-box archival.

**Done when (🟢, expanded in §11):** attach a photo to an item and see it in a gallery, then delete it and watch the underlying file disappear (ref-count); create tags and a note and attach them; add custom fields and find an item by a custom value via global search; bind a barcode to a definition, then **scan** it to jump to that item and to pre-fill an intake; run a global search that returns hits grouped by type; export a CSV and open it in a spreadsheet. CI stays green, including the no-drift contract gate; migrations `0001`–`0027` apply cleanly to a fresh DB.

**Non-goals (explicitly out of M5 — deferred, re-scoped, or later milestones):**
- **No backup/restore UI** (re-scoped out of M5 — roadmap §5 M5 had it as "core"). A self-hosted operator backs up by copying the bind-mounted `DATA_DIR` (which holds both the SQLite file **and** `media/`). A first-class in-app backup/restore flow is a later refinement (§12).
- **No CSV import** (deferred). Export ships; import — with its column-mapping, foreign-key-by-name resolution, validation, dry-run, and partial-failure handling — is deferred to pair with the **LLM-assisted formatting** backend (§2.12, M9), which is exactly suited to messy spreadsheet ingestion (§12).
- **No external product lookup.** M5 builds the `ProductLookupProvider` seam and ships only the **internal** provider. Open Food Facts / other public product DBs / LLM vision are M9 (§2.12). Adding one later is a new provider class + a settings toggle, with **no change** to the scan flow or UI.
- **No LLM semantic search.** M5 ships a LIKE `SearchProvider`; the semantic provider is the reserved seam (M9).
- **No server-generated thumbnails.** M5 validates images and downscales over-large originals to a sane cap; gallery rendering uses the (capped) original with lazy-loading. A dedicated thumbnail pipeline is a refinement (§12).
- **No per-request media authorization / content sanitization hardening.** Media is served via a static mount keyed by the unguessable content hash (a capability URL). Per-request authorization, upload content sanitization, and SVG/active-content handling are the **M6** hardening pass (roadmap §5 M6, mirroring how M4 deferred SSRF) — M5 applies only the basic safe defaults in §4.2.
- **No nestable tags.** The roadmap §3 mentioned "Tag (nestable)"; M5 ships **flat** tags (the existing Category tree already covers hierarchy). A deliberate, ratified simplification (§2).
- **No asset-tag generation / label printing.** The internal asset-tag + QR/label generation stays in the parking lot; M5 delivers the *scanning* side only. The `barcodes.symbology` field leaves room for a future `internal` tag kind.

## 2. Locked decisions (settled in M5 planning; rationale in roadmap §2/§3 + the planning discussion)

| Area | Decision |
|---|---|
| **One generic cross-cutting mechanism, not per-table columns** | Attachments, tags, and notes are **polymorphic**: a `(model_type, model_id)` pair references the owner (roadmap §2.8). Allowed owner types are a small, centrally-validated set — `item_definition`, `stock_instance`, `location`. There is **no DB foreign key** on `model_id` (it can't point at multiple tables); existence and **delete-cascade are enforced in the service layer** (§4.3). |
| **Attachments: filesystem + content-addressed + reference-counted** | Bytes live on disk under `DATA_DIR/media/<sha256[:2]>/<sha256>` (sharded). A `media_files` row holds the hash + size + validated content-type + image dimensions; an `attachments` row is one **reference** from an owner to a `media_files` row (with the original filename + optional caption). Identical uploads **de-duplicate** (same hash ⇒ one physical file, N references). Deleting the last reference deletes the `media_files` row **and** the physical file — **no orphans** (the author's explicit requirement). |
| **Images validated + size-capped; no thumbnails** | On upload, **Pillow** verifies the bytes really are a decodable image and records dimensions; over-large images are **downscaled** to a configured max edge (default 2048px) before hashing/storing, capping storage growth. Non-image attachments (PDF, etc.) are stored as-is up to a size limit. **No** server thumbnail variants in M5 (§12). |
| **Media served via static mount, hash-keyed (capability URL)** | The content-hash path is effectively unguessable, so M5 serves media through a `StaticFiles` mount at `/media` (efficient: range requests, caching). The mount is registered **before** the SPA catch-all route in `create_app` (the catch-all `@app.get("/{full_path:path}")` would otherwise swallow `/media/*`). Per-request authorization is **M6** (§1 non-goals). |
| **Tags: flat, multi-entity** | A `tags` table (unique name, optional colour) + a polymorphic `tag_links` table. **Flat** (no tree — the Category tree covers hierarchy). Name uniqueness is enforced case-insensitively in the service layer (DB unique on `name`). |
| **Custom fields: JSON key/value map, searchable** | A nullable `custom_fields` **JSON** column on **both** `item_definitions` and `stock_instances`. Values are a flat map of `str → (str|int|float|bool|null)` (no nesting in M5, to keep it searchable and renderable). It is **included in global search** as a substring match on the serialized text — so values *are* findable. What JSON KV deliberately gives up vs a structured field schema: **typed/range queries**, **key-consistency facets**, and **per-field typed UI** (a ratified trade for low complexity — §12). |
| **Barcodes: own table, one code → one definition, provider seam** | A `barcodes` table (`definition_id` FK, `code` **globally unique**, `symbology`, optional label). One definition may carry **many** codes; a code resolves to **one** definition (deterministic lookup). Scanning is **client-side** (ZXing in the browser, 1D + 2D); the decoded value hits `GET /barcodes/lookup?code=` which runs the **`ProductLookupProvider`** chain. M5 ships **`InternalProvider`** only. |
| **Global search: app-layer LIKE, provider seam** | A `SearchService` runs case-insensitive substring queries across the entity repositories (reusing their existing `q` filters where present), returns results **grouped by type** and per-type capped. **No SQLite FTS** (keeps the DB swappable — §2.11). A `SearchProvider` protocol fronts it so an LLM **semantic** provider can be added later without touching the endpoint/UI. |
| **CSV export-only (JSON too); import deferred** | `GET /export/{entity}?format=csv\|json` streams `item_definitions` / `stock_instances` / `locations` via the service layer (stdlib `csv`; no new model). Foreign keys are exported by **both id and name** (e.g. `category_id` + `category_name`) for human readability. **Import is out** (§1 non-goals). |
| **Backup/restore deferred** | Not built in M5. The single bind-mounted `DATA_DIR` (SQLite file + `media/`) **is** the backup unit; operators copy it. (§1 non-goals, §12.) |
| **New deps (foundations touch)** | Backend: **`python-multipart`** (multipart uploads) + **`pillow`** (image validation/downscale). Frontend: **`@zxing/browser`** (+ `@zxing/library`) for in-browser 1D/2D decoding. These are stack additions ⇒ `AGENTS.md` "Tech stack & commands" backend + frontend dependency lines gain them (done with Step 1 / Step 11). Pillow installs from a manylinux wheel — **no extra apt packages** for JPEG/PNG/WebP. |
| **`DATA_DIR` becomes a real setting** | `.env.example` documents `DATA_DIR` but `config.py` doesn't implement it. Step 1 adds a `data_dir` setting (default `./data`), derives `media/` from it, creates the directory on startup, and keeps the existing `database_url` behaviour unchanged. |
| **Error codes (additions)** | New: `attachment.not_found` (404), `attachment.file_too_large` (413), `attachment.unsupported_type` (415), `tag.not_found` (404), `tag.duplicate_name` (409), `note.not_found` (404), `barcode.not_found` (404), `barcode.duplicate` (409 — code already bound to a definition). Search/export validation rides Pydantic ⇒ existing `validation.invalid_input`. So M5 adds **eight** `ErrorCode` entries, all flowing through the M1.5 uniform error envelope. |

> These extend M0–M4 foundations. The new tables/columns, every new schema, and the new endpoints flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope. The new backend/frontend deps + the `/media` static mount + the `DATA_DIR` setting are the "foundations" touch (AGENTS.md dependency lines + a `data_dir` config field).

## 3. Data model

Six new tables + two additive JSON column deltas. All use M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `Base`, rules in the service layer, DB access **only** through repositories. The two column deltas are nullable plain adds (no table rebuild).

### 3.1 `media_files` — new table (migration `0020`)

The physical-file registry (content-addressed). One row per unique byte-content.

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `sha256` | String(64) | no | **Unique** (`uq_media_files_sha256`). Content hash; also the storage key. |
| `byte_size` | Integer | no | Size on disk (post-downscale for images). |
| `content_type` | String(128) | no | **Our** validated MIME (never the user-supplied header verbatim). |
| `width` | Integer | yes | Image pixel width (Pillow); NULL for non-images. |
| `height` | Integer | yes | Image pixel height; NULL for non-images. |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |

On-disk path: `DATA_DIR/media/<sha256[:2]>/<sha256>` (no extension; type is in the DB).

### 3.2 `attachments` — new table (migration `0021`)

One **reference** from an owner entity to a `media_files` row. The polymorphic, ref-counting join.

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `media_file_id` | FK→media_files.id | no | `ondelete=RESTRICT` (DB-level safety: a `media_files` row can't be deleted while referenced; the service deletes the row only after the **last** reference is removed). |
| `model_type` | String(32) | no | `item_definition` / `stock_instance` / `location` (centrally validated, §4.3). |
| `model_id` | Integer | no | Owner id (no hard FK — polymorphic). |
| `original_filename` | String(255) | yes | For display / download. |
| `title` | String(255) | yes | Optional caption. |
| `sort_order` | Integer | no | `default 0`; ordering within an owner. |
| `uploaded_by` | FK→users.id | yes | `ondelete=SET NULL`. |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |

Indexes: `(model_type, model_id)` (list an owner's attachments); `(media_file_id)` (ref-count). **Reference count** = `COUNT(attachments WHERE media_file_id = x)` — no separate counter column (the join *is* the count).

### 3.3 `tags` + `tag_links` — new tables (migrations `0022`, `0023`)

`tags`:

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `name` | String(64) | no | **Unique** (`uq_tags_name`). Case-insensitive uniqueness enforced in the service. |
| `color` | String(32) | yes | Optional (a Mantine colour name or hex). |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |

`tag_links` (polymorphic):

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer | no | PK. |
| `tag_id` | FK→tags.id | no | `ondelete=CASCADE` (deleting a tag drops its links). |
| `model_type` | String(32) | no | Allowed owner type. |
| `model_id` | Integer | no | Owner id (polymorphic). |
| `created_at` | DateTime(tz) | no | `server_default=now()`. |

Unique `(tag_id, model_type, model_id)` (`uq_tag_links_tag_owner` — can't tag the same thing twice); index `(model_type, model_id)`.

### 3.4 `notes` — new table (migration `0024`)

| Column | Type | Null | Notes |

### 3.5 `item_definitions.custom_fields` + `stock_instances.custom_fields` (migrations `0025`, `0026`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `custom_fields` | Text | yes | JSON object string; flat map `str → (str\|int\|float\|bool\|null)`. NULL = none. (De)serialized + validated by Pydantic in the service/schema layer; **no** DB JSON functions (portability, §2.11). |

> Stored as `Text` (not a DB-native JSON type) to stay portable. The map is validated app-layer: keys non-empty + length-capped, value scalars only, a sane field-count cap (default 50).

## 4. Backend design

### 4.1 Layering (extends M2/M3/M4)

- **Repositories** (`app/repositories/`): `MediaFileRepository` (get-by-hash, create, delete, count-references-for), `AttachmentRepository` (create, list-for-owner, get, delete, list-by-media-file), `TagRepository` (CRUD, get-by-name-ci, list), `TagLinkRepository` (link, unlink, list-for-owner, list-owners-for-tag, exists), `NoteRepository` (CRUD, list-for-owner), `BarcodeRepository` (create, get-by-code, list-for-definition, delete). All extend the existing `q`-filter list patterns; **no raw queries** outside repositories.
- **Services** (`app/services/`):
  - `MediaStorage` (`app/services/media_storage.py`) — the only thing that touches the filesystem: `store(bytes, declared_type) -> MediaFile` (validate → downscale → hash → write-if-absent → upsert row), `path_for(media_file)`, `delete_physical(media_file)`. Pure I/O + Pillow; no HTTP concerns.
  - `AttachmentService` — `upload(model_type, model_id, file) -> Attachment` (resolve owner exists → `MediaStorage.store` → create reference), `list_for(owner)`, `delete(attachment_id)` (delete reference → if ref-count now 0, delete `media_files` row + physical file), and `delete_for_owner(model_type, model_id)` (the cascade helper).
  - `TagService` — tag CRUD (case-insensitive dup guard → `tag.duplicate_name`), `attach`/`detach`(`model_type,model_id`), `list_for_owner`, `set_tags_for_owner` (replace set), `detach_all_for_owner` (cascade helper).
  - `NoteService` — note CRUD scoped by owner; `delete_for_owner` (cascade helper).
  - `BarcodeService` — bind a code to a definition (global-unique guard → `barcode.duplicate`), unbind, list for a definition; **`lookup(code)`** delegating to the provider chain.
  - `ProductLookupService` / providers (`app/services/product_lookup/`) — a `ProductLookupProvider` Protocol (`lookup(code) -> ProductLookupResult | None`) + `InternalProvider` (resolves via `BarcodeRepository.get_by_code` → the bound definition). The service iterates the configured provider list and returns the first hit.
  - `SearchService` + `SearchProvider` (`app/services/search/`) — a `SearchProvider` Protocol (`search(q, types, limit) -> SearchResults`) + `LikeSearchProvider` (queries each entity repo, §4.5). The service is the seam for a future semantic provider.
  - `ExportService` — `export(entity, format) -> Iterator[str]` (streamed CSV/JSON rows), reading through existing services/repositories.
  - A shared **owner-resolution** helper (`app/services/owners.py`) maps `model_type → repository.get` so attachments/tags/notes validate that the owner exists and so the cascade hooks know what to clean.
- **Generic cascade**: the existing `ItemDefinitionService.delete`, `StockInstanceService.delete`, and `LocationService.delete` each call `AttachmentService.delete_for_owner`, `TagService.detach_all_for_owner`, `NoteService.delete_for_owner` for their `(model_type, id)` **before** removing the row (same transaction; the physical-file deletion is a post-commit best-effort step, §4.2). This is the polymorphic price (§2.8): the DB can't cascade `model_id`.

### 4.2 Media storage, validation & serving (the easy-to-get-wrong I/O)

- **Upload** (`POST /attachments`, multipart): the route reads the `UploadFile`, the service caps total bytes (default 25 MB → `attachment.file_too_large`/413), sniffs the type. **Images**: open with Pillow (`Image.open` + `.verify()`); reject undecodable → `attachment.unsupported_type`/415; record `width`/`height`; if either edge > `media.max_image_edge` (default 2048) re-encode downscaled (preserving aspect + EXIF orientation). **Non-images**: an allow-list of content types (PDF, plain text, common docs) — anything else → `attachment.unsupported_type`. **SVG is rejected** (active content). The validated `content_type` is what we persist and later serve (never the raw client header).
- **Content-address + dedup**: hash the (possibly re-encoded) bytes; if a `media_files` row with that `sha256` exists, reuse it (no second write); else write the file under the sharded path and insert the row. Writes are **atomic** (temp file + rename).
- **Ref-count delete**: removing an attachment deletes only the reference row; the service then checks `MediaFileRepository.count_references` and, at zero, deletes the `media_files` row and unlinks the physical file. Physical unlink and downscale-write happen **outside** the DB transaction (best-effort; a failed unlink is logged, never crashes the request — a stray file is harmless and sweepable).
- **Serving**: a `StaticFiles` mount at `/media` over `DATA_DIR/media`, **registered before** the SPA catch-all. Basic safe defaults: serve with the stored `content_type`, `X-Content-Type-Options: nosniff`, and `Content-Disposition: attachment` for non-image types. Per-request authorization + deeper sanitization = M6 (§1 non-goals). The mount is conditional on the media dir existing (created on startup from `data_dir`).

### 4.3 Polymorphic owners & cascade

A single `OWNER_TYPES = {"item_definition", "stock_instance", "location"}` registry (one place) feeds: schema validation (`model_type` must be in the set → else `validation.invalid_input`), owner-existence checks (`resolve_owner(model_type, model_id)` → 404 of the owner's not-found code if missing), and the cascade map used by the three `delete` services. Adding a future owner type (e.g. `category`) is a one-line registry change.

### 4.4 Barcode lookup & the provider seam

```
GET /barcodes/lookup?code=<scanned>
  → ProductLookupService.lookup(code):
      for provider in providers:            # M5: [InternalProvider]
          result = provider.lookup(code)
          if result is not None:
              return result
      return None
```

`ProductLookupResult` carries `source` (`"internal"`), an optional matched `definition_id` (+ a lightweight definition summary), and an optional `draft` (name/brand/category hints) for the not-yet-in-system case. M5's `InternalProvider` only ever returns a `definition_id` match. The endpoint returns `{ found: bool, source, definition, draft }`. **Future** providers (Open Food Facts, LLM vision — M9) implement the same Protocol and are appended to the list behind a settings toggle; the scan flow and frontend don't change.

### 4.5 Global search (LIKE) & the provider seam

`LikeSearchProvider.search(q, types, limit)` runs, for each requested type (default all):
- **item_definition**: name/description LIKE (reuse the existing `q` filter) **plus** `custom_fields` text LIKE.
- **stock_instance**: serial / model_number / manufacturer LIKE, **plus** a join to `barcodes.code`, **plus** `custom_fields` text LIKE.
- **location** / **category**: name LIKE (reuse existing `q` filters).
- **tag**: name LIKE.

Each type is independently capped (default 20) and the result is `{ type: [summaries], ... , totals }`. All case-insensitive `func.lower(...).contains(...)` (the established portable pattern). The `SearchProvider` Protocol is the reserved seam for an LLM semantic provider (M9) — `SearchService` just iterates/merges providers.

### 4.6 Export

`ExportService.export(entity, format)` streams rows for `entity ∈ {item_definitions, stock_instances, locations}`:
- **CSV** via stdlib `csv` into a generator (header row + data rows), foreign keys flattened to **id + resolved name** columns (`category_id`,`category_name`,`location_id`,`location_name`,`definition_id`,`definition_name`), `custom_fields` as a JSON string column, tags as a comma-joined `tags` column. `Decimal`/dates rendered with stable string formats (no locale).
- **JSON** as a streamed array of the same flattened records.

Served via `StreamingResponse` with `Content-Disposition: attachment; filename="<entity>-<date>.csv"`. Authenticated (session). Read-only through the repository/service layer (§2.11).

> `model_type` is validated against the owner registry (§4.3); a bad value → `validation.invalid_input`, a missing owner → that owner's not-found code.

### 4.8 Schemas (`app/schemas/`)

- `AttachmentResponse` — `id, model_type, model_id, media: {sha256, content_type, byte_size, width, height, url}, original_filename, title, sort_order, created_at`. (`url` = `/media/<ab>/<sha256>`.) `AttachmentUpdate` — `title?, sort_order?`.
- `TagResponse` / `TagCreate` / `TagUpdate` (`name`, `color?`); `TagLinkResponse`; `TagSetRequest` (`model_type, model_id, tag_ids[]`).
- `NoteResponse` / `NoteCreate` (`model_type, model_id, body`) / `NoteUpdate` (`body`).
- `BarcodeResponse` / `BarcodeCreate` (`code`, `symbology?`, `label?`); `BarcodeLookupResponse` (`found, source, definition?, draft?`).
- `SearchResponse` — `{ item_definitions: [...], stock_instances: [...], locations: [...], categories: [...], tags: [...], totals: {...} }` with lightweight per-type summaries.
- `Export` — no response schema (streamed file); the path is documented with a binary/file media type.
- `CustomFieldsModel` — a validated `dict[str, str|int|float|bool|None]` (key/value/count caps) folded into the Definition and Instance create/update/response schemas.

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0–M4 §5 (`make check` all green; build; `make codegen` no-drift). The **logic that must have unit tests** in M5:

**Backend**
- **Media storage & ref-count** (§4.2 — required): identical bytes uploaded twice ⇒ **one** `media_files` row + **two** attachments (dedup); deleting one attachment keeps the file; deleting the **last** deletes the `media_files` row **and** the physical file (no orphan); deleting an **owner** cascades all its attachments/tags/notes and drops now-unreferenced files; a failed physical unlink is logged, not raised.
- **Image validation/downscale**: a non-image (or corrupt image) upload is rejected (`attachment.unsupported_type`); an oversize upload → `attachment.file_too_large`; an over-large image is downscaled (dimensions recorded; capped edge); SVG rejected; the persisted `content_type` is our validated value.
- **Polymorphic owner validation/cascade** (§4.3): a bad `model_type` → `validation.invalid_input`; a missing owner → the owner's not-found code; the cascade map cleans exactly the right rows.
- **Tags**: case-insensitive duplicate name → `tag.duplicate_name`; attach is idempotent (unique link); `set_tags_for_owner` replaces the set; deleting a tag drops its links; detach-all on owner delete.
- **Notes**: CRUD scoped to owner; cascade on owner delete; 404 on foreign/missing id.
- **Barcodes**: binding a code already bound (to the same or another definition) → `barcode.duplicate`; a definition delete cascades its codes; `lookup` returns the internal match and `found=false` for an unknown code.
- **Custom fields**: valid map round-trips on definition + instance; nested/object values rejected; key/count caps enforced; NULL = none; **searchable** (a value substring surfaces the owner in global search).
- **Search**: each type matched by substring (definition incl. custom fields; instance incl. serial + barcode + custom fields; location/category/tag by name); per-type cap honoured; case-insensitive; empty `q` handled.
- **Export**: CSV header + rows; foreign keys flattened to id+name; tags joined; custom_fields serialized; CSV escaping correct (commas/quotes/newlines in values); JSON shape matches; streamed (not buffered whole).
- **Migration round-trip**: `0020`–`0027` upgrade clean on a DB at `0019` and downgrade cleanly; existing rows unaffected (new columns NULL).

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): the **AttachmentPanel** uploads (mock multipart), renders the gallery from the media URL, and deletes; **tag** chips + picker attach/detach and the list **tag filter** narrows; **notes** panel CRUD; the **custom-fields editor** adds/edits/removes KV rows and round-trips; the **barcode scanner** decodes (decoder mocked) → calls lookup → known code routes to the item, unknown offers create; **global search** input → results page grouped by type; **export** buttons trigger the download; dates/numbers via M1.5 `formatDate`/`formatQuantity`; all strings present in **en + zh**.

## 11. 🟢 Deploy self-test points (stitched into the M5 milestone walkthrough)

Manual walkthrough the author runs at milestone end (expands the roadmap M5 🟢 bullet). Assumes the M1–M4 run flow (compose up over **HTTPS or localhost** so the camera works; logged in as the admin; a location tree, a category, and some stock exist).

1. **Attach a photo:** on an item (or instance/location) detail, upload an image → it appears in the **gallery**. Upload the **same** image again → it shows but no second file is written (dedup). Upload a non-image (PDF) → shows as a downloadable file; upload a `.exe`/SVG → rejected with a clear message.
2. **Ref-count delete:** delete the attachment → the gallery clears; the underlying file is gone once the last reference is removed (a still-referenced shared file survives).
3. **Tags:** create a couple of tags (with colours), attach them to items across types; **filter** the items list by a tag.
4. **Notes:** add a note to an item; edit and delete it.
5. **Custom fields:** add custom fields to a definition (e.g. `voltage=220`) and an instance; they render on detail and round-trip on edit.
6. **Global search:** search a term that lives in a name, a serial, a **custom field value**, and a **barcode** → results come back **grouped by type**; each links to its subject. Switch ZH⇄EN — labels follow.
7. **Barcode bind + scan:** bind a code to a definition; then **scan** that code with the camera → it routes to the item and offers fast intake. Scan an **unknown** code → offered "create item" with the code pre-bound.
8. **Export:** export item definitions / instances / locations as **CSV** and **JSON** → open the CSV in a spreadsheet; foreign keys show as id **and** name; tags and custom fields are present.
9. **Backup (manual, deferred feature):** stop the app, copy the bind-mounted `DATA_DIR` (SQLite file **+** `media/`) elsewhere, bring it up against a clean dir restored from that copy → everything (including photos) is back. (Confirms the deferred-backup story holds.)

## 12. Open questions / deferred

- **Backup/restore (re-scoped out):** M5 relies on copying `DATA_DIR`. A first-class in-app **backup download + restore** flow (and a consistent-snapshot story for a running SQLite DB) is a candidate later milestone / refinement.
- **CSV import (deferred):** the natural home is alongside the **LLM-assisted formatting** backend (§2.12, M9) — let the model map/clean arbitrary spreadsheet columns; then validate + dry-run + commit. Until then, export-only.
- **External product lookup (deferred to M9):** Open Food Facts (free, no key; strong on packaged food, patchy on local/non-food), its sister DBs, ISBN lookups, and **LLM vision** all implement the M5 `ProductLookupProvider` seam — add as providers + a settings toggle; no scan-flow/UI change.
- **LLM semantic search (deferred):** the `SearchProvider` seam fronts the LIKE provider; a semantic provider slots in later (M9-era).
- **Thumbnails (deferred):** M5 serves capped-size originals with lazy-loading. A server-side thumbnail pipeline (and HEIC/AVIF support, which needs system libs) is a refinement.
- **Media authorization & content sanitization (M6):** per-request authorization on `/media` (vs the current capability-URL static mount) + upload sanitization + active-content handling land in the M6 hardening pass (roadmap §5 M6 / §2 open questions).
- **Custom fields structure:** JSON KV is searchable as text but gives up typed/range queries, key-consistency facets, and per-field typed UI. If real use demands structured filtering, a `custom_field_definitions` + values model is a contained follow-up.
- **Tag colour / palette + nestable tags:** flat + free colour for now; a managed palette and (if ever wanted) the roadmap's original "nestable tags" are refinements.
- **Attachment reordering / cover image:** `sort_order` exists; a richer drag-reorder + a designated cover image is a UI refinement.
- **Content-hash collisions / cross-owner dedup:** sha256 collision is negligible; cross-owner dedup is intentional (one file, many references) and handled by the ref-count.
