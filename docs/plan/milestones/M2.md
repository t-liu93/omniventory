## 1. Goal & non-goals

**Goal (the roadmap's M2 promise):** turn the static M1 instance registry into a **living stock ledger**. Every change to a lot's quantity becomes a **typed, append-only movement**; the lot's quantity is **derived** from that ledger and never blind-overwritten (roadmap §2.3 — the red line M2 exists to honour). On top of the ledger M2 delivers **FIFO consumption**, **undo/reversal**, per-definition **`min_stock`** with a **low-stock** signal, and the **three per-definition stock-tracking modes** (`exact` / `level` / `none`) that the M1 §12 forward-requirement asked us to design here.

**Done when (🟢, expanded in §11):** intake 10 of a consumable then consume 3 (FIFO) and see the derived quantity become 7; undo the consume and watch it restore to 10; set `min_stock` on the definition, drop the stock below it, and see low-stock surface on the dashboard; register a `level`-mode item and flip it to "low"; register a `none`-mode item with no quantity at all. CI stays green, including the no-drift contract gate.

**Non-goals (explicitly out of M2 — deferred or later milestones):**
- **No best-before / expiry.** `best_before_date` and `default_best_before_days` are **M3**. M2's FIFO orders lots by `received_at` only; M3 prepends `best_before` as the primary FIFO key (§4.3).
- **No reminders.** Low-stock is a **computed read** (an endpoint + a dashboard tile) in M2; the proactive "N days ahead" notification engine that *consumes* the low-stock signal is **M4**.
- **No multi-unit conversion.** Movements use the definition's single `unit` string (M1). Buy-by-the-case / use-by-the-piece is **M8**.
- **No partial split-move.** A `move` relocates a **whole lot** (§2). Splitting part of a lot's quantity into a new lot at the destination is deferred (§12).
- **No cross-cutting features.** Photos/attachments, tags, notes, custom fields, barcodes are **M5**; movements carry a plain `note` string only.
- **No soft-delete / archive.** Deleting a lot **cascade-deletes its movements** (§3.3). Audit-preserving soft-delete is deferred (carried from M1 §12).
- **No multi-user / roles beyond M0.** Movements record the acting `user_id` (the single admin today) as the spine for M6; no role checks yet.
- **No qualitative-level ledger.** `level` and `none` modes deliberately have **no movement ledger** (M1 §12); only `exact` mode does.

## 2. Locked decisions (settled in M2 planning; rationale in roadmap §2/§3 + investigation ch.3)

| Area | Decision |
|---|---|
| **Stock-tracking mode** (M1 §12) | A per-definition **`stock_tracking_mode`** with **all three modes implemented in M2**: `exact` (Decimal quantity from the ledger), `level` (qualitative `high`/`medium`/`low`, manually set, **no ledger**), `none` (presence-only, **no quantity, no ledger**). Stored as a **String validated app-layer** against `STOCK_TRACKING_MODES` (no DB CHECK — the set may grow; roadmap §2.11), default **`exact`**. (The code *branches* on this value, so — unlike M1's `kind` which is a no-logic UI hint and therefore an FK lookup — the mode is a closed app-level constant set.) |
| **Quantity is ledger-derived (the red line)** | For `exact` lots, `stock_instances.quantity` is a **denormalized cache**; the **`stock_movements` ledger is the source of truth**. After every movement the service **recomputes** `quantity = SUM(quantity_delta)` for that lot **inside the same transaction** — it **never** does `quantity += delta` (no blind-overwrite, roadmap §2.3). A unit-tested invariant pins `quantity == SUM(deltas)`. Chosen over live-`SUM`-on-read to keep list reads cheap and preserve M1's API shape; portability is kept because the recompute is **app-layer, not a SQL trigger** (§2.11). |
| **Append-only ledger** | Movements are **immutable**: never edited, never deleted (except cascade with their lot). A mistake is fixed by **appending** a compensating movement, not mutating history. |
| **Undo = reversal entry** | "Undo" appends a **reversal movement** with `quantity_delta = −original`, `type = correction`, and `reverses_movement_id → original`. A movement can be reversed **at most once** (partial-unique on `reverses_movement_id`), a **reversal cannot itself be reversed**, and a reversal that would drive the lot **negative is rejected** (§4.4). |
| **FIFO consume** | `consume` is a **definition-level** operation: it walks the definition's `exact` lots **oldest-first by `(received_at, id)`** (M3 prepends `best_before`), decrementing each via a `consume` movement until the requested quantity is met. **Insufficient total stock is rejected** (422) — M2 never lets stock go negative. |
| **Move semantics** | **Whole-lot only.** `move` changes the lot's `location_id` and records a `move` movement (`delta = 0`, `from_location_id` → `to_location_id`). Partial split-move is deferred (§12). |
| **`min_stock` & low-stock** | `min_stock` is a nullable `Numeric` on the **definition** (defaults live on the definition, roadmap §2.4), **meaningful only for `exact` mode**. Low-stock is **computed**: `exact` → `SUM(lot quantities) < min_stock` (strictly below; see §12 for the `<` vs `≤` note) when `min_stock` is set; `level` → **any** lot is at `low`; `none` → never. |
| **`received_at`** | A new `stock_instances.received_at` (`DateTime(tz)`, default `now()`, **back-datable** on intake) — the FIFO ordering key and the physical-receipt timestamp from the roadmap spine §3. Distinct from `created_at` (row creation) so back-dated intake orders correctly. |
| **`quantity` becomes nullable** | `exact` lots always carry a numeric quantity (service-managed); `level`/`none` lots carry **`NULL`** (the signal is `stock_level`, or nothing). The M1 serial CHECK is rewritten to `serial IS NULL OR quantity IS NULL OR quantity = 1` (batch-mode migration). |
| **Create vs. movement** | `POST /instances` still accepts an initial `quantity`; for `exact` mode the service **records an initial `intake` movement** equal to it (so even registration goes through the ledger — this is also what the M1 §12 backfill does for pre-existing rows). `quantity` is **removed from `InstanceUpdate`**: once created, an `exact` lot's quantity changes **only** through movement endpoints. `stock_level` is added to create/update for `level` mode. |
| **Movement actor** | Every movement stores `user_id` (the acting user from `RequestContext`, nullable FK → `users`) — the audit spine for M6. Today it is always the single admin; the backfill sets it `NULL` (system). |
| **API shape** | **Explicit, intention-revealing operation endpoints** (`/intake`, `/move`, `/adjust`, `/discard`, `/consume`, `/reverse`) rather than one overloaded generic-movement POST — each has a distinct required-field set, so the typed contract stays self-documenting and the client code stays simple. |
| **Empty lots are kept** | A lot consumed to `0` is **retained** (history + identity), and FIFO simply skips `quantity = 0` lots. No auto-deletion. |

> These extend M0/M1/M1.5 foundations; they are **not** "tech stack" changes, so `AGENTS.md` does **not** change for M2. The ledger schema and the new error codes flow through the existing no-drift contract gate (§6) and the M1.5 uniform error envelope (§4.7).

## 3. Data model

Three schema deltas (two existing tables altered, one new table) + the mode semantics. All use the M0 conventions: SQLAlchemy 2.0 typed `Mapped[...]` on the shared `Base`, `created_at` via `server_default=func.now()`, access **only** through a repository, business rules in the service layer.

### 3.1 `item_definitions` — additions (migration `0010`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `stock_tracking_mode` | String(16) | no | `server_default='exact'`; validated app-layer against `STOCK_TRACKING_MODES = ("exact","level","none")`. No DB CHECK (roadmap §2.11). |
| `min_stock` | Numeric(18,6) | yes | reorder point; **only meaningful for `exact`** mode. `NULL` = no low-stock threshold. |

> `default_best_before_days` stays **absent** (added by M3, its consumer — M1 §2 "Definition defaults timing"). A plain `op.add_column` suffices for both (nullable / server-defaulted columns need no batch rebuild on SQLite).

### 3.2 `stock_instances` — alterations (migration `0012`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `quantity` | Numeric(18,6) | **yes** (was `no`) | `exact` lots: ledger-derived cache. `level`/`none`: `NULL`. |
| `stock_level` | String(16) | yes | **new**; `level` mode only: `high`/`medium`/`low`, validated app-layer against `STOCK_LEVELS`. `NULL` for `exact`/`none`. |
| `received_at` | DateTime(tz) | yes | **new**; `server_default=func.now()`; FIFO key + physical-receipt time. Back-dateable on intake. (Nullable so the column adds cleanly; the service always sets it for `exact` lots.) |

- **CHECK rewrite** (batch mode — SQLite can't `ALTER`/re-`CHECK` in place): `ck_stock_instances_serial_qty_1` becomes **`serial IS NULL OR quantity IS NULL OR quantity = 1`**. The partial unique index `(definition_id, serial) WHERE serial IS NOT NULL` is unchanged.
- The serial constraint is **still enforced in the service** for `exact` lots: any movement (esp. `intake`) that would push a *serialized* lot's recomputed quantity off `1` is rejected (§4.3).

### 3.3 `stock_movements` — the ledger (migration `0011`)

The append-only typed transaction log. **Current quantity of an `exact` lot = `SUM(quantity_delta)` over its movements.**

| Column | Type | Null | Notes |
|---|---|---|---|
| `id` | Integer PK | no | |
| `instance_id` | FK → `stock_instances.id` | no | the lot affected; **`ondelete="CASCADE"`** (a lot's ledger dies with the lot — no soft-delete in M2). |
| `type` | String(20) | no | `intake` / `consume` / `move` / `adjust` / `discard` / `correction`; validated app-layer against `MOVEMENT_TYPES`. |
| `quantity_delta` | Numeric(18,6) | no | **signed**: `+` intake, `−` consume/discard, `±` adjust/correction, **`0`** for move. |
| `from_location_id` | FK → `locations.id` | yes | `ondelete="SET NULL"`; set on `move`. |
| `to_location_id` | FK → `locations.id` | yes | `ondelete="SET NULL"`; set on `move` (and recorded on `intake` for provenance). |
| `occurred_at` | DateTime(tz) | no | `server_default=func.now()`; when it physically happened (back-dateable). |
| `note` | String(1000) | yes | free-text; rich attachments/tags are M5. |
| `reverses_movement_id` | FK → `stock_movements.id` (self) | yes | `ondelete="SET NULL"`; the original this entry reverses. **Partial unique** `WHERE reverses_movement_id IS NOT NULL` (a movement is reversed at most once). |
| `user_id` | FK → `users.id` | yes | `ondelete="SET NULL"`; acting user (M6 audit spine). |
| `created_at` | DateTime(tz) | no | row creation. |

Indexes: `ix_stock_movements_instance_id`; `ix_stock_movements_instance_occurred` on `(instance_id, occurred_at)` for history ordering; the partial-unique on `reverses_movement_id`.

> **Data backfill (in `0012`, after the table exists in `0011`).** Every pre-existing `stock_instance` (all `exact` by default) gets **one `intake` movement** with `quantity_delta = its current quantity`, `occurred_at = received_at = created_at`, `user_id = NULL` (system). This realises M1 §12's promise — quantity becomes ledger-derived without changing any displayed number — and is part of `0012`'s `upgrade()` (downgrade deletes those movements before reverting the column changes).

### 3.4 The three tracking modes (semantics)

| Mode | `quantity` | `stock_level` | Ledger? | Movements allowed | Low-stock rule |
|---|---|---|---|---|---|
| **`exact`** (default) | Decimal, ledger-derived | `NULL` | **yes** | intake / consume / move / adjust / discard / correction (+ reverse) | `SUM(qty) < min_stock` when `min_stock` set |
| **`level`** | `NULL` | `high`/`medium`/`low` (manual) | no | none (movement endpoints reject) | **any** lot at `low` |
| **`none`** | `NULL` | `NULL` | no | none | never |

Cross-field validation (service layer, §4 + §5): a lot's fields must match its definition's mode — `exact` ⇒ numeric quantity, no `stock_level`; `level` ⇒ `stock_level` set, no quantity; `none` ⇒ neither. A movement against a non-`exact` definition is rejected. **Switching a definition's mode** when it already has lots is constrained (§4.1, §12).

### 3.5 Migration list

| Rev | Step | Does |
|---|---|---|
| `0010` | 1 | `item_definitions`: add `stock_tracking_mode` (default `exact`) + `min_stock`. |
| `0011` | 2 | create `stock_movements` (the ledger). |
| `0012` | 3 | `stock_instances`: add `stock_level` + `received_at`; **batch-alter** `quantity` → nullable and rewrite the serial CHECK; **backfill** one `intake` movement per existing lot + set `received_at = created_at`. |

All reversible (downgrade undoes in reverse; `0012` downgrade deletes the backfilled movements, batch-reverts the CHECK + `quantity NOT NULL`, drops the two added columns).

## 4. Backend design

### 4.1 Repository + service layering (extends M1)

- **Repositories** (`app/repositories/`): new `StockMovementRepository` (`append`, `get`, `list_for_instance`, `sum_delta_for_instance`, `find_reversal_of`, `delete_for_instance`). Extend `StockInstanceRepository` with `sum_quantity_for_definition` and `list_active_lots_for_definition` (quantity > 0, ordered by `received_at, id`). Extend `ItemDefinitionRepository` only as needed for the low-stock scan. **Pure data access; no business rules; no raw queries in routes.**
- **Services** (`app/services/`):
  - `StockInstanceService` (extended): mode-aware create/update; the `recompute_quantity(instance)` helper (`= sum_delta_for_instance`); records the **initial `intake`** on `exact`-mode create; cross-field/mode validation; serial⇒qty=1 re-check after recompute.
  - **`StockMovementService`** (new): the operations — `intake`, `discard`, `adjust`, `move`, `consume_fifo`, `reverse`. Each appends movement(s), recomputes the affected lot(s) **in one transaction**, and enforces the mode/serial/non-negative guards. This is the home of the **easy-to-get-wrong logic** (§5).
  - **`LowStockService`** (new, small): the computed low-stock scan (§4.5).
- **Mode changes on a populated definition:** the service **rejects** changing `stock_tracking_mode` when the definition already has lots whose fields would become invalid under the new mode (e.g. `exact`→`none` while lots carry ledger history) → `item_definition.tracking_mode_change_conflict` (409). Clean re-modelling is a deferred concern (§12).

### 4.2 Quantity derivation (cache + recompute)

The single rule, applied everywhere a movement is written:

```
within one DB transaction (per request):
    repo.append(movement)                    # the ledger is the source of truth
    inst.quantity = repo.sum_delta_for_instance(inst.id)   # recompute the cache
    # never: inst.quantity = inst.quantity + delta   (no blind-overwrite — roadmap §2.3)
    assert_serial_qty_1(inst.serial, inst.quantity)        # serialized lots stay at 1
```

`sum_delta_for_instance` is a `SELECT COALESCE(SUM(quantity_delta), 0)` — `Decimal` throughout (never float, roadmap §2.9). The invariant `quantity == SUM(deltas)` is asserted in tests after every operation sequence (§5).

### 4.3 Movement operations & FIFO

- **`intake(instance, quantity, occurred_at?, note?)`** — append `intake` (`+quantity`), recompute. For a serialized lot, intake that would exceed `1` is rejected. Creating an instance (`POST /instances`, `exact`) records the **initial** intake.
- **`discard(instance, quantity, ...)`** — append `discard` (`−quantity`); rejected if it would go negative.
- **`adjust(instance, counted_quantity, ...)`** — stock-take to an **absolute** value: `delta = counted_quantity − current`; append `adjust` (signed); rejected if `counted_quantity < 0`.
- **`move(instance, to_location_id, ...)`** — append `move` (`delta = 0`, `from = inst.location_id`, `to = to_location_id`); set `inst.location_id = to_location_id`. Whole-lot only (§2). `to_location_id` must exist.
- **`consume_fifo(definition, quantity, ...)`** — the headline:
  ```
  assert definition.mode == 'exact'
  lots = repo.list_active_lots_for_definition(def.id)   # quantity > 0, ORDER BY received_at, id
  if sum(lot.quantity for lot in lots) < quantity: raise stock.insufficient (422)
  remaining = quantity
  for lot in lots:
      take = min(lot.quantity, remaining)
      movement(lot, type=consume, delta = -take, occurred_at)
      recompute(lot)
      remaining -= take
      if remaining == 0: break
  ```
  Produces one `consume` movement **per lot touched**; spans multiple lots oldest-first; partial-lot consumption is exact `Decimal`.
- All operations reject definitions/instances not in `exact` mode (`stock.movement_not_applicable`, 409).

### 4.4 Reversal / undo

```
reverse(movement_id, note?):
    m = get(movement_id) or 404
    if m.reverses_movement_id is not None: raise stock.cannot_reverse_reversal (409)
    if repo.find_reversal_of(m.id) is not None: raise stock.movement_already_reversed (409)
    inst = m.instance
    # compute prospective new quantity; reject if it would go negative
    if inst.quantity - m.quantity_delta < 0: raise stock.reverse_would_go_negative (409)
    r = movement(inst, type=correction, delta = -m.quantity_delta,
                 reverses_movement_id = m.id, occurred_at = now, note)
    if m.type == 'move':                       # restore location too
        r.from_location_id, r.to_location_id = m.to_location_id, m.from_location_id
        inst.location_id = m.from_location_id
    recompute(inst)
```

The 🟢 "undo restores" path = reversing the `consume`. The non-negative guard means undoing an old `intake` after the stock was consumed is rejected (keeps the ledger consistent); undoing the **latest** action always succeeds. Partial-unique on `reverses_movement_id` is the DB backstop for the once-only rule.

### 4.5 Low-stock (computed)

`GET /low-stock` → a list of low definitions with the reason and the numbers the UI needs:

```
for def in all definitions:
    if def.mode == 'exact' and def.min_stock is not None:
        total = repo.sum_quantity_for_definition(def.id)
        if total < def.min_stock:
            yield { definition_id, name, mode:'exact', reason:'below_min_stock',
                    current: total, threshold: def.min_stock }
    elif def.mode == 'level':
        if any lot.stock_level == 'low':
            yield { definition_id, name, mode:'level', reason:'level_low',
                    current: null, threshold: null }
    # none → skip
```

Pure read; no persistence. M4 will *consume* this signal for proactive reminders.

- `InstanceCreate` gains `stock_level`; keeps `quantity` (initial intake, `exact`). `InstanceUpdate` **drops `quantity`**, gains `stock_level`. `InstanceResponse` gains `stock_level` + `received_at`, and `quantity` becomes `Decimal | None`.
- New: `MovementResponse` (full ledger row incl. `type`, `quantity_delta`, locations, `occurred_at`, `reverses_movement_id`, `user_id`); operation bodies `IntakeRequest` / `DiscardRequest` / `AdjustRequest` / `MoveRequest` / `ConsumeRequest` / `ReverseRequest`; `LowStockItem` (the §4.5 shape). `Decimal` on the wire for all quantities/money (roadmap §2.9).

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0/M1/M1.5 §5 (`make check` = ruff + mypy + pytest + eslint + tsc + vitest, all green; build passes; `make codegen` no-drift). The **logic that must have unit tests** in M2 (roadmap DoD — quantity math, stock in/out, consumption order, threshold triggers):

**Backend**
- **Ledger invariant:** after any sequence of operations, `instance.quantity == SUM(quantity_delta)`; the cache is **never** blind-set.
- **Intake / discard / adjust:** intake adds; discard subtracts; discard/adjust below `0` rejected (`stock.negative_quantity`); adjust computes the correct signed delta to the absolute counted value.
- **FIFO consume:** single lot; **spanning multiple lots oldest-first** by `(received_at, id)`; partial-lot `Decimal` precision; **insufficient total rejected** (`stock.insufficient`) with nothing written; `level`/`none` definitions rejected.
- **Move:** whole-lot location change recorded with `from`/`to`, `delta = 0`; non-existent `to_location_id` → 404; quantity unchanged.
- **Reversal / undo:** reversing a `consume` restores the quantity (the 🟢 path); a movement reversed twice → `stock.movement_already_reversed`; reversing a reversal → `stock.cannot_reverse_reversal`; a reversal that would go negative → rejected; reversing a `move` restores `location_id`.
- **Serial ⇒ qty=1 under the ledger:** an `intake` that would push a serialized lot above `1` is rejected; the rewritten DB CHECK still blocks a direct bad write (`serial IS NULL OR quantity IS NULL OR quantity = 1`).
- **Modes:** `exact` create records an initial intake; `level` requires `stock_level` and forbids quantity/movements; `none` forbids both quantity and `stock_level` and all movements; field/mode mismatch → `instance.field_mode_mismatch`; mode change on a populated definition → 409.
- **Low-stock:** `exact` below / at / above `min_stock` (strictly-below boundary); `exact` with no `min_stock` → never flagged; `level` low vs not; `none` never; mixed set returns the right reasons + numbers.
- **Migration round-trip:** `0010`–`0012` upgrade clean on a DB at `0009` and downgrade cleanly, **including the `0012` batch-alter (CHECK rewrite + nullable quantity) and the intake backfill** (and its deletion on downgrade).

**Frontend** (vitest + Testing Library, mock the typed client — M0 style): definition form mode-switch shows/hides `min_stock`; instance form branches by mode (quantity vs `stock_level` vs neither) and keeps the client-side serial⇒qty=1 rule; consume/intake/adjust/move/discard call the right endpoint and surface server errors via `mapApiError`; movement-history renders; reverse action works; low-stock dashboard tile renders the count + list.

## 6. Contract-first codegen & no-drift gate

Unchanged mechanism (M0 §6 / M1 §6): every step that touches the API **re-runs `make codegen`** (regenerates repo-root `openapi.json` + `frontend/src/api/schema.d.ts`) and commits both; the CI **contract** job (`make codegen` + `git diff --exit-code`) fails on drift. M2 grows the schema with the operation endpoints, `MovementResponse`/`LowStockItem`, the new definition/instance fields, and the new error codes; the frontend consumes the new typed paths via the existing `openapi-fetch` client.

## 7. Frontend design

### 7.1 Definition form & detail (mode + `min_stock`)

- `DefinitionFormModal` (in `pages/Items.tsx`) gains a **`stock_tracking_mode`** `Select` (exact / level / none, localized) and a **`min_stock`** `NumberInput` shown **only when mode = exact**. The definition detail card shows the mode (a badge) and `min_stock`.
- Editing the mode of a definition that already has lots surfaces the 409 (`tracking_mode_change_conflict`) via `mapApiError`.

### 7.2 Instance form (mode-aware) — `components/InstanceFormModal.tsx`

The modal already owns the client-side serial⇒qty=1 rule (M1 §7.3). M2 makes it **branch on the parent definition's mode** (passed in alongside the existing `definitions`/`locations` props):
- **exact:** `quantity` field (initial intake on create; **hidden/locked on edit** — quantity changes go through ledger actions) + serial⇒qty=1.
- **level:** a `stock_level` `Select` (high/medium/low); no quantity.
- **none:** neither; just identity/location/durable fields.

### 7.3 Ledger actions + movement history (`ItemDetail` + `InstanceDetail`)

- **`ItemDetail`** (exact definitions): a **Consume** button (FIFO, `POST /definitions/{id}/consume`) with a quantity prompt; the instances table shows each lot's current `quantity` (or a `level` badge / "—" by mode) and a per-lot action menu (**Intake / Move / Adjust / Discard**). A **low-stock badge** when the definition is below `min_stock`.
- **`InstanceDetail`**: a **movement-history** table (type, signed delta, from→to, occurred_at, actor, "reversal of #N" link) with a **Reverse (undo)** action on each reversible row, plus the same per-lot action buttons. Quantity/level render by mode.
- All quantities go over the wire as **strings** (Decimal); render via `formatQuantity` / `formatDate` (M1.5 locale-aware).

### 7.4 Low-stock dashboard tile + view

- `Dashboard.tsx`: the consumable card becomes a **live low-stock tile** — a count from `GET /low-stock` and a short list (definition name + current/threshold or "low"), linking to a low-stock view (a filtered section/page). Empty state when nothing is low.

### 7.5 i18n

New strings in **both** `en` and `zh` catalogs (M1.5 rule): a new **`stock`** namespace (movement types, operation labels, FIFO/consume copy, reversal, low-stock) registered in `src/i18n/index.ts`; additions to `items` (mode/min_stock/level), `instances` (history, actions), `dashboard` (low-stock tile), and the new **error codes** under `errors`. Tests stay pinned to `en` (M1.5).

### 7.6 Tests (vitest + Testing Library)

Per §5 "Frontend": mode-switch field visibility; instance-form mode branching + serial rule; each ledger action hits the right endpoint and surfaces errors; movement-history + reverse render; low-stock tile. Keep the M0 "mock the typed client" style.

## 9. Step breakdown (atomic, ordered)

Each step is independently testable, test-backed, lands exactly one commit (one per-step autosquash in orchestrator mode), re-runs `make codegen` if it touched the API, and inherits the global DoD (§5) + the **anti-freelancing rule**: implement *only* this step — no other steps, no opportunistic refactors, no touching the host's real environment.

> **Ordering rationale:** schema first, in FK/dependency order — definition columns (`0010`) and the ledger table (`0011`) must exist before the instance alterations + backfill (`0012`) that write into the ledger. Backend operations (4) and the low-stock read (5) sit on that schema. Frontend (6–8) consumes the typed contract the backend lands.

### Step 1 — Definition: tracking mode + min_stock (backend)
- **Goal:** the per-definition `stock_tracking_mode` (default `exact`) + `min_stock`, validated and on the contract.
- **Build:** migration `0010`; `STOCK_TRACKING_MODES` + `STOCK_LEVELS` constants (a small `app/core/stock.py` or extend an existing core module); extend `ItemDefinition` model + `DefinitionCreate/Update/Response`; `ItemDefinitionService` validates the mode and stores `min_stock`; new error code `validation.unsupported_tracking_mode`; `make codegen`.
- **Tests:** default mode `exact`; invalid mode rejected (422); `min_stock` stored/echoed; migration `0010` up/down.
- **Out of scope:** ledger, instance changes, mode-change-on-populated guard (lots don't exist as movements yet — the guard lands with the service in step 4 / refined where lots gain ledgers).
- **Commit:** `feat(backend): per-definition stock tracking mode and min_stock`

### Step 2 — Stock movement ledger table + repository (backend)
- **Goal:** the append-only ledger table and its pure data-access layer.
- **Build:** `models/stock_movement.py` (all §3.3 columns, self-FK, partial-unique on `reverses_movement_id`, CASCADE on `instance_id`); migration `0011`; `StockMovementRepository` (`append`, `get`, `list_for_instance`, `sum_delta_for_instance`, `find_reversal_of`, `delete_for_instance`); `MovementResponse` schema; `MOVEMENT_TYPES` constant; `make codegen` (schema only — no endpoints yet).
- **Tests:** migration `0011` up/down; repo append/list/sum/`find_reversal_of`; partial-unique blocks a second reversal row at the DB level.
- **Out of scope:** any service operations or endpoints (step 4).
- **Commit:** `feat(backend): stock movement ledger table and repository`

### Step 3 — Instance alterations + ledger wiring + backfill (backend)
- **Goal:** make `exact` quantity ledger-derived, add `stock_level`/`received_at`, branch instance CRUD by mode, and backfill existing rows.
- **Build:** migration `0012` (add `stock_level` + `received_at`; batch-alter `quantity`→nullable + rewrite serial CHECK; backfill one `intake` per existing lot + `received_at = created_at`); update `StockInstance` model; update `Instance{Create,Update,Response}` (drop `quantity` from Update; add `stock_level`; nullable quantity in Response); `StockInstanceService` — mode-aware create (record initial `intake` for `exact`; require `stock_level` for `level`; nothing for `none`), `recompute_quantity` helper, serial⇒qty=1 re-check, field/mode validation; error codes `instance.field_mode_mismatch`, `validation.unsupported_stock_level`; `make codegen`.
- **Tests:** create per mode; backfill correctness (qty unchanged, one intake each); invariant `quantity == SUM(deltas)`; serial⇒qty=1 holds with nullable quantity (service 422 + DB CHECK); field/mode mismatch rejected; migration `0012` up/down incl. batch-alter + backfill (and downgrade deletes backfilled movements).
- **Out of scope:** the operation endpoints + FIFO/reverse (step 4); low-stock (step 5).
- **Commit:** `feat(backend): ledger-derived quantity, stock levels, and intake backfill`

### Step 4 — Movement operations service + endpoints (backend)
- **Goal:** intake / discard / adjust / move / FIFO-consume / reverse over the ledger.
- **Build:** `StockMovementService` (§4.3–4.4: each operation appends + recomputes in one txn; FIFO; reversal rules; non-negative + mode + serial guards; mode-change-on-populated guard now meaningful → `tracking_mode_change_conflict` wired into `ItemDefinitionService`); request schemas (`IntakeRequest`/`DiscardRequest`/`AdjustRequest`/`MoveRequest`/`ConsumeRequest`/`ReverseRequest`); routes (`/instances/{id}/{intake,discard,adjust,move}`, `/definitions/{id}/consume`, `/movements/{id}/reverse`, `GET /instances/{id}/movements`); the §4.7 stock error codes; `make codegen`.
- **Tests (the easy-to-get-wrong logic — required, per §5):** FIFO (single/multi-lot/partial/insufficient/precision); intake/discard/adjust (incl. negative rejection); move (location + delta 0 + history); reverse (restore, double-reverse, reverse-of-reversal, would-go-negative, move-restore); serial⇒qty=1 via intake; movements rejected on `level`/`none`.
- **Out of scope:** low-stock (step 5); frontend.
- **Commit:** `feat(backend): stock movement operations, FIFO consume, and reversal`

### Step 5 — Low-stock computed endpoint (backend)
- **Goal:** the `GET /low-stock` signal across modes.
- **Build:** `LowStockService` (§4.5) + repo helpers (`sum_quantity_for_definition`, any-lot-low); `LowStockItem` schema; `GET /low-stock` route; `make codegen`.
- **Tests:** `exact` below/at/above `min_stock`; no-`min_stock`; `level` low/not; `none` never; mixed.
- **Out of scope:** frontend tile (step 8).
- **Commit:** `feat(backend): computed low-stock endpoint`

### Step 6 — Frontend: definition mode/min_stock + mode-aware instance form
- **Goal:** author the two contract fields in the UI and branch the instance form by mode.
- **Build:** `DefinitionFormModal` mode `Select` + conditional `min_stock`; definition detail shows mode/min_stock; `InstanceFormModal` branches by the parent definition's mode (quantity vs `stock_level` vs neither; quantity locked on edit); new `stock` namespace + `items`/`instances` strings (en+zh) registered in `src/i18n/index.ts`; map the new error codes in `errors` (en+zh).
- **Tests:** mode-switch field visibility; instance-form branching + serial rule; mode-change 409 surfaced.
- **Out of scope:** ledger actions (step 7); low-stock tile (step 8).
- **Commit:** `feat(frontend): definition tracking mode, min_stock, and mode-aware instance form`

### Step 7 — Frontend: ledger actions + movement history
- **Goal:** intake/consume/move/adjust/discard flows + history + undo in the UI.
- **Build:** `ItemDetail` Consume (FIFO) + per-lot action menu + low-stock badge + per-lot quantity/level rendering; `InstanceDetail` movement-history table + per-lot actions + Reverse(undo); `stock`/`instances` strings (en+zh).
- **Tests:** each action hits the right endpoint + surfaces errors via `mapApiError`; history renders; reverse works.
- **Out of scope:** dashboard tile (step 8).
- **Commit:** `feat(frontend): stock ledger actions, FIFO consume, and movement history`

### Step 8 — Frontend: low-stock dashboard tile + view
- **Goal:** surface the low-stock signal on the dashboard.
- **Build:** `Dashboard.tsx` live low-stock tile (count + short list from `GET /low-stock`, empty state) linking to a low-stock view/section; `dashboard` strings (en+zh).
- **Tests:** tile renders count + list; empty state; link navigates.
- **Out of scope:** the M4 reminder engine that consumes this signal.
- **Commit:** `feat(frontend): low-stock dashboard tile`

## 11. 🟢 Deploy self-test points (stitched into the M2 milestone walkthrough)

Manual walkthrough the author runs at milestone end (expands the roadmap M2 🟢 bullet). Assumes the M1 run flow (compose up; logged in as the admin; a location tree + a category exist):

1. **Consumable definition (exact):** create definition `AA Batteries` (mode = exact, unit = `pcs`, `min_stock = 4`).
2. **Intake 10 → FIFO consume 3 → derived 7:** register a lot of `AA Batteries` with initial quantity **10** (an `intake` of +10 appears in its history). **Consume 3** of the definition (FIFO) → the lot's derived quantity is **7**; two `consume`/`intake` rows are visible in the ledger.
3. **Undo restores:** **reverse** the consume → quantity is back to **10**; the history shows a reversal linked to the consume.
4. **Low-stock surfaces:** consume down to **3** (below `min_stock = 4`) → the **dashboard low-stock tile** lists `AA Batteries` with current 3 / threshold 4. Top back up with an **intake** → it leaves the tile.
5. **Move (whole lot):** **move** the lot to another location → its `location_id` changes and a `move` row (from→to, delta 0) appears; reversing the move restores the original location.
6. **Adjust (stock-take):** **adjust** the lot to an absolute counted value (e.g. 5) → an `adjust` row with the correct signed delta; quantity becomes 5.
7. **Level mode:** create definition `Assorted Screws` (mode = level); register a lot with `stock_level = low` → it appears in the low-stock tile (reason: low); no quantity/ledger actions are offered.
8. **None mode:** create definition `Wall Art` (mode = none); register an instance → no quantity, no level, no ledger; never low-stock.
9. **Serial ⇒ qty=1 under the ledger:** for a serialized exact lot (qty 1), an **intake** that would push it above 1 is **rejected** (422 in the UI); the DB CHECK independently blocks a direct bad write.
10. **CI green:** the same gates pass in GitHub Actions, including the no-drift contract gate; migrations `0001`–`0012` apply cleanly to a fresh DB in the docker smoke test.

## 12. Open questions / deferred

- **Low-stock boundary `<` vs `≤`:** M2 flags `exact` low-stock at **`total < min_stock`** (matching the roadmap's "drop below it"). If the author prefers reorder-point semantics (`≤`, i.e. *at* the minimum already reorders), it's a one-line change — confirm during the manual walkthrough.
- **Partial split-move:** M2 moves whole lots only. Moving *part* of a lot's quantity into a new lot at the destination (consume-at-source + intake-at-dest, linked) is deferred — revisit when the consumable UX demands it (possibly M7 shopping-list flows).
- **Mode change on a populated definition:** M2 **blocks** changing `stock_tracking_mode` once incompatible lots exist (409). A guided re-modelling flow (e.g. exact→level discarding the ledger with confirmation) is deferred — pairs naturally with the M5/M6 soft-delete/archive direction (M1 §12).
- **Soft-delete / archive of lots & movements:** M2 hard-deletes (lot delete cascades its ledger). The audit-preserving direction (soft-delete, carried from M1 §12) is revisited with M5/M6.
- **`level` granularity:** M2 uses a fixed `high`/`medium`/`low`. Whether the set should be user-configurable, and whether `level` lives per-lot (as now) or should aggregate to the definition, is left open — revisit if real use shows friction.
- **FIFO key in M3:** M2 orders by `(received_at, id)`; M3 **prepends `best_before`** as the primary key so perishables consume nearest-expiry-first. M2's `consume_fifo` is written so M3 only changes the `ORDER BY`.
- **Consume from a specific lot:** M2 exposes only definition-level FIFO consume (+ per-lot `discard`/`adjust`). A lot-scoped consume could be added later if needed.
- **`received_at` granularity:** stored as `DateTime(tz)`; the UI may only expose a date. Revisit if precise intra-day ordering ever matters (id breaks ties today).
