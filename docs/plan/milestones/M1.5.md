## 1. Goal & non-goals

**Goal (the roadmap's M1.5 promise):** make the UI **bilingual (ZH + EN)** *before* more surfaces accrue — a foundational, cross-cutting capability done early so every later milestone (M2 ledger, M3 best-before, M4 reminders, …) authors strings **through i18n from the start** instead of hardcoding English and back-filling later. M1.5 also sets the **wire/display split** in stone: the backend speaks **stable error codes**, the frontend owns **all** human-facing text.

**Done when (🟢, expanded in §11):** you can switch language at runtime (ZH ⇄ EN) and the whole UI follows; the choice **persists with the account** across sessions/devices; the login/setup screens render in the resolved language **before** auth; a backend error shows up **localized via its code**; dates/numbers render in the active locale. CI stays green, including the no-drift contract gate.

**Non-goals (explicitly out of M1.5 — deferred or later milestones):**
- **No new domain features.** M1.5 is a cross-cutting *capability* layer; it adds exactly **one** DB column (`users.preferred_language`) and **zero** new business tables.
- **No backend-side localization.** The backend never returns human display strings to end users; it returns **codes** (+ a dev-facing English `message` for logs). Per-field localization of Pydantic validation messages is **out** (M1.5 maps the *whole* validation failure to one code; see §4.1).
- **No third+ language.** Exactly **EN + ZH**. The catalog/structure must make adding a language a *content* task (drop in a locale file), not a *code* task — but no other language ships in M1.5.
- **No translation-management tooling / extraction CI.** No Crowdin/Weblate, no automated unused-key linting. (Noted in §12.)
- **No household-level default language.** Resolution is account-bound + client-side (§4.3); a workspace-wide default is deferred (§12).
- **No locale-aware *parsing*** of user input (e.g. typing `1,5` for `1.5`). M1.5 does locale-aware **rendering** only (Intl for output); inputs stay canonical.

## 2. Locked decisions (settled in M1.5 planning; rationale in roadmap §2.6 + the planning discussion)

| Area | Decision |
|---|---|
| **Library** | **`react-i18next`** (locked by the roadmap) on top of **`i18next`**, plus **`i18next-browser-languagedetector`** for the pre-login detection chain. New frontend deps (note at wrap-up; §12). |
| **Languages & codes** | Exactly **`en`** and **`zh`** (canonical short codes). `navigator.language` values like `zh-CN` / `zh-TW` / `zh-Hans` normalize to `zh`; everything else falls back to `en`. The stored column is wide enough (`String(16)`) to allow BCP-47 subtags later **without a migration** — but M1.5 only ever stores `en`/`zh`. |
| **Error contract** (the big one) | **Full clean refactor, not an additive hedge.** Pre-1.0, no production data → we replace FastAPI's default `{"detail": ...}` body with a **uniform flat error envelope** `{ "code", "message", "params"? }` across **every** error path (domain `AppError`, stray `HTTPException`, Pydantic `RequestValidationError`). `code` is the stable machine key the frontend localizes; `message` is **dev-facing English** (logs/curl/unknown-code fallback), **never shown verbatim to end users**. Flat (not nested under `detail`/`error`) so the call site reads `result.error.code` with `openapi-fetch`. |
| **Where text lives** | **All** human-facing strings live in the **frontend** i18n catalogs. The backend emits **no** display text. The single seam between them is the **error-code registry** (§4.2) — a code is a contract, renamed only with a coordinated FE+BE change + codegen. |
| **`preferred_language` storage** | A **nullable** `users.preferred_language` (`String(16)`). **`NULL` = "never explicitly chosen"** → the frontend uses its pre-login resolution chain (so a brand-new admin who set up in a `zh` browser still sees ZH). A concrete value is **authoritative** and follows the account across devices. No forced `'en'` default (that would override browser intent). |
| **Read/update endpoint** | Read via the existing **`GET /api/auth/me`** (now carries `preferred_language`); update via **`PATCH /api/auth/me`** with body `{ preferred_language }`, returning the updated `MeResponse`. A general "current-user preferences" patch — M6 extends it with more fields, no new endpoint. |
| **Validation of language** | The supported set is enforced **app-layer** (`SUPPORTED_LANGUAGES = ("en", "zh")` in `app/core/languages.py`); an unsupported code → `AppError("validation.unsupported_language", 422)`. No DB CHECK (the set will grow; keep it in app code per roadmap §2.11). |
| **Resolution layering** | **Logged-in:** `me.preferred_language` is authoritative **when set**; when `NULL`, fall through to the client chain. **Pre-login:** localStorage explicit pick → `navigator` detect → fallback **`en`**. (Roadmap §5 M1.5.) Detailed in §4.3. |
| **Formatting scope** | M1.5 **includes** locale-aware **rendering** of dates & numbers via a thin **`Intl`** wrapper bound to the active language (`src/i18n/format.ts`). The existing `formatQuantity` (trailing-zero trim — see commit `18f9ec3`) folds into this wrapper and **keeps** that behavior while becoming locale-aware. |
| **Test-language pinning** | The test environment **pins i18n to `en`** (deterministic) so existing assertions that match English copy keep working after extraction; language-switch behavior is tested explicitly by forcing `zh`. |

> These extend M0/M1 foundations. The **error-contract refactor is a genuine cross-cutting change** to the API shape, so it is gated by the no-drift contract check (§6) and reviewed as a refactor (§10). `react-i18next` et al. are new frontend deps — note them in the M1.5 report; the `AGENTS.md` "Tech stack" line may gain `react-i18next` at wrap-up if the author wants the list exhaustive.

## 3. Data model (minimal — one column)

M1.5 adds **one** column; no new tables.

### 3.1 `users.preferred_language` (migration `0009`)

| Column | Type | Null | Notes |
|---|---|---|---|
| `preferred_language` | `String(16)` | **yes** | `NULL` = never explicitly chosen (→ client resolution). Concrete values in M1.5: `'en'` / `'zh'`. Validated **app-layer** against `SUPPORTED_LANGUAGES`; no DB CHECK (set will grow). |

- Added to the existing `users` table via Alembic. SQLite `ALTER TABLE ... ADD COLUMN` with a nullable column needs **no** batch-rebuild — a plain `op.add_column` suffices; downgrade is `op.drop_column` (batch mode on SQLite for the drop).
- The migration **backfills nothing** (existing rows stay `NULL` = inherit client resolution). No data exists pre-1.0 anyway.

> No household-level `default_language` in M1.5 (deferred, §12). The resolution chain (§4.3) covers the unconfigured case via browser detect → `en`.

## 4. Backend design

### 4.1 The uniform error-code contract (refactor)

**Today** every error is `raise HTTPException(detail="<english string>")`, serialized by FastAPI/Starlette to `{"detail": ...}`. M1.5 replaces this wholesale.

**New core module `app/core/errors.py`:**
- `class AppError(Exception)` — carries `status_code: int`, `code: str`, `params: dict | None = None`, `message: str | None = None` (dev-facing English; defaults to a generic per-code string). This is what services/routes raise from now on.
- `class ErrorResponse(BaseModel)` — the wire schema: `code: str`, `message: str`, `params: dict | None = None`. This is the **documented** error body (declared in OpenAPI so the contract is typed and gate-checked, §6).
- **Three exception handlers** registered in `app.main.create_app`, all emitting the **same** `ErrorResponse` shape:
  1. `AppError` → `{code, message, params}` at `exc.status_code`.
  2. `StarletteHTTPException` (stray `HTTPException`, e.g. the SPA-fallback 404 in `main.py`, or anything not yet migrated) → `{code: "http.<status>", message, params: null}` so even un-migrated paths obey the envelope.
  3. `RequestValidationError` (Pydantic 422) → **one** code `validation.invalid_input`, `message` = generic, `params: { "fields": [ {loc, type} … ] }` (machine-readable field list; the frontend may surface "check these fields" but does **not** localize each Pydantic message). A `500` safety-net handler emits `internal.error` without leaking internals.

**Refactor every existing raise site** (`api/deps.py`, `api/routes/auth.py`, `services/{tree,location,category,item_definition,stock_instance}.py`) from `HTTPException(detail=…)` to `AppError(code=…, params=…)`, keeping the old English string as the `message`. This is the bulk of Step 1.

### 4.2 Error-code registry (the FE↔BE contract)

Codes are **`<resource>.<reason>`**, stable, machine-readable. The registry is the single source of truth (kept as an enum/constants in `app/core/errors.py`; the frontend mirrors it as i18n keys under the `errors` namespace). Representative set (the implementer assigns a code to **every** existing raise; this table is the spine, not exhaustive):

| Code | Status | Raised by (today) | `params` |
|---|---|---|---|
| `auth.not_authenticated` | 401 | `deps` (no session) | — |
| `auth.session_invalid` | 401 | `deps` (expired/invalid session) | — |
| `auth.account_inactive` | 401 | `deps` (user gone/inactive) | — |
| `auth.invalid_credentials` | 401 | `auth.login` | — |
| `auth.account_disabled` | 401 | `auth.login` | — |
| `auth.setup_already_complete` | 409 | `auth.setup` | — |
| `validation.invalid_input` | 422 | Pydantic request validation | `{fields}` |
| `validation.unsupported_language` | 422 | `PATCH /auth/me` (new, Step 2) | `{value, supported}` |
| `tree.cycle` | 409 | `tree` reparent (self/descendant) | `{kind}` |
| `tree.delete_has_children` | 409 | `tree` delete non-empty | `{kind}` |
| `location.not_found` | 404 | `location`, `definition`, `instance` | `{id}` |
| `location.parent_not_found` | 404 | `location` | `{id}` |
| `location.delete_in_use` | 409 | `location` delete w/ instances or container link | `{id}` |
| `location.container_link_conflict` | 409 | `location` container-as-item uniqueness | `{id}` |
| `category.not_found` / `category.parent_not_found` | 404 | `category`, `definition` | `{id}` |
| `item_kind.not_found` | 404 | `definition` (bad `kind_id`) | `{id}` |
| `item_definition.not_found` | 404 | `definition`, `instance` | `{id}` |

| `stock_instance.not_found` | 404 | `instance`, `location` link | `{id}` |
| `stock_instance.serial_requires_qty_one` | 422 | `instance` service (`serial⇒qty=1`) | — |
| `stock_instance.serial_duplicate` | 409 | `instance` (dup `(definition_id, serial)`) | `{serial}` |
| `http.<status>` | * | catch-all for un-migrated `HTTPException` | — |
| `internal.error` | 500 | safety-net handler | — |

\* Verify against the actual raise in `services/item_definition.py` during Step 1; assign whatever the existing 409 there represents. The rule is **one code per distinct failure**, not per call site.

### 4.3 `preferred_language` endpoint & resolution

- **`GET /api/auth/me`** — `UserResponse` gains `preferred_language: str | None`; `MeResponse` wraps it unchanged. (Avoids a second round-trip: the gate gets the language with the user.)
- **`PATCH /api/auth/me`** — body `UserPreferencesUpdate { preferred_language: str | None }` (all-optional, PATCH-style). Validates against `SUPPORTED_LANGUAGES`; persists via `UserRepository` (no raw queries); returns the updated `MeResponse`. Requires a valid session (`get_current_user`). Setting `null` is allowed (explicit "unset" → re-inherit client resolution).
- **`SUPPORTED_LANGUAGES = ("en", "zh")`** lives in `app/core/languages.py`; both the validator and (later, if exposed) any discovery endpoint read it. M1.5 does **not** expose the set over the API — the frontend hardcodes the same list; keeping them in sync is a 2-line concern, flagged in §12 for a future `/api/i18n/languages` if the list grows.

> **Resolution (where the value is applied — frontend, §7.3):** logged-in → `me.preferred_language` if non-null (authoritative), else the client chain; pre-login → localStorage explicit pick → `navigator` detect (normalized) → `en`.

## 5. Quality gates & easy-to-get-wrong logic (Definition of Done)

Inherits M0 §5 / M1 §5 (`make check` = ruff + mypy + pytest + eslint + tsc + vitest, all green; build passes; `make codegen` no-drift). The **logic that must have unit tests** in M1.5:

**Backend**
- **Error envelope uniformity:** a representative `AppError` (e.g. `location.not_found`) returns `{code, message, params}` at the right status; a Pydantic validation failure returns `validation.invalid_input` with a `fields` list; a stray `HTTPException` (the SPA 404) returns `http.404`. **No** endpoint returns the legacy bare-string `detail` anymore.
- **Code stability:** the migrated auth errors carry the exact codes (`auth.invalid_credentials`, `auth.setup_already_complete`, …); user-enumeration safety is preserved (login email-not-found and wrong-password both → `auth.invalid_credentials`).
- **`preferred_language`:** default is `NULL`; `me` returns it; `PATCH` persists `en`/`zh`/`null`; an unsupported code → `validation.unsupported_language` (422) and **no** write happens.
- **Migration round-trip:** `0009` upgrades clean on a DB at `0008` and downgrades cleanly (column dropped).

**Frontend**
- **Resolution precedence:** localStorage pick > `navigator` > `en`; `zh-CN`/`zh-TW`/`zh-Hans` → `zh`; an unknown browser locale → `en`.
- **Switch is live & total:** changing language re-renders a sample of each surface (a pre-login screen + an authed page) into the target language; `<html lang>` updates.
- **Account persistence:** an authed switch calls `PATCH /api/auth/me`; the gate applies a non-null `me.preferred_language` on load.
- **Error localization:** a known code → the localized string with interpolated `params`; an **unknown** code → the generic localized fallback (never a raw English `message` to the user); `serial⇒qty=1` (422) and a delete-guard (409) both render localized.
- **Catalog integrity:** `en` and `zh` have the **same key set** (a unit test asserts no missing/extra keys across the two — cheap guard against drift).
- **Formatting:** dates/numbers render per active locale; `formatQuantity` still trims trailing zeros (preserve `18f9ec3`) and is locale-aware.

## 6. Contract-first codegen & no-drift gate

Unchanged mechanism (M0 §6). M1.5 changes the **error response shape** and adds `preferred_language` + `PATCH /api/auth/me`, so **every step that touches the API re-runs `make codegen`** and commits `openapi.json` + `frontend/src/api/schema.d.ts`. Because the error envelope is now a **declared `ErrorResponse` component** (attached to routes via a shared `responses=` helper), the contract documents it and the frontend can type the error branch (`result.error` → `ErrorResponse`). The CI **contract** job fails on drift exactly as before.

## 7. Frontend design

### 7.1 i18n architecture (`src/i18n/`)

```
src/i18n/
├── index.ts          # i18next init: detector, fallback, namespaces, <html lang> sync
├── languages.ts      # SUPPORTED_LANGUAGES = ['en','zh']; normalize(navigatorLang) → 'en'|'zh'
├── format.ts         # Intl wrappers: formatDate / formatNumber / formatQuantity (locale-bound)
├── errors.ts         # mapApiError(error) → localized message via the errors namespace
└── locales/
    ├── en/{common,auth,nav,locations,categories,items,instances,errors}.json
    └── zh/{common,auth,nav,locations,categories,items,instances,errors}.json
```

- **Init** (`index.ts`): `i18next.use(LanguageDetector).use(initReactI18next).init({ supportedLngs:['en','zh'], fallbackLng:'en', ns:[…], detection:{ order:['localStorage','navigator'], lookupLocalStorage:'omniventory_lang', caches:['localStorage'] }, load:'languageOnly' })`. `load:'languageOnly'` collapses `zh-CN`→`zh`. Subscribe to `languageChanged` → set `document.documentElement.lang`.
- **Provider:** i18next is global; wrap the root once (`<I18nextProvider>` in `main.tsx`, or rely on the singleton + `useTranslation`). Test env imports the same init but **pins `lng:'en'`** (§2).
- **Namespaces** keep catalogs cohesive and let later milestones add a namespace per feature without one giant file.

### 7.2 String extraction (Step 4 — the bulk of the UI work)

Replace **every** hardcoded human string with `t(...)` across the existing surfaces and populate both `en` and `zh`:
`pages/{Login,Setup,Dashboard,Locations,Categories,Items,InstanceDetail}.tsx`, `shell/AppShell.tsx` (nav labels, logout, color-scheme toggle), `components/{TreeBrowser,InstanceFormModal,EmptyState,ErrorState,LoadingState,PageShell}.tsx`. Includes button labels, placeholders, titles, `aria-label`s, empty/error/loading copy, confirm dialogs. `en` strings are the existing copy verbatim (so test assertions survive); `zh` is the faithful translation.

### 7.3 Language resolution & switching

- **Gate wiring (`App.tsx`):** after `GET /auth/me` resolves, if `user.preferred_language` is non-null → `i18n.changeLanguage(it)` **and** write it to localStorage (so a reload before `me` re-resolves still shows the right language). If null → leave the detector's choice. Pre-login screens already render via the detector chain (localStorage → navigator → `en`).
- **`<LanguageSwitcher>`** (one shared component): a compact EN/中文 toggle.
  - **Pre-login** (Setup/Login): `changeLanguage` + localStorage only (no session to persist to).
  - **Authed** (AppShell header): `changeLanguage` + localStorage **and** `PATCH /api/auth/me { preferred_language }` so it follows the account to other devices. Failure of the PATCH is non-fatal (local change still applies; surface a toast via the error mapper).

### 7.4 Error localization (`errors.ts`, Step 5)

`mapApiError(error: ErrorResponse | unknown): string`:
1. Defensively read `error.code` (+ `error.params`).
2. If the code exists in the `errors` namespace → `t('errors.'+code, params)`.
3. Else → `t('errors.generic')` (a localized "Something went wrong"); in `import.meta.env.DEV`, append the raw `message` for debugging.

Wire it into the call sites that currently show English text or raw `error.detail`: `Login` (invalid credentials), `Setup` (409), `TreeBrowser` (cycle / delete-guard / container conflicts), `InstanceFormModal` (`serial⇒qty=1` 422). These stop reading `error.detail` and call `mapApiError(error)`.

### 7.5 Locale formatting (`format.ts`, Step 7)

Thin `Intl` wrappers bound to `i18n.language`: `formatDate(d)` (`Intl.DateTimeFormat`), `formatNumber(n)` (`Intl.NumberFormat`), and `formatQuantity(q)` which **preserves** the existing trailing-zero trim (commit `18f9ec3`) while applying the locale's decimal grouping/separator. Replace the current `utils.ts` `formatQuantity` and the ad-hoc `created_at`/date renderings (e.g. in `InstanceDetail`, `Items`) with these.

### 7.6 Tests (vitest + Testing Library)

Resolution precedence; `zh-*`→`zh`; live switch re-renders a pre-login + an authed surface in `zh` and updates `<html lang>`; authed switch PATCHes; gate applies `me.preferred_language`; `mapApiError` known/unknown codes + `params` interpolation; `serial⇒qty=1` and delete-guard render localized; catalog key-parity test; `formatQuantity` trailing-zero + locale behavior. Keep the M0/M1 "mock the typed client" style; pin test language to `en` except where switching is the subject.

## 9. Step breakdown (atomic, ordered)

Each step is independently testable, test-backed, lands exactly one commit (one per-step autosquash in orchestrator mode), re-runs `make codegen` if it touched the API, and inherits the global DoD (§5) + the **anti-freelancing rule**: implement *only* this step — no other steps, no opportunistic refactors, no touching the host's real environment.

> **Ordering rationale:** the backend contract (1–2) must land first so the frontend consumes the new typed error shape + `preferred_language` path. Within the frontend: i18n infra (3) → string extraction (4) → error mapping (5, needs Step 1's codes + Step 3's infra) → switcher/persistence (6, needs Step 2's endpoint + Step 3) → locale formatting (7, needs Step 3).

### Step 1 — Backend: uniform error-code contract (refactor)
- **Goal:** replace `{"detail": str}` everywhere with the flat `ErrorResponse { code, message, params }` envelope, via `AppError` + three exception handlers; migrate every existing raise site to a stable code.
- **Build:** `app/core/errors.py` (`AppError`, `ErrorResponse`, the code registry §4.2); register `AppError` / `StarletteHTTPException` / `RequestValidationError` (+ 500 safety-net) handlers in `app/main.create_app`; refactor all raises in `api/deps.py`, `api/routes/auth.py`, `services/{tree,location,category,item_definition,stock_instance}.py`; attach a shared `responses={401,404,409,422: ErrorResponse}` helper to the routers so OpenAPI documents the envelope; `make codegen`.
- **Tests:** representative `AppError` → correct `{code,status,params}`; Pydantic 422 → `validation.invalid_input` + `fields`; stray `HTTPException` → `http.404`; auth codes exact + user-enumeration safety preserved; no legacy bare-string `detail` remains (assert shape on a sample of each status).
- **Out of scope:** `preferred_language` (Step 2); any frontend change.
- **Commit:** `refactor(backend): uniform error-code contract for i18n`
- *(May split into 1a infra+handlers / 1b migrate-raise-sites if it runs large; keep each green.)*

### Step 2 — Backend: `preferred_language` field + `me` read/update
- **Goal:** per-user language preference, read on `GET /auth/me`, updated via `PATCH /auth/me`.
- **Build:** `users.preferred_language` (nullable `String(16)`) on the model; migration `0009` (add column; reversible); `SUPPORTED_LANGUAGES` in `app/core/languages.py`; extend `UserResponse` with `preferred_language`; `UserPreferencesUpdate` schema; `PATCH /api/auth/me` (validate against the set → `validation.unsupported_language`; persist via `UserRepository`; return `MeResponse`); `make codegen`.
- **Tests:** default `NULL`; `me` returns the field; `PATCH` persists `en`/`zh`/`null`; unsupported code → 422 `validation.unsupported_language` + no write; migration 0009 up/down.
- **Out of scope:** any frontend consumption (Steps 3–6).
- **Commit:** `feat(backend): per-user preferred_language with me read/update`

### Step 3 — Frontend: i18n scaffolding + language resolution
- **Goal:** stand up `react-i18next` with the detection chain, namespaces, `<html lang>` sync, and the supported-language helpers — no string extraction yet.
- **Build:** add `i18next`, `react-i18next`, `i18next-browser-languagedetector`; `src/i18n/{index.ts,languages.ts}`; init (detector order localStorage→navigator, `fallbackLng:'en'`, `load:'languageOnly'`, namespaces); wire into `main.tsx`; seed empty/near-empty `en`+`zh` namespace files (the structure); pin test env to `en`.
- **Tests:** resolution precedence (localStorage > navigator > `en`); `zh-CN`/`zh-TW`→`zh`; unknown → `en`; `changeLanguage` updates `<html lang>`.
- **Out of scope:** translating existing strings (Step 4); error mapping (Step 5); switcher (Step 6); formatting (Step 7).
- **Commit:** `feat(frontend): react-i18next scaffolding and language resolution`

### Step 4 — Frontend: extract all M0–M1 UI strings (en + zh)
- **Goal:** every existing hardcoded human string flows through `t(...)`; `en`+`zh` catalogs fully populated.
- **Build:** replace strings across all pages/shell/components (§7.2); populate `locales/en/*` (verbatim existing copy) + `locales/zh/*` (faithful translation); add the **catalog key-parity** test.
- **Tests:** key-parity (en≡zh key sets); a sample surface renders the `en` copy (assertions survive); switching to `zh` shows translated copy.
- **Out of scope:** error-code mapping (Step 5), switcher/persistence (Step 6), formatting (Step 7).
- **Commit:** `feat(frontend): extract M0–M1 UI strings into i18n catalogs`

### Step 5 — Frontend: localize backend errors via stable codes
- **Goal:** the backend's error codes render as localized text; no raw English `detail`/`message` reaches the user.
- **Build:** `src/i18n/errors.ts` (`mapApiError`); populate the `errors` namespace (en+zh) keyed by §4.2 codes + a `generic` fallback; rewire `Login`, `Setup`, `TreeBrowser`, `InstanceFormModal` to call `mapApiError(error)` instead of reading `error.detail`/hardcoded strings.
- **Tests:** known code → localized + `params` interpolated; unknown code → `errors.generic`; `serial⇒qty=1` (422) and a delete-guard (409) render localized.
- **Out of scope:** switcher/persistence (Step 6); formatting (Step 7).
- **Commit:** `feat(frontend): localize backend errors via stable codes`

### Step 6 — Frontend: language switcher + account-bound persistence
- **Goal:** a runtime switcher (pre-login + authed) that persists to the account and is applied by the gate.
- **Build:** `<LanguageSwitcher>` (shared); mount on Setup/Login (localStorage only) + AppShell header (localStorage + `PATCH /api/auth/me`); `App.tsx` gate applies non-null `me.preferred_language` on load and syncs localStorage; PATCH failure surfaces via `mapApiError` (non-fatal).
- **Tests:** live switch re-renders a pre-login + an authed surface in `zh`; authed switch PATCHes (client mocked); gate applies `me.preferred_language`; pre-login switch writes localStorage only.
- **Out of scope:** locale formatting (Step 7).
- **Commit:** `feat(frontend): language switcher with account-bound persistence`

### Step 7 — Frontend: locale-aware date & number formatting
- **Goal:** dates/numbers render in the active locale; `formatQuantity` keeps trailing-zero trim and becomes locale-aware.
