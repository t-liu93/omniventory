## 1. 已锁定的决策

这些在第一轮 planning 讨论中已定。改动其一即属「根基」变更(技术栈一旦在 M0 敲定,也要同步更新 `AGENTS.md` 的「Tech stack & commands」)。

### 1.1 技术栈
- **后端**:Python · **FastAPI** · **SQLAlchemy**(ORM)· **Alembic**(迁移)· **SQLite**(轻量,默认)。
  - 业务/领域逻辑留在**应用层/服务层**——*不*使用 SQLite 专有 SQL、视图或触发器——让数据库保持可替换(将来真有需要可换 Postgres)。这也避开了 investigation 里「逻辑下沉到 SQL 会把你绑死在单一数据库」的坑(Grocy)。
- **前端**:**React + TypeScript** · **Mantine**(组件库)· 通过 `react-feather` 用 **Feather Icons**(官方 React 版本)· Vite。一个**响应式**(responsive)Web 应用,**桌面端与移动端同等重要**,并**同时具备 PWA 能力**(可安装、可离线)。v1 不做原生 App。
  - 选 React 而非作者更熟的 Vue 的理由:v1 用这个响应式 PWA 同时覆盖桌面端与移动端,且将来的 **React Native** 移动端可复用 React 技能/生态。在 agentic-coding 时代,手写熟练度不是决定因素。
- **部署**:单 Docker 容器(应用 + 内嵌前端 + SQLite 卷)。

### 1.2 租户与用户 —— **单租户、多用户**
- 一个部署 = 恰好**一个**「家/公司」(一个有名字的**单例 `Household/Workspace`** 实体:名称、货币、时区、设置)。
- **多个用户账号**共享该「家」内的全部数据;区分靠**账号 + 角色**,而非租户。
- 业务表**不带 per-row 租户归属字段**(只有一个租户)——模型更简单。
- **未来转多租户的对冲(便宜的保险,从第一天就做)**:
  1. 单例 `Household/Workspace` 实体现在就建(单租户 = 只有一行)。
  2. 所有 DB 访问走**仓储/服务层 + 单一「当前上下文」抽象**——绝不散落裸查询。将来转多租户就变成:加 `household_id` + Alembic 回填到现有那一行 + 打开一处集中的作用域过滤 + 把 user↔household 变成成员关系 + 唯一约束改成按租户。可控,而非重写。

### 1.3 范围决策
- **四项「选配」能力全部纳入范围**,作为正式里程碑:条码扫描 + 商品识别、多单位换算、购物清单、耐用品维护计划。
- **CSV 导入导出 + 备份恢复**属核心(M5),非选配。
- **集成**(靠后的里程碑):**SMTP**(并入 M4 提醒通道)、**MQTT**(Home Assistant / IoT 双向)、以及**OpenAI 兼容的 LLM 后端**(扫商品/收据 → 自动归类;收据 → 批量建档)。LLM 后端**现在只留抽象/hook;真正功能往后做**。
- 明确**不做** InvenTree 式的 BOM / 生产订单 / 采购销售订单 / 零件变体——对家庭场景过度设计。

## 2. 全局约束(项目红线)

每个里程碑都成立。多数是 investigation §3.3 的取舍,在此变成硬约束。

1. **定义 / 实例分离是脊梁。** 「这是什么品类」(Item **Definition**:名称、分类、单位、默认值、阈值)与「具体这一批/这一件」(Stock **Lot/Instance**:数量、位置、序列号、批次、日期、价值)分开。绝不压成一张表(Homebox 的坑 → 同物多批/多到期管不了)。
2. **「逐件」与「成批」共用一张实例表,用约束统一:** `serial ⇒ quantity = 1`(DB *与* 应用层双重强制)。带序列号的耐用品(②)与成批耗材(③)同表。
3. **数量永远由流水账推导,绝不覆盖。** 每次出/入/移动/调整都是一条类型化事务;当前数量 = 聚合得出。支持消耗速率、审计、撤销。(Homebox 的覆盖式 float 是反例。)
4. **有效期 / 保修 / 价值 / 位置落在实例/批次上;默认值落在定义上**(`default_best_before_days`、`min_stock`、默认位置)。批次级 best-before(Grocy 的精度)优于物品级。
5. **位置是自引用树**(任意深度)。**容器本身也可以是一件被追踪的物品**(工具箱既是位置又是有价值的资产——Homebox 的好洞察)。绝不用扁平位置(Grocy 的坑 → 做不了耐用品的层级定位)。
6. **一个统一的提醒引擎,多种触发源。** 扫描 `best_before`、`warranty_expires`、`min_stock`/低库存,以及(后续)维护到期。**可配「提前 N 天」**,可按物品、按用户;**事件触发 + 每日定时兜底**(双保险);**默认开启**;通知到**责任人**;通道**可插拔**(in-app、SMTP、MQTT、webhook)。避开三个项目各自的提醒缺陷(硬编码 / 默认关 / 只发订阅者 / 只能拉取)。
7. **保修是一等公民字段**,也是一种提醒源(三个参考项目都缺像样的保修处理)。
8. **横切能力做成通用/可复用,不在每张表重复堆字段:** 照片/附件(通用 `model_type + model_id`)、标签、备注、自定义字段、条码。
9. **金额和数量用 `Decimal`;** 数量支持小数(线材按米、液体按升)。涉及金额处定死舍入规则。
10. **现在单租户,但为多租户预留**(见 §1.2 对冲):单例 `Household/Workspace` 实体 + 集中的数据访问/上下文层。
11. **逻辑在应用层,不在数据库。** 业务逻辑不进 SQL 视图/触发器(可移植 + 可测试)。
12. **OpenAI 兼容的 LLM 后端是一个预留抽象。** 即便还不实现,也为它(视觉归类、收据 → 批量建档)留好接缝。

## 3. 领域模型脊梁(概览)

详细 schema 落在各里程碑设计文档;这里是万物所挂的共享骨架。

- **Household/Workspace** —— 单例配置(名称、货币、时区、默认值)。
- **User** —— 账号 + 角色;认证。(多用户在 M6 完整展开。)
- **Location** —— 自引用树;位置也可以*就是*一件物品(容器即物品)。
- **Category** —— 树。
- **Item Definition** —— 名称、分类、kind(耐用/耗材/易腐 提示)、单位、默认位置、`default_best_before_days`、`min_stock`/补货点、`trackable`(可序列号)、自定义字段、条码、照片。
- **Stock Lot / Instance** —— FK→Definition、FK→Location、`quantity`(Decimal)、`serial`(可空;`serial ⇒ qty=1`)、`batch`、`best_before_date`、状态(开封/冷冻……)、`warranty_expires`、status、`purchase_price`/`date`/`source`、`received_at`、责任用户。
- **Stock Movement(流水账)** —— 类型(intake / consume / move / adjust / discard / correction)、FK→Lot、数量增量、from/to 位置、时间戳、用户、备注、撤销链接。数量由这些推导。
- **Reminder / Notification** —— 触发源(best_before、warranty、low_stock、maintenance-due)、按物品 & 按用户的提前天数、通道适配器、投递日志。
- **横切** —— Attachment(通用)、Tag(可嵌套)、Note、CustomField、Barcode。

## 5. 里程碑(详情)

> 每个里程碑都可独立部署、可演示。🟢 是部署自测检查点,里程碑的 `M<x>.md` 设计文档会展开它们,里程碑报告会把它们串成人工 walkthrough。

### M0 —— 脚手架与地基
**目标:** 一个空但能跑、CI 绿、能登录的应用。
- 仓库布局(`backend/`、`frontend/`);包管理器;lint/format/类型检查/测试 工具链(后端:ruff + mypy + pytest;前端:eslint + tsc + vitest);CI。
- FastAPI 应用 + 配置/设置 + 健康检查;SQLAlchemy + Alembic 接好;首个迁移。
- 单 Docker 容器;前端(React + Vite + Mantine + react-feather)内嵌/托管;**响应式**布局 + PWA 外壳。
- **认证骨架**(先单管理员;session/JWT)+ **单例 `Household/Workspace`** 实体 + **集中的数据访问/上下文层**(多租户对冲)。
- **契约先行**:OpenAPI → TS 类型生成,CI 强制无漂移。
- 🟢 应用启动;`/api/health` 绿;登录可用;一个 Alembic 迁移干净应用;CI 绿;`docker build` 跑通整体。

### M1 —— 统一核心模型 & 耐用品登记(②)
**目标:** 在真实的位置层级里登记并浏览耐用品。
- 位置树 CRUD(含容器即物品);分类树。
- Item Definition CRUD;Stock Lot/Instance CRUD,强制 `serial ⇒ qty=1`。
- 实例上的耐用品字段:序列号 / 型号 / 制造商、`warranty_expires`(先存;提醒在 M4)、价值/购入。
- 列表 / 详情 / 树浏览 / 搜索 UI。
- 🟢 建一棵嵌套位置树;把一件带序列号的耐用品登记进去;编辑;搜索;`serial ⇒ qty=1` 约束能挡住非法输入。

### M1.5 —— 国际化(i18n)地基
**目标:** 趁界面还没铺更多之前,把 UI 做成双语(中 + 英)—— 这是一项横切的「地基」能力,早做,好让后续每个里程碑都通过 i18n 写文案,而不是硬编码英文。
- 前端:**react-i18next** 脚手架;把现有 M0–M1 的全部 UI 文案抽进词条表(中 + 英);一个语言切换器(登录前 / setup 页也要有)。
- **分层语言解析:** 已登录 → 账户落库的 `preferred_language`(权威,跨设备跟人走);登录前 → 登录/setup 页的显式选择(记在客户端)→ 浏览器自动探测 → 兜底 **EN**。
- 后端:每用户 `preferred_language` 字段 + 一个读/写接口。因为 API 消息走**稳定错误码**,后端**自己不做**本地化——它只回 code,前端把 code 映射成本地化文案(在此定下,让「传输/展示」从一开始就分离)。
- 🟢 运行时切换语言(中 ⇄ 英)整个 UI 跟着变;选择跟账户跨会话/设备持久化;登录页在认证前就按解析出的语言显示;一条后端报错经其 code 本地化显示。

> 已锁定:**react-i18next**、中 + 英、后端错误码。详细设计文档(`docs/plan/milestones/M1.5.md`)现已写就;它敲定了语言持久化细节(账户绑定的 `preferred_language`,可空 = 继承客户端解析;登录前回退 localStorage → 浏览器 → **EN**),把后端错误码并入**一次完整的统一错误信封重构**(扁平 `{code, message, params}`),并把 locale 感知的日期/数字格式化纳入范围。

### M2 —— 库存流水账 & 耗材(③)
**目标:** 用出入库流水追踪耗材库存,并检测低库存。
- 流水账表(intake / consume / move / adjust / discard / correction);**数量由其推导**;**FIFO** 消耗顺序(先 best_before 再 received);**撤销/反向**。
- Definition 上的 `min_stock`/补货点;低库存计算端点 + 仪表盘卡片。
- UI 里的 入库 / 消耗 / 移动 流程。
- 🟢 入库 10,消耗 3(FIFO),看到推导数量 = 7;撤销可还原;设 `min_stock`,降到其下,低库存浮现。

### M3 —— 保质期/临期 & 易腐品(①)
**目标:** 追踪 best-before 日期,并把临期的东西暴露出来。
- Lot 上的 `best_before_date`;Definition 上的 `default_best_before_days`(入库时自动推算)。
- 临期 / 过期 过滤 + 仪表盘;FIFO 已遵循 best_before。
- *(可选,可延后到一次细化:* 开封/冷冻/解冻 「+N 天」调整——Grocy 式。*)*
- 🟢 入库一件带默认保质天数的易腐品 → best_before 自动算出;列出 N 天内将到期的;过期项被标记。

### M4 —— 统一提醒与通知引擎(①②③ 主动)
**目标:** 差异化亮点——一个引擎,跨所有源做「提前 N 天」主动提醒。
- 汇合触发源:best_before、`warranty_expires`、低库存(维护到期在 M7 加入)。
- 按物品**和**按用户的可配提前天数;**事件触发 + 每日定时扫描**(双保险);**默认开启**;指派/通知责任用户;汇总。
- **可插拔通道**:in-app + **Email SMTP**(MQTT 通道在 M9)。
- 🟢 配置提前天数;跑每日扫描;在 in-app 和邮件里收到一条汇总提醒,覆盖一件临期物、一份临期保修、一件低库存物。

### M5 —— 横切能力 + 条码 + 数据进出
**目标:** 一个真正自托管应用的日常易用性与数据可移植性。
- 通用附件/照片、标签、备注、自定义字段、全局搜索。
- 入库/识别时的**条码扫描 + 商品查询**。
- **CSV 导入导出**;**备份/恢复**(DB + 媒体 的 ZIP)。
- 🟢 加一张照片;扫码入库;导入一个 CSV;导出备份并恢复到干净实例。

### M6 —— 多用户 & 角色
**目标:** 单「家」内的真正多用户 + 加固。
- 多个用户账号;邀请;角色/权限(admin / member / viewer);提醒里体现责任人指派;基础审计。
- 安全加固:渲染内容处的输入清洗,对外通知器/webhook 的 SSRF 防护,限流。
- 🟢 邀请第二个用户;角色强制正确放行/拦截;基于指派的提醒发给正确的人。

### M7 —— 购物清单 & 维护计划
**目标:** 闭合耗材回路与耐用品养护回路。
- 由低库存自动生成购物清单(+ 手动项;勾选 → 可选入库)。
- 耐用品维护计划(周期性;作为维护到期源喂给 M4 提醒引擎)。
- 🟢 低库存自动填充购物清单;勾选可入库;一个排定的维护触发提醒。

### M8 —— 多单位换算
**目标:** 「按盘买、按件用」(Grocy 式),可选启用。
- Definition 上的 采购/库存/消耗/计价 单位 + 换算系数;入库/消耗 遵循换算;计价随之。
- 🟢 定义单位(1 盘 = 24 件);入库 2 盘;消耗 5 件;库存与价值正确。

### M9 —— 集成与扩展(post-1.0)
**目标:** 接入智能家居,解锁 AI 辅助录入。
- **MQTT** 桥(Home Assistant / IoT):发布库存 / 临期 / 低库存 事件,接受命令;MQTT 也可作提醒通道。
- **公开 REST API + webhook + OpenAPI 客户端。**
- 基于预留 hook 的 **LLM(OpenAI 兼容)** 功能:扫商品/收据 → 自动归类;收据 → 批量建档。
- 🟢 一条临期事件经 MQTT 抵达 Home Assistant;上传一张收据经 LLM 草拟出一批商品。

## 6. 停车场(延后 / 「以后再说」)

未排期;核心稳定后再议。
- 开封/冷冻/解冻 「+N 天」易腐品细化(若没并入 M3)。
- OIDC / SSO;2FA。
- **资产标签 & 标签打印**(配合 M5 条码扫描):对**没有厂商序列号**的 durable,自动发一个**内部资产标签**——短的可读码(如 `OMNI-000123`),**不是裸 UUID**,且与厂商 `serial` 字段**分开**。生成适合标签打印机的 **QR/条码标签 SVG**(二维码 + 可读码),贴到箱子/工具上,之后用**扫码定位 / 归档**(扫码本身落在 M5)。闭环:自动发码 → 打印 → 贴 → 扫。
- **「把实例提升为容器位置」一键 UX**:一步建好镜像 location 节点 + `item_instance_id` 容器即资产关联(省掉"被追踪资产同时当地点"时的手动两步——见容器即资产桥,M1 §3.1)。
- React Native 原生移动端(响应式 PWA 覆盖 v1)。
- Postgres + RLS(仅当真要做多租户 SaaS 时)。
